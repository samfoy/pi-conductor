## 3. Critical invariant: reload preservation

The `1c72856` "preserve runs on `/reload`" feature is load-bearing for the
developer iteration loop — losing it costs every in-flight sub-agent on every
edit.

### 3a. Reload still skips disk writes

`src/shutdown.ts:88` retains the original `if (event.reason === "reload")
return;` early exit. The new `reconcileRunning` call (`shutdown.ts:101-105`) is
**after** that guard, so reload bypasses both SIGTERM and the disk write-back.

Test: `tests/shutdown.test.ts:206` — `reason=reload does NOT call
reconcileRunning (lifecycle preservation, 1c72856)` — explicitly cites the
landmark commit and asserts `reconciled.length === 0`. Plus the original
`tests/shutdown.test.ts:62` (`reason=reload does NOT kill running sub-agents`)
and `tests/shutdown.test.ts:77` (`reason=reload preserves run records (no
mutation)`) still pass — the existing landmark tests aren't relaxed by Slice 2.

### 3b. Quit reconciles after SIGTERM

`shutdown.ts:91-99` SIGTERMs running/paused first, accumulates a
`stillActive[]`, then calls `reconcileRunning(stillActive, event.reason)` only
if `stillActive.length > 0` and the dep is provided.

Test: `tests/shutdown.test.ts:183` — `reason=quit reconciles running/paused
runs after SIGTERM`. Asserts:
- `reconcileRunning` invoked exactly once
- only `running` + `paused` ids passed (`completed`, `queued` filtered)
- `reason` forwarded for marker construction

### 3c. SIGTERM logic untouched by the new disk-write side

`shutdown.ts:91-98` is the existing SIGTERM loop verbatim — `r.proc?.kill("SIGTERM")` plus the swallow-throw. The reconciler dep is purely additive
(`stillActive` accumulator + new conditional call); no `Run.proc` mutation
moves into reconciliation. Confirmed via `git show 9b2bc1c -- src/shutdown.ts`:
the kill loop body is unchanged; only an `if`-block + a single try/catch are
added below it.

## 6. Forward-looking — Slice 3

Slice 3 implements the cold-archive + delete `executeReclaim`. Two angles:

### Is the `reconcileOrphans` API contract sufficient inspiration?

**Yes.** Properties that should propagate to `executeReclaim`:

1. **Best-effort, never-throw.** `reconcileOrphans` returns `{reconciled, failed}`; same shape (`{archived, deleted, failed}`) is the right contract for the executor. Plan §"Slice 3" already implies this.
2. **ENOENT-silent on missing dirs.** Same race window applies to executor (a previous GC pass may have already deleted the dir; running the executor twice must be safe).
3. **Idempotent re-run.** `reconcileOrphans` skips already-non-running records; `executeReclaim` should skip already-archived dirs (i.e., `.archived` sidecar already present) and missing dirs.
4. **Action-kind filter at the top.** Lets callers pass `plan.actions` directly; minimizes coupling.
5. **`now` injected, not read from `Date.now()`.** Keeps tests deterministic. Slice 3 should follow.

### Race-condition concerns between reconcile and executor

- **Race 1 — reconciler runs after executor in the same pass.** If the same plan contains both a `reconcile-orphan` for `id-X` and a `delete` for some other id, they're independent; no concern.
- **Race 2 — could an entry simultaneously be `reconcile-orphan` AND `delete`?** No: policy rule 1's "active in registry" check + the registry-vs-disk pairing means an orphan (running on disk, absent in memory) can be reconciled. After reconcile, it's `killed` on disk; on the *next* GC pass it becomes a candidate for `delete` based on its now-`finishedAt`. Plan correctly serializes these across passes — the same pass should not emit both kinds for the same id. **Test gap** for Slice 3 to consider: invariant test `actions for the same id never include both delete and reconcile-orphan`.
- **Race 3 — D8 two-gate invariant must propagate to executor.** `reconcile.ts` currently relies on the policy engine's upstream `inMemory === undefined` gate; the comment at `reconcile.ts:18-20` is explicit. `executeReclaim` should preserve the same upstream-gate-only contract — DO NOT re-check `Run.proc` inside the executor; that's the policy's job. Otherwise the executor accumulates registry coupling. (Plan currently sketches `executor.ts` as importing `runs.ts` only for the registry handle, which is borderline — recommend keeping that to a single `isActive(id)` call passed in as a dep.)
- **Race 4 — concurrent `processLine` writes during cold-archive.** Inspector surprise #3 (`runs.ts:770` fire-and-forget appendFile). Two-gate invariant in policy engine + filter `inMemory !== undefined` already prevents this. Slice 3 should add a defensive `existsSync(transcriptPath)` precondition before unlink — symmetric to the `ENOENT silently skipped` pattern in reconcile.ts.

**Recommendation for Slice 3:** mirror reconcile's API shape; preserve D8's "policy is the gate, executor is dumb" contract; add the never-both-kinds-for-same-id invariant test. No blockers from Slice 2 carry forward.
