## 2. The four open decisions

### D1. Color depth — **256-color baseline**

**Decision.** Target 256-color. The host's `Theme.getColorMode()` reports `"truecolor" | "256color"` and named slots resolve correctly in either mode (inspector §6). We never address colors by RGB — only by named slot (`accent`, `success`, `error`, `dim`, `muted`, `warning`, etc.). Truecolor terminals get the theme's nicer hues for free; 256-color terminals get the fallback. 16-color terminals are out of scope (the host does not advertise a `"16color"` mode).

**Alt considered.** Truecolor-only: would let us pick exact greens/ambers, but we'd duplicate theme-color logic and break on shared dev boxes that report `xterm-256color`.

**Alt considered.** 16-color portable: rules out half the named slots (`thinkingText`, `mdQuoteBorder`, etc.) and makes the redesign visibly worse on the host's own dev terminals. No evidence anyone using pi-conductor is on a 16-color tty.

### D2. Thinking-block default — **summary-visible by default; full body behind `t`**

**Decision.** Default for both Ctrl+G overlay and foreground stream becomes: render a one-line `· thinking (N chars / M lines)` summary in dim color, no body. `t` toggles full body. Footer hint shows `t thinking` always. Foreground stream gains the same `t`-aware behavior **as a render-time switch** but does not gain a key listener (the foreground card has no input loop); the model toggle in the overlay remains the source of truth.

**Rationale.** Sub-issue #3 says the toggle hint is hidden inside the overlay. Adding a one-line summary surfaces *that thinking happened* (which is the affordance users want) without dumping content. `t` then unfolds.

**Alt considered.** Flip default to body-visible. Rejected: thinking blocks are >500 chars routinely; body-on-by-default re-creates sub-issue #6 (vertical waste) and shifts the complaint from "I can't see thinking" to "I can't escape thinking."

**Alt considered.** Keep hidden, only add hint. Rejected: the user explicitly wants signal that thinking happened — a footer hint without a body cue does not convey that.

### D3. Status-line / body relationship — **keep separate; visually tie via shared accent + drop one ruler**

**Decision.** The header stays a header (it has stable status / elapsed / usage data), but:
1. Drop the **bottom** ruler of the header. Keep only the top ruler.
2. Color the header glyph and persona name with `theme.fg("accent")`; same color is reused on the first turn separator and on the first character of every tool-call line, so the eye reads "the body and the header share an accent."
3. The first turn separator immediately under the header replaces the dropped bottom ruler structurally (no extra blank line).
4. The status string in the header (`running 0:34`) becomes `running · last: bash aws sts… · 0:34` — pulling the `Working: <activity>` content (sub-issue #5) into the header itself.

**Rationale.** Merging into a single block forces us to choose which header data wins on narrow widths (≤80 cols); keeping the rows but unifying the chrome buys us the visual tie without the layout fight. Drops 1–2 lines of vertical chrome (sub-issue #6) and addresses sub-issue #8 directly.

**Alt considered.** Full merge into a single multi-cell row. Rejected: at 80 cols `glyph persona (id) — running · last: <tool> · elapsed [usage]` overflows; ellipsising the live-activity field makes sub-issue #5 worse, not better.

### D4. Ctrl+G overlay scope — **constrained chrome polish + the `t` default change + a scroll indicator**

**Decision.** Treat Ctrl+G as a chrome-polish slice, not a structural redesign. The transcript body it renders is shared with the foreground stream — fixing the 8 sub-issues there propagates automatically. The overlay-specific work is:

1. Color the keybind hints (footer) using `keyHint` / `keyText` from the host components index (inspector §6).
2. Add a one-line scroll indicator above the footer: `↑ N hidden  ↓ M hidden  · L lines total` (suppressed when scrollOffset == 0 *and* transcript fits).
3. Drop the now-redundant `renderRulers` ruler in the empty state (no focused run); replace with a `border:borderMuted` framed `Box` from pi-tui.
4. Adopt the host's `truncateToWidth` / `wrapTextWithAnsi` only for lines that have ANSI bytes (i.e. the tool-call line once we color the tool name). Plain text lines keep the current `wrap()` until we have a reason to break it.

**Rationale.** The user's "ugly as sin" complaint is mostly *missing chrome* (no color, no scroll cue, no border around the empty state) and *redundant chrome* (four near-identical horizontal rulers per turn). Polish addresses both; full redesign would bloat the slice and risk regressing the snapshot tests.

**Alt considered.** Full redesign of the overlay as a side-panel `Box`-bordered viewport with explicit scroll gutter and panes. Rejected: pi-tui's `Box` API isn't documented in our inspector pass (§6 conspicuous gaps); spec-ing it would balloon to v0.9.

## 3. Architectural fork — styling layer

Inspector surprise #3: the pure renderers emit no ANSI. Color requires a fork.

**Decision: Option (A) — styling lives in the Component layer.**

Pure renderers (`renderHeader`, `renderTranscript`, `renderFooter` and friends) keep returning plain `string[]`. The `FocusedStreamOverlay` Component (and a thin foreground-stream Component-equivalent helper) post-processes those lines, applying `theme.fg(slot, line)` based on a small line-classification function — also pure, also snapshot-testable on its own.

```
renderTranscript(run, opts) → string[]
        │
        ▼
classifyLine(line) → { kind: "header" | "turnSep" | "tool" | "thinking" | "text" | "footer", glyph?, accent? }
        │
        ▼ (Component-only)
applyTheme(line, kind) → ansi-styled line
```

### Why (A) over (B) and (C)

- **TDD impact.** Option (A) keeps the 50+ existing tests in `tests/transcript.test.ts` passing without re-snapshotting against ANSI byte sequences. The new `classifyLine` becomes a tiny pure helper with its own spec; styling is checked once per slot via a small set of "header line is accent, tool line is muted" assertions in a new component-layer test that can stub `theme.fg` to a sentinel `[ACCENT]…[/]` string. Option (B) would force every existing snapshot to migrate to ANSI-stripped diff helpers — large blast radius, low reward.

- **Code locality.** Today only `widget.ts` knows about `theme.fg`. (A) extends that pattern: there is exactly one place per Component that introduces ANSI. (B) would have ANSI bleeding into the renderer's `wrap`/`truncateOrPad` helpers — and those use raw `String.length` (inspector §8), which silently mis-wraps on ANSI input. We'd have to migrate to `wrapTextWithAnsi` / `visibleWidth` in the renderer just to avoid breakage. Bigger surface.

- **Dep surface.** Both (A) and (B) only use already-exported primitives. (B) additionally tightens our coupling to `wrapTextWithAnsi` semantics — fine, but more API surface to track on host upgrades. (A) only depends on `theme.fg`, which is already in production use in `widget.ts`.

- **Tag-based hybrid (C)** would invent a new mini-DSL (`[[accent]]…[[/]]`) and a parser for it. Pure overhead; rejected.

### Consequences

- New file: `src/transcript-classify.ts` — pure `classifyLine(s: string): LineKind`. Discriminates on first char(s) of the line: `▸`/`▾` → tool, `┃` → thinking, `── turn` or pure `─` → separator, header is a known prefix produced by `renderHeader` (we can mark it explicitly via a sentinel in the renderer rather than sniff text — see slice plan).

- New file: `src/transcript-style.ts` — pure `applyTheme(line, kind, theme): string`. Theme is the third argument so it stays unit-testable with a fake.

- `FocusedStreamOverlay.render` and `renderForegroundStream` (Component-equivalent path) call `applyTheme` *after* the existing renderer; foreground does it inside `renderForegroundStream` since the foreground card surfaces ANSI fine in pi's tool-card slot.

- Snapshot tests for `renderTranscript` stay monochrome. New tests cover `classifyLine` and `applyTheme` independently. Component-layer tests assert "kind X gets slot Y" using a stubbed theme.

## 4. The dedup question — **Pre-slice it**

**Decision.** Land a tiny dedup slice **before** the redesign begins.

**Slice 0** (S, pre-redesign):
- Move `STATUS_GLYPH` to `src/types.ts` (or a new `src/status-glyph.ts` if `types.ts` should stay declaration-only). All four call sites import from there: `transcript.ts`, `foreground-stream.ts`, `widget.ts` (replace switch), `event-handler.ts`.
- Decide on the duplicate `summarizeArgs` (`transcript.ts:193`) vs `formatToolCallShort` (`event-handler.ts:110`): consolidate the *core* short-form helper into `src/tool-summary.ts` shared between transcript and event-handler. Keep the `$ ` prefix and `shortenPath` opinions of `formatToolCallShort` *only* in event-handler.ts (`run.lastToolCall` consumers expect them); transcript.ts uses the bare core.

**Justification.**
- The user's stated preference is small daily CRs (`memory: cr_review_strategy: ~1 CR/day cadence`). One small clear-the-decks commit lets the 8 sub-issue slices each remain genuinely small.
- Folding dedup into the redesign would force every behavioral slice to also touch four files for glyph/summary import paths. That muddies the diff and makes review of the *behavioral* change harder.
- The dedup is risk-free: pure re-export, covered by existing tests at all four call sites.
- Slice 0 also unblocks Slice 4 (`✓/✗` outcome marker) which needs a single shared list of glyphs to extend with `▶ pending` etc.

**Cost.** ~1 commit, ≤80 lines diff. Slows the redesign by half a day at most.

## 8. Risks & open questions

### Risks

- **`run.messages` may not always carry a usable result `is_error` flag.** Sub-issue #2's outcome marker depends on it. Mitigation: a second source — `result.text` starting with `Error:` or containing a known error envelope — is already used by `event-handler.ts`; we'll mirror that heuristic. Open: can we *detect* "no result yet" reliably to render a `▶ pending` glyph? Pin in Slice 2's design.

- **Slice 11 (move `model.refresh()` out of render) is the only behavior-changing structural slice** in the overlay. If it regresses cycling-keystroke responsiveness we'll see it as a stale-frame in tests. Mitigation: Slice 11 lands last and behind a small integration test that drives keystrokes through the dispatch and asserts the snapshot updates.

- **Theme slot availability across themes.** `theme.fg("accent")` returns the configured theme's accent slot — we don't know how the `rose-pine` theme (user's default) renders against the host's other themes (`gruvbox`, `dracula`, etc., per the host components index). Mitigation: assertions check that *some* slot is applied, not a specific RGB; visual review left to dogfood.

- **`keyHint` / `keyText` signatures are not surfaced in the inspector pass** (§6 conspicuous gap). Slice 9 may need a 5-minute spike to confirm signatures before its TDD pass starts; if the API is awkward, fall back to inline `theme.fg("accent", key) + " " + theme.fg("muted", label)`.

- **Foreground-stream throttle vs new outcome line.** `createUpdateThrottle` is 500ms leading+trailing. Adding the outcome line means a tool result that arrives within 500ms of its call may render the call without the outcome and never re-render before the throttle fires. Mitigation: the throttle's *trailing* edge will fire eventually; users on a fast-completing tool will see a half-second flash of "call without outcome" then the outcome appears. Acceptable; otherwise we'd need to make tool-result events bypass the throttle (out of scope).

### Open questions for the user

- **Confirm 256-color baseline (D1) is right** for the user's terminal mix. If the user runs pi inside `screen` or a remote tmux that downgrades to 16 color, half the named slots will degrade. Easy answer: the user's `~/.pi/agent/settings.json` already specifies a truecolor-friendly theme (`rose-pine`); 256-color is a safe floor.

- **`<activity>` display: does "thinking" live in the header, or only as a turn-body event?** D3 puts it in the header; the v0.8.1 worth-thinking-about list also names "dedicated `Working: <activity>` status." We treat both as the same field and put it in the header. Confirm the user wants header-only, or wants it duplicated as a transient body line.

- **Slice 0 dedup of `summarizeArgs`/`formatToolCallShort` — does it touch the **out-of-scope** `event-handler.ts`?** Yes, by necessity. The user's "out of scope" list (context.md) didn't enumerate event-handler.ts as forbidden, but the inspector flagged it as out-of-scope for the in-scope-7-files framing. Confirm the dedup commit is permitted to touch it. If not, defer Slice 0's `tool-summary.ts` consolidation; STATUS_GLYPH still consolidates.

- **Does the user want a v0.9-deferred ticket filed for the `Box`-bordered overlay viewport?** Recommend: add a one-liner to `docs/v0.8.2-backlog.md` under v0.9 candidates rather than a separate ticket.

### Defer to v0.9 with rationale

- **`Box`/`DynamicBorder`-based overlay frame.** Pi-tui's `Box` API isn't documented in inspector §6; speccing it requires reading host source. The line-hint scroll indicator (O2) and chrome polish (O1, O3, O4, O5) close the "ugly as sin" complaint without the structural change.

- **Per-agent scroll cycle annotation** (O7's deferred sibling). Needs design work on the cycle UX itself — separate slice.

- **Migrating `wrap()` to `wrapTextWithAnsi`** wholesale. Only needed if we ever inject ANSI into pure-renderer text (we do not, per the Option A pick). If a future slice does, this becomes Slice N.
