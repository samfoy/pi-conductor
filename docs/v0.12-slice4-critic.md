### ✗ Blocker

None. Both gate criteria the task wording would have flagged
(Mutation C(i) and Mutation E) reduce on inspection to test-shape
gaps, not production correctness defects:

- **C(i) clearTimeout drop is benign.** Even with the leak, the
  30s timer's callback at `src/runs.ts:1574` does
  `pendingAcks?.delete(id)` (no-op since `routeRpcResponse` already
  deleted) and `reject(...)` on an already-resolved promise (silent
  per ES spec). Net production impact: one wasted timer handle held
  for ≤ 30s per send. Memory/handle pressure, not correctness.
- **Mutation E**: schema enum drop — tests directly invoke
  `tool.execute(params)`, bypassing pi's JSON-schema validation
  layer. This is the standard pi-conductor unit-test shape (slices
  1–3 have the same property). Inherent test-infrastructure, not a
  slice 4 defect.

The `cfg.defaultSteerable` mutation gap (Mutation A) is the closest
to a Blocker, but the task instructions explicitly say *"Note it as
a future-refactor opportunity but don't block the slice"* unless
the production cascade is genuinely untested in a way that hides a
real bug. Inspection confirms the code is correct
(`src/tools.ts:287–293`); only the per-layer JSON-merge path is
untested. NOTE.

### ⚠ Notes (carry to slice 5 / v0.12 closure)

1. **Timer-leak detection is shallow on both ack paths**
   (Mutation C(i) + Mutation D). Dropping `clearTimeout` in either
   `routeRpcResponse` (`src/event-handler.ts:155`) or the EPIPE
   catch (`src/runs.ts:1600`) leaves all 5 ack tests green. The
   tests resolve the ack promise within `mock.timers` virtual time
   that never reaches the 30s tick, so a stale timer would fire
   harmlessly post-test. Slice 6's live integration test
   (`CONDUCTOR_LIVE_TESTS=1`) could pin handle counts via
   `process._getActiveHandles()` if production telemetry ever
   shows leaks. Low priority.

2. **Production cascade (`cfg.defaultSteerable` path) untested
   end-to-end** (Mutation A). The 8 cases in
   `tests/steerable-spawn-cascade.test.ts` test
   `collapseSteerableCascade` in isolation; the production caller
   at `src/tools.ts:287–293` always passes `project: undefined,
   user: undefined` (relying on `mergeConfig` having pre-collapsed
   them into `cfg.defaultSteerable`). No integration test exercises
   `cfg.defaultSteerable=true → run.steerable=true` when per-call
   is unset. Recommend a sequel test in slice 5 or 6 that stubs
   `loadConfig` to return `{defaultSteerable: true}` and asserts the
   omit-steerable spawn path yields `run.steerable=true`. Easy add
   given the existing `tests/ensemble-spawn.test.ts` shape.

3. **Q3 EPIPE wording test pin uses regex, not character match**
   (`tests/ensemble-send-steerable.test.ts:312` — `/finished before
   steer was delivered/`). Plan §"Critic gates" mentions "W3
   string-pin witness" for slice 1 reject-reason strings; no
   equivalent pin landed for the Q3 EPIPE wording. The wording
   appears at 3 production sites with different parenthesised
   suffixes (`(stdin queue gone).` / `(${err.message}).` / `.`),
   making strict char-pinning awkward. Acceptable for v0.12 per
   the plan's W3 scope; flag for slice 7 closure if user-visible
   wording polish is on the agenda.

4. **Schema-validation layer not exercised by unit tests**
   (Mutation E). Standard pi-conductor pattern; pi's JSON-schema
   validator runs upstream of `tool.execute`. If the team ever adds
   an integration harness that hands params through pi's full
   pipeline (e.g. via `pi --mode rpc` against the conductor itself),
   schema enum tests would naturally appear. Not a v0.12 lift.

5. **Default-false symmetry case lives in `tests/ensemble-spawn.test.ts:113`**
   and reds under Mutation B. Builder's claim that it pins the
   default-cascade-fallthrough path is correct; without it, only
   the `steerable=true` branch would be observed.
