## 2. Run-record creation & lifetime

Every artifact and its writer:

| artifact | writer | when | content | re-derivable? |
|----------|--------|------|---------|---------------|
| `<runDir>/` | `mkdirSync` in `spawnRun` (`runs.ts:583`) and `enqueueOrSpawn` placeholder branch (`queue.ts:111`) | at spawn or enqueue | dir | n/a |
| `record.json` | `writeRecord(run)` (`runs.ts:1093-1100`) — called from `spawnRun:607`, `runPiSubprocess` close `:751`, `forceTerminate:1029`, `pauseRun:1068`, `resumeRun:1087` | spawn, every status change, every send finalize, pause, resume, force-kill | `toRunRecord(run)` JSON (`types.ts:268-289`) — id, persona, task, status, timestamps, usage, paths, **embedded full systemPrompt**, **embedded task prompt** | yes (in-memory Run) but **not from transcript alone** because it includes resolved persona-system-prompt + cwd |
| `transcript.jsonl` | `appendFile(...)` inside `processLine` (`runs.ts:770`) — fire-and-forget on every parsed JSON line from pi stdout | continuously while running | one JSON event per line — entire pi `--mode json -p` stream including thinking blocks, tool args, tool results | **this IS the source of truth** for messages/usage |
| `final.md` | `writeFinal(run)` (`runs.ts:1102-1108`) — called from `runPiSubprocess` close `:751`, `forceTerminate:1030` | terminal transition | `getFinalText(messages)` (last assistant text-content) or `"(no output)"` | yes — derivable from transcript |
| `session/<ts>_<uuid>.jsonl` | written by **the pi child subprocess** (not by conductor) when `--session-dir <runDir>/session/` is passed (`runs.ts:255, 613-618`) | as pi runs | pi's own session journal (a different schema from `transcript.jsonl`) | no — pi owns the format and `ensemble_send` resumes from it via `pi --session <path>` |
| `session/seeded.jsonl` | `seedSessionFile(...)` (`session-seed.ts:86`) for personas with `inheritContext != none` | at spawn before pi launches | filtered parent-context replay (one msg per line) | no — seeded path overwrites; `--session <seeded>` resumes from it |

Notes:

- **All writes are best-effort** (`try { } catch {}` swallows). A full disk silently drops persistence; runs continue in memory.
- `transcript.jsonl` writes are **fire-and-forget** (`void appendFile(...)` — `runs.ts:770`). No flush, no close, no fsync. Concurrent appendFile calls are not serialized in user-space; ordering relies on `appendFile` being O_APPEND under the hood (which it is on Linux). This is fine for a single writer per file.
- **Queued-but-never-spawned runs** still create a run dir + placeholder `Run` (`queue.ts:111-130`). If the queue is drained-by-cancel, those leave empty dirs. None observed in this dogfood inventory but the path exists.
- **`forceTerminate` does NOT discover sessionPath** — only the close-handler `finalize` closure does (`runs.ts:728-741`). On force-kill the close handler still fires (the SIGKILL race-fix at `runs.ts:734-741`) so the path is captured. *But*: spawn-failed runs (`runs.ts:683`) and the early `inspector-*` orphans never had a session, so their dir stays incomplete.

## 3. Pi session linkage (for ensemble_send)

The chain that ties a finished `Run` back to a pi session:

1. Spawn passes `--session-dir <runDir>/session/` (`runs.ts:255`). Pi creates one `<timestamp>_<uuid>.jsonl` inside.
2. On terminal, `discoverSessionPathIfMissing(run, sessionDir)` (`runs.ts:118-129`) reads `sessionDir`, finds the most-recently-modified `.jsonl`, and stores it as `run.sessionPath`.
3. On `ensemble_send`, `validateSendable` (`runs.ts:842-878`) requires `run.sessionPath` non-null and `existsSync(run.sessionPath)` true.
4. `sendToRun` calls `runPiSubprocess` with `pi --session <path>` (`runs.ts:236, 257`), and re-points `sessionDir = dirname(sessionPath)` so the post-resume close-handler refreshes `sessionPath` to the same file (mtime-bumped).

**GC implication.** The `session/<...>.jsonl` is the *only* artifact required for resumability. Deleting `transcript.jsonl` and `final.md` does NOT break `ensemble_send`. Deleting `session/` DOES (`validateSendable` returns `"session file is missing on disk"`).

**Coordination with pi's own session lifecycle.** The session file lives **inside `<runDir>/session/`**, NOT in `~/.pi/agent/sessions/`, so pi doesn't manage it — conductor owns it 100%. Pi's `~/.pi/agent/sessions/` is a separate cache for its own interactive sessions (parent conductor included). GC of conductor records is **independent** of pi's sessions root.

## 4. Existing config + flags

- `ConductorConfig` (`types.ts:75-93`) — **no retention/TTL/GC fields**. Closest neighbours: `defaultTimeoutMinutes`, `maxConcurrent`, `maxConcurrentWriteCapable`, `defaultMode`. v0.9 will need new fields.
- `DEFAULT_CONFIG` (`types.ts:107-117`) — same; nothing to flip on/off today.
- `RunRegistry` (`runs.ts:475-543`) — **in-memory, no eviction**. `register(run)` adds to `Map`; nothing ever removes. There's `countActive` / `countActiveBy` / `countQueued` but no `unregister` or `prune`. Across a session the registry monotonically grows; on `session_shutdown` reason≠"reload" the registry is dropped with the runtime, and the next session starts empty (it does **not** rehydrate from disk). The on-disk records are therefore the *only* persistent state.
- No `lifecycle.ts` module exists. Lifecycle work is split between `index.ts` (`session_start`/`session_shutdown` wiring), `shutdown.ts` (the SIGTERM-on-quit helper), and `runs.ts` (the run state machine).

## 5. References / consumers of run records

Inventory of everything that reads from `<runDir>/`:

| consumer | source | reads | impact of GC |
|---|---|---|---|
| `/conductor history` | `commands.ts:354-419` via `buildHistoryReport` (`history.ts`) | `record.json` + `final.md` of every dir under `runsRoot()`, sorted by `record.json` mtime | **direct consumer** — GC removes entries from the listing. By design the user wants this. |
| `ensemble_send` (LLM tool) | `tools.ts` → `sendToRun` → `validateSendable` (`runs.ts:842-878`) | `run.sessionPath` (in-memory) and `existsSync(run.sessionPath)` | breaks if `session/` is GC'd while the run is still in the *in-memory* registry. Same-session sends after GC would be impossible. |
| `ensemble_status` | `tools.ts:formatStatusForLLM` (~line 880-925) | **in-memory registry only** — `r.transcriptPath` is reported as a string but not read from | **safe** — pure in-memory. |
| `ensemble_spawn` result `transcriptPath` | `tools.ts:886, 935, 948` | path string only | safe — nobody reads it within conductor; the LLM may surface the path to the user. |
| `<sub-agent-completed>` notification card | `notifications.ts:48` | path string only — `<transcript>${run.transcriptPath}</transcript>` | safe; advisory pointer for the LLM/user. |
| Foreground spawn final message | `foreground-stream.ts:125-126` — `Transcript: ${run.transcriptPath}` | path string only | safe. |
| `/conductor doctor` | `doctor.ts` | **none** of the run records — only registry counts (`registry.list().length`, `countActive()`, `queue.size()`) | safe. |
| `RunRegistry` rehydration on startup | none | n/a | **conductor does NOT rehydrate from disk** — see §4. So GC is invisible to a fresh session. |
| Allocator collision check | `runs.ts:147` — `allocateRunId` checks `existsSync(runDir(id))` to avoid id reuse | dir presence | GC actually *enables* id reuse, which is fine — the id space is huge anyway. |

**Bottom line.** The only reader that walks the run-records tree is `/conductor history`. Everything else either holds in-memory references (`Run.sessionPath`) or just emits paths as strings. GC's blast radius is small and well-bounded.

## 7. Minimum viable GC schema (per run)

A GC pass needs to know, for each run dir, at least:

- **id** — `record.json` filename / dir name (e.g. `analyst-rj5i`).
- **status** — `record.status`. Determines: terminal vs running (don't GC running). Also enables status-tiered TTLs (e.g. keep `failed` longer than `completed`).
- **finishedAt** — `record.finishedAt` (or `startTime` fallback for runs that never finalized). Drives TTL math.
- **mtime of record.json** — fallback ordering signal already used by `/conductor history` (`commands.ts:402-411`). Survives clock skew between machines if records ever migrate.
- **sessionPath** — `record.sessionPath`. Tells GC whether the run is *resumable*; pinning policy may want to keep resumable-and-recent runs even past TTL.
- **size on disk** — `du` on the dir. Drives "biggest first" eviction modes and dashboard "you'd reclaim X MB" preview.
- **persona** — `record.persona`. Allows per-persona policies (e.g. designer transcripts >500 MB are auto-archived; inspector transcripts <1 MB stay forever).
- **pinned flag** — *does not exist today*. Would need to be either: (a) a `pinned: true` field added to `RunRecord` (and `toRunRecord`), or (b) a sentinel file `<runDir>/.pinned`. The latter avoids a schema bump.
- **partial-write marker** — *does not exist today*. A `running` record that is `> N hours` past `lastEventAt` with no live process is a stale orphan (3 such on this machine). Needs detection logic, not just a field.

Existing on-disk fields the policy can read directly from `record.json`: `id, persona, status, startTime, finishedAt, exitCode, errorMessage, usage, sessionPath, mode, model`.

Existing fields it would NOT have if it only read `record.json`: `lastEventAt` (in-memory only, see `Run.lastEventAt` at `types.ts:165-171` — *not* in `RunRecord`). If GC wants idle-detection it needs `lastEventAt` persisted, OR it falls back to `mtime(transcript.jsonl)` which is the next-best proxy.

## 8. Surprises / red flags

1. **Stale `running` records on disk.** Three runs (`inspector-2h68`, `inspector-frqv`, `inspector-j0u2`) have `status: "running"` in `record.json` but their host pi session is long-dead. `forceTerminate` runs in-process via `proc.kill()` — there is **no path** to flip a record to `killed` if the conductor process itself dies (Ctrl+C on the parent, OS reboot, OOM). On `session_shutdown` reason="quit" `handleSessionShutdown` (`shutdown.ts:67-83`) sends SIGTERM to children but **never re-`writeRecord`s** them, so the on-disk record is frozen at "running" forever. *GC must treat `running` records older than some threshold as orphaned and either (a) recover them to `killed` or (b) special-case them in TTL math.* This will get worse, not better, as users use conductor more.

2. **Designer / planner transcripts blow past 500 MB routinely.** Top 4 transcripts are all `designer` or `planner` (851M, 657M, 600M, 537M). One run is **8.5% of total disk usage** by itself. A naive "TTL = N days" policy is the wrong knob — the dominant signal is bytes, not age. A 30-day-keep-everything policy on this dogfood would be ~60 GB. *Recommend: dual policy (size cap + age cap), or per-persona overrides, with an explicit "cold-archive transcript, keep record+final" tier that reclaims 99% of the bytes for 1% of the metadata loss.*

3. **`transcript.jsonl` writes are fire-and-forget across multiple async calls.** `runs.ts:770` does `void appendFile(run.transcriptPath, line + "\n").catch(() => {})` per parsed line, with no in-flight tracking. If GC unlinks `transcript.jsonl` *while* a run is still streaming (e.g. user runs `/conductor gc` against a wedged-but-still-active run), the in-flight writes silently land on a now-unlinked inode, and on Linux the file just disappears when the writers close — but on the GC side, the dir's still there with `record.json` only, possibly inconsistent with the in-memory `Run`. *GC must skip non-terminal runs entirely, AND check `Run.proc` is undefined or `isTerminal(run.status)`.* In-memory state must win over on-disk state when they disagree.

4. **`record.json` embeds the entire task prompt and persona systemPrompt.** Sample `analyst-rj5i/record.json` is 7.3 KB — almost all of which is the verbatim task prompt and the analyst persona body (`runs.ts:597-606`, `types.ts:225-238`). Multiplied by 228 records = ~1.7 MB of duplicated persona bodies. Not a disk problem (record.json is 0.02% of total) but it means **`record.json` is NOT a safe file to share publicly** in case the task contains secrets (AWS account IDs, prod log queries, etc.). GC's "archive" mode should probably treat `record.json` with the same sensitivity as `transcript.jsonl`.

5. **No `RunRecord.pinned` field.** Adding pinning means either a schema bump (forward-compat fine, backward-compat for rollback ⚠ — old conductor will silently drop the field on next `writeRecord`) or a sidecar file. Sidecar is safer and matches `.pinned`'s "advisory marker" semantics, but doctor / history will need to learn to display it.

6. **Allocator collision check uses `existsSync(runDir(id))`.** `runs.ts:147`. After GC removes a dir, the id becomes available for reuse. `inspector-2h68` could theoretically come back as a fresh run with the *same id* but a different persona-task. This is not a correctness bug (registry uses string keys) but it confuses any tooling that expects ids to be stable forever (dashboards, vault notes citing run ids, etc.). *Recommend: change collision check to also exclude recently-GC'd ids, or use a longer-suffix pool.*

7. **The pi `session/` dir is OS-dependent.** Pi names files `<ISO-timestamp>_<uuid>.jsonl`. After `ensemble_send` the same file is reused — its **mtime updates but the path is stable**. `findSessionFile` (`runs.ts:80-103`) picks "most recent .jsonl" so it tolerates multiple files in the dir, but no production code creates that case. *If GC ever decides to compact `session/` (e.g. delete all but the latest jsonl), the path-stability invariant must be preserved or the in-memory `run.sessionPath` goes stale and the next `ensemble_send` fails the `existsSync` check.*

## Summary numbers

- **228 run records, 11 GB total, biggest single run is `designer-xdix` (853 MB).**
- **Transcripts dominate (10.0 GB / 98%).** Records (1.8 MB) and final.md (0.8 MB) are noise.
- **3 stale `running` records on disk** — orphan-recovery is required, not optional.
- **Status mix: 207 completed, 16 terminal-error (failed/killed/timeout), 3 stale running, 2 healthy in-flight at survey time.**
- **Only one consumer** walks the runs tree (`/conductor history`); GC blast radius is small.
- **No retention config, no eviction in `RunRegistry`, no rehydrate-on-start** — disk is the only persistent state and grows monotonically.
