## 1. Goals & non-goals

### Goals

- **Harness-enforced verification** for sub-agents that claim work is
  done. Today, `personas/builder.md` instructs the persona to run "the
  strongest focused check available" before returning a `commit_hash`,
  but enforcement is prompt-only. The witnessed pain class is builders
  returning `commit_hash` with incomplete verification — the same pain
  that motivated `verifier`-as-closer in `src/conductor-prompt.ts:248`.
  An `on_complete_hook` is the harness saying "I will run this command
  myself and refuse to mark you completed if it exits non-zero."
- **Surface failures structurally** to the parent conductor with the
  command, exit code, and a tail of stdout/stderr. The conductor LLM
  must be able to disambiguate "persona claims success but hook
  refused" from a normal completion or a normal failure.
- **Land the persona-frontmatter cascade layer** that v0.10 deferred
  (`docs/v0.10-watchdog-design.md` §3, D335). Cascade shape:
  `per-call > project config > user config > persona frontmatter > built-in default (none)`.
- **Compose with existing concurrency caps.** No worktrees, no Brazil
  detection, no merge-back protocol. The hook runs serially after the
  pi subprocess exits, inside the existing
  `maxConcurrentWriteCapable=1` invariant — there is no new race
  surface.
- **Fully testable.** Resolver is pure (cascade inputs → resolved
  hook). Enforcer is a thin shell around `child_process.spawn` with
  injectable deps. No `CONDUCTOR_LIVE_TESTS=1` requirement.

### Non-goals

- **Replacing critic / verifier review loops.** Hooks gate
  *mechanically verifiable* claims (does `npm test` pass? does `tsc
  --noEmit` succeed?). Critic loops gate *judgment* claims (is the
  approach right? is the test load-bearing?). They are layered, not
  alternatives.
- **Reverting commits on hook failure.** Builders may have already
  committed. Reverting would entangle us with git history we don't own
  (worktrees aren't shipped; v0.11 is single-tree). The hook failure
  is reported upward; the conductor decides whether to send the
  builder back via `ensemble_send` to undo or amend. Punting the
  revert decision is intentional and aligns with §11's "reviewer veto
  trumps producer push-back" — the conductor does the routing, not the
  hook.
- **Hooks that mutate the result envelope.** A hook can fail or pass.
  It cannot rewrite `final.md` or inject text into the persona's
  output. Anything we want to surface is in the structured `<hook>`
  block we add to the completion envelope.
- **Hooks for read-only personas.** The default-resolved hook for
  `oracle`, `redteam`, `inspector`, `analyst`, `profiler`, `scribe`,
  `verifier` is none. Configuration may opt them in (e.g. a project
  forces `oracle` to run a doc-lint), but no shipped persona declares
  one.
- **Tool-restricted hook execution.** Hooks run with the conductor's
  full env and shell. We are not sandboxing them. This mirrors the
  v0 stance from §2 of the original PRD ("we don't fake security
  boundaries pi cannot enforce").
- **Hooks during `ensemble_send` resumption.** A `send` continues an
  existing run; the hook's job is the *terminal* gate. It runs once
  per terminal transition, not per `send` round. (See §5 lifecycle.)
- **Persona-frontmatter opt-out of safety invariants.** v0.11
  introduces `on_complete_hook` to frontmatter, but
  `WRITE_CAPABLE_PERSONAS` (`src/personas.ts:38–42`) stays
  harness-owned. Frontmatter requests a hook; harness owns the
  lifecycle, the kill-tree, and the failure semantics.

## 2. The chosen lifecycle slot

The pi event protocol gives us four candidate seams for hook
execution (per `src/runs.ts:799–928`):

| # | Seam | Description | Decision |
|---|---|---|---|
| a | **Inside the pi sub-agent process** (child-side) | Persona itself shells out to the hook before its final assistant turn | **Reject.** Persona controls the hook → defeats the gate. Crashed sub-agents skip the hook entirely. |
| b | `applyEvent` returns `finalize` (event-driven) | Hook runs from the `processLine` path the moment we know the run is about to terminate | **Reject.** Hook IO would race with `proc.stdout` flush and the `proc.on("close")` arrival. Hook running while pi is still alive risks cwd contention if hook touches the same files pi is writing. |
| c | **`proc.on("close")` handler, before `finalize()`** | Hook runs synchronously after pi has exited, before we flip `run.status` to terminal | **SHIP.** Pi is gone; cwd is quiet; we own all the I/O. Race-free wrt close-handler vs `forceTerminate` because the existing `applyCloseHandlerTerminal` guard (`src/runs.ts:1100`) already arbitrates. |
| d | After `finalize()` resolves the `done` promise | Hook runs in a follow-up tick after the run is already terminal | **Reject.** The completion notification (`pushCompletionNotification`, `src/index.ts:274`) has already been sent to the parent before we knew the hook result. Conductor already saw "completed". Surfacing failure later is a worse UX than blocking the envelope until we know. |

**Chosen: (c) parent-side, post-`proc.close`, pre-`finalize`.** Concretely:

1. `proc.on("close", code => …)` fires (`src/runs.ts:920–923`).
2. **New step:** if the resolved hook is non-empty AND the natural
   close was a `completed` candidate (exit 0, no `forceTerminate`
   already settled), spawn the hook as a child process with the
   contract from §3.
3. Wait for hook to terminate (or hit hook-timeout → SIGTERM → 2s →
   SIGKILL, mirroring `forceTerminate`'s shape).
4. If hook exit 0: call existing `finalize("completed", 0)` path
   unchanged. Persisted record + envelope identical to today.
5. If hook exit non-zero, killed, or timed out: call new
   `finalize("hook_failed", ...)` path. The completion envelope
   carries a `<hook>` block (§4).

Critical: skip the hook entirely when the natural close is already
non-zero (the persona crashed or returned `failed`), when
`forceTerminate` already settled the run (killed/timeout/stalled), or
when the hook itself is unset. The hook is a *quality gate on success
claims*, not a generic post-mortem hook.

### Why not child-side (option a) — extended rationale

Builders today already shell out to test commands inside their pi
subprocess. The whole point of v0.11 is that we observed builders
*skip* that step or claim its results without running it. Putting the
hook inside the persona's pi process means:

- The persona prompt would need to declare it. Personas already lie
  about prompt-declared verification.
- Crashes (segfault, timeout, OOM) bypass the hook silently.
- The persona could `exec`-replace itself or run the hook in a
  background subshell.

Parent-side is the only seam where the conductor is the source of
truth for whether the hook ran.

## 3. The hook contract

### 3.1 Frontmatter shape

```yaml
on_complete_hook: "npm test --silent && npx tsc --noEmit"
on_complete_hook_timeout_seconds: 600    # optional; default 300
```

**Shell string, not array form.** Rationale:

- The frontmatter parser (`src/personas.ts:84–166`) already accepts
  scalars and string-lists; adding a structured object is one more
  parse path and one more validation surface.
- All shipped personas that would declare a hook will compose
  multi-step commands (e.g. `npm test && tsc --noEmit && lint`). The
  `&&` short-circuit is shell-native; an array form would require us
  to re-implement it.
- Shell-injection hazard is moot: the hook *is* the persona/project's
  declared command. There is no untrusted input being interpolated
  into the shell string. (Compare watchdog's `kill_on_stall: true`
  arg: also a trust-the-config posture.)

We pin this with a counter: if a future persona needs structured args
(e.g. `{command: [...], cwd: "..."}`), we extend to an *additional*
form, not replace the string form.

### 3.2 Unset vs empty

| Frontmatter | Resolution |
|---|---|
| `on_complete_hook` field absent | unset → next layer in cascade |
| `on_complete_hook: ""` (empty) | **explicit-disable** → bypass cascade entirely; no hook runs |
| `on_complete_hook: "true"` | shell command `"true"` (always-pass) — degenerate but valid |
| `on_complete_hook: "npm test"` | run `bash -c "npm test"` |

Empty-string-as-explicit-disable is the only way for a project config
to nullify a persona-frontmatter hook. (`null` is rejected by our
scalar parser; `false` would parse as boolean, not string, and we
treat that as a parse error.) This mirrors `defaultMode: "off"`'s
explicit-string sentinel pattern.

### 3.3 Cascade resolution

Layer precedence (highest wins, first non-undefined non-empty stops):

```
1. per-call               ensemble_spawn(..., on_complete_hook: "...")
2. project config         <project>/.pi/conductor/config.json :: personaOverrides[name].onCompleteHook
3. user config            ~/.pi/agent/extensions/conductor/config.json :: personaOverrides[name].onCompleteHook
4. persona frontmatter    personas/<name>.md :: on_complete_hook
5. built-in default       (none — no shipped persona declares a hook)
```

Empty string at any layer means **explicit-disable** and short-circuits
the cascade with a result of "no hook". Undefined means "fall
through". This is a cleaner pattern than v0.10's watchdog cascade,
where `kill_on_stall` is boolean-only and has no explicit-disable
sentinel — boolean cascade falls through `undefined` only. We adopt
the explicit-disable rule because string hooks need a cancel path that
the per-call layer can express.

A new pure helper `resolveOnCompleteHook(runArgs, cfg, persona)` lives
next to `resolveKillOnStall` (`src/watchdog.ts:275`) — same shape:

```ts
export function resolveOnCompleteHook(
  perCall: HookSpec | undefined,
  cfg: ConductorConfig,
  persona: Persona,
): ResolvedHook | undefined {
  // Returns undefined when no hook should run.
  // Returns { command: string, timeoutSeconds: number, source: HookSource } otherwise.
}
```

The pure-helper placement lets the same WDD mutation-witness pattern
the v0.10 chain used (`docs/wdd.md`, parallel-formula rule) pin the
cascade arithmetic.

### 3.4 Execution

- **Spawn:** `child_process.spawn(command, { shell: true, cwd, env })`.
  Shell mode = `/bin/sh -c <command>` (or `cmd.exe /d /s /c` on
  Windows; we don't support Windows today and won't here).
- **cwd:** the run's `run.cwd` (`src/runs.ts:706`). This is the parent
  conductor's cwd at spawn time, the same directory the persona was
  working in. Builders commit to that tree; tests must run there.
- **env:** inherited from the conductor process plus injected
  variables (§3.5).
- **stdio:** `stdout` and `stderr` piped, captured to
  `runDir(id)/hook.log`.
- **Timeout:** `on_complete_hook_timeout_seconds` from cascade,
  default 300s. On timeout: SIGTERM, 2s grace, SIGKILL. Mirrors
  `forceTerminate` (`src/runs.ts:1132`).
- **Detached:** `detached: true` + `proc.unref()`-style cleanup so the
  hook is owned by its own process group; SIGTERM kills the group, not
  just the shell. Same pattern as the watchdog kill path.

### 3.5 Injected env contract

```
CONDUCTOR_RUN_ID=<run.id>
CONDUCTOR_PERSONA=<persona.name>
CONDUCTOR_FINAL_TEXT_PATH=<run.finalPath>     # path to final.md (already written by writeFinal earlier in finalize? — see §5)
CONDUCTOR_TRANSCRIPT_PATH=<run.transcriptPath>
CONDUCTOR_RUN_DIR=<runDir(run.id)>
CONDUCTOR_HOOK_LOG=<runDir(run.id)>/hook.log  # the file the hook is being written to (echo at own risk)
CONDUCTOR_PARENT_CWD=<run.cwd>
```

`CONDUCTOR_COMMIT_HASH` is **not** injected — extracting it from the
final text would require a parser the conductor doesn't currently
own. Hooks that care about the commit can `git rev-parse HEAD` from
`CONDUCTOR_PARENT_CWD`. Documented in the persona reference.

The env contract is documented in `personas/builder.md` and
`personas/simplifier.md` as comments, not in shipped frontmatter.
(See §3.6.)

### 3.6 No defaults shipped + cheap-rerun guidance

**Tune hook commands to be cheap to re-run.** Hooks fire on every
terminal transition, including the second/third pass through a
builder ⇄ critic loop driven by `ensemble_send` (§4.6). A 90-second
`npm test` adds 4–5 minutes across a 3-iteration loop. The
`personas/builder.md` recommended-hook comment (slice 5) calls this
out explicitly:

```
# Recommended (cheap re-run):
#   on_complete_hook: "npx tsc --noEmit"
# Higher-confidence but slower; only if your test suite is fast:
#   on_complete_hook: "npm test --silent && npx tsc --noEmit"
```

The shape is the same as today's CONTRIBUTING.md guidance for
pre-commit hooks: cheap, deterministic, idempotent.



No shipped persona declares a hook. Justification:

- A shipped default could fire on someone's repo with `npm test`
  hanging on a missing dependency the first time they spawn a
  builder. That surprise is worse than the v0.7 "ON-by-default"
  surprise we already reverted.
- `personas/builder.md` and `personas/simplifier.md` will document a
  *recommended* hook in a comment block: "for npm projects, set
  `on_complete_hook: 'npm test --silent && npx tsc --noEmit'` in your
  project config."
- Users opt in via project config (`<project>/.pi/conductor/config.json`).

This is the same default-off stance v0.10 took for `kill_on_stall`
(D5: `defaultKillOnStall: false`). Revisit only after dogfood evidence
shows the gate's value across a meaningful sample of repos.

## 4. Failure model

### 4.1 New terminal status: `hook_failed`

We add `"hook_failed"` to `RunStatus` in `src/types.ts`. Reasoning:

- Reusing `"failed"` with a structured reason would force every
  reader of `Run.status` to discriminate via a side field. Existing
  readers are: `src/history.ts`, `src/notifications.ts`,
  `src/widget.ts`, `src/gc/policy.ts`, `tests/personas.test.ts` (the
  status snapshot), and the `isTerminal` predicate. All of them
  treat the status as the discriminator today. Adding a new status
  preserves that contract.
- The conductor LLM needs to disambiguate. The completion envelope
  surfaces `<status>hook_failed</status>` directly; reusing `failed`
  would require it to inspect `<error>` text.
- GC policy already has per-status TTL (`failedTtlDays`); we can give
  `hook_failed` its own TTL if dogfood shows the diagnostic value
  warrants it (default: same as `failedTtlDays`).

`hook_failed` is added to `TERMINAL_STATUSES` and `isTerminal`
returns true for it. UI surfaces (`history`, widget, panel) treat it
as a distinct color/glyph (§6).

### 4.2 The `<hook>` block in the completion envelope

Rendered by `formatCompletionNotification` (`src/notifications.ts:18`)
when the run terminated with `hook_failed` OR when a hook ran and
passed (so the parent has positive evidence either way). New block:

```xml
<hook>
  <command>npm test --silent &amp;&amp; npx tsc --noEmit</command>
  <exit-code>1</exit-code>
  <duration>23s</duration>
  <log-path>/path/to/runs/builder-7f3a/hook.log</log-path>
  <tail bytes="3984" lines="42">
    ...last N lines of stdout/stderr...
  </tail>
</hook>
```

**Tail cap: 50 lines OR 4 KB, whichever first** (matches the existing
truncation rule in `bash` tool result handling). Capped well below the
per-transcript GC cap of 100 MB (`gc.transcriptSizeCapBytes`).

Render slot: between `<warning>` (if any) and `<result>`. Hook is a
*structural* result the parent must read before trusting `<result>`.

### 4.3 What about commit_hash?

If the persona claimed `commit_hash: abc1234` in `final.md` and the
hook then failed:

- We do NOT revert the commit. Reverting commits we don't own is
  dangerous — branch state, push history, peer reviewers' local
  fetches are all affected.
- The `<hook>` block carries the failure signal. The conductor
  reads "claimed completed but hook_failed" and decides:
  - Default §11 path (recommended): `ensemble_send(builder_id, "hook
    failed: <tail>; revise per these results")`. Same loop cap as
    builder ⇄ critic (≤3 iterations).
  - User-directed escape: tell the user what happened and ask
    whether to revert manually.

The conductor-prompt §11 addendum (§7 below) makes this explicit.

### 4.4 Hook crash / timeout

| Hook outcome | Resulting `Run.status` | `errorMessage` field |
|---|---|---|
| Exit 0 | `completed` (unchanged) | unset |
| Exit non-zero | `hook_failed` | `"hook exited <code>"` |
| Timed out | `hook_failed` | `"hook timed out after Ns"` |
| spawn error (binary not found) | `hook_failed` | `"hook spawn failed: <err>"` |
| Killed by signal (rare) | `hook_failed` | `"hook killed by SIG<X>"` |

We do not invent a `hook_timeout` or `hook_killed` status — those are
implementation details inside the same gate-failed bucket. The
discrimination shows up in the `<hook>` envelope, not the status.

### 4.4a Hook handle plumbing on `Run` (forceTerminate compatibility)

The hook's child process needs first-class plumbing on `Run` so
every existing terminal short-circuit path can kill it. New optional
field:

```ts
export interface Run {
  // ... existing fields ...
  /**
   * v0.11 on_complete_hook (slice 2): handle to the in-flight hook
   * subprocess. Set when the slice-2 enforcer spawns the hook;
   * cleared on hook exit (close + error paths).
   * `forceTerminate` and any equivalent terminal short-circuit
   * (`cancelRun`, watchdog hard-kill with `kill_on_stall=true`,
   * session_shutdown) MUST SIGTERM this handle (with the SIGKILL
   * fallback at 2s) when set, mirroring the existing `run.proc`
   * pattern. Not persisted in `RunRecord` (process handles never
   * are; same as `run.proc`, `run.timeoutTimer`).
   */
  hookProc?: ChildProcess;
}
```

**`forceTerminate` audit (`src/runs.ts:1132`):** today the function
kills only `run.proc`. v0.11 adds a parallel kill branch on
`run.hookProc` (SIGTERM → 2s → SIGKILL), runs after the existing
`run.proc.kill("SIGTERM")` block, idempotent (try/catch wraps both
sides). The same audit covers any equivalent path that today only
reaches `run.proc` — the timeout timer, `cancelRun`, the watchdog
hard-kill enforcer (`src/watchdog.ts`'s `kill` callback eventually
hits `forceTerminate`), and shutdown sweeps. Wiring centralises in
`forceTerminate`; the other paths inherit by calling it.

**Hook exit handler idempotency invariant.** When the hook's
`close`/`error` callback fires, it MUST check `isTerminal(run.status)`
(or equivalently `applyCloseHandlerTerminal`'s false-return)
*before* using the hook result to drive `finalize`. If the run is
already terminal — `forceTerminate` ran while the hook was in
flight — the enforcer logs a warning, clears `run.hookProc`, drops
the hook result, and exits. No `applyCloseHandlerTerminal` call.
No `finalize` call. The race-bail path through
`applyCloseHandlerTerminal` (`src/runs.ts:1100`) cannot be reused
here because we are not the close handler — the bail must short-circuit
on the hook side directly.

### 4.5 Watchdog ⇄ hook interaction

The watchdog observes `run.lastEventAt` while `run.status === "running"`
(`src/watchdog.ts:130`). The hook executes after `proc.close` and
before status flips to terminal — but `run.status` is still `"running"`
during hook execution, and `lastEventAt` will not advance because the
pi process is gone and we don't synthesize fake events.

**Without intervention this would trip the watchdog on slow hooks.**
A 3-minute `npm test` would fire the soft-stall advisory the moment
the hook crosses 120s of silence.

**Mitigation: skip stall classification during hook execution.** Add a
boolean `Run.hookExecuting?: boolean` (default undefined/false). The
watchdog detector (`evaluateRun`, `src/watchdog.ts:115`) returns
`{kind: "none"}` and `classifyStall` returns `null` when
`run.hookExecuting === true`. The enforcer sets this flag right before
spawning the hook and clears it before calling `finalize()`. Note
that `hookExecuting` is the watchdog-suppression flag; `hookProc`
(§4.4a) is the kill handle. Two distinct concerns, two distinct
fields — both cleared in the same hook-exit path.

Document the hook timeout (default 300s, configurable) as the
upper bound on this silence; the watchdog hard threshold (default
600s) is no longer the right gate for a hook anyway because the hook
is a known-finite step, not a heartbeat-based concern.

### 4.6 `ensemble_send` resumption

Hooks fire *only* on terminal transitions. An `ensemble_send` that
revives a finished run (`src/runs.ts` resumeFinishedRun path) flips
status from terminal back to `"running"` — when *that* run finishes,
the hook runs again. This is the desired behavior: each terminal
transition is a fresh gate.

A common case: hook fails → conductor `ensemble_send`s the builder
with the hook tail → builder amends the commit → hook re-runs. If the
amended commit still fails, the second `hook_failed` envelope shows
the new failure tail. (§7 covers the loop semantics.)

## 5. Lifecycle integration in `runs.ts`

The `finalize` closure (`src/runs.ts:821–882`) is the integration
point. Today it executes:

```
1. clearTimeout
2. processLine(buffer)            (drain pending stdout)
3. applyCloseHandlerTerminal      (status → terminal; race guard)
4. errorMessage fallback for failed
5. applySubstanceCheck            (v0.8.1 advisory)
6. discoverSessionPathIfMissing
7. run.proc = undefined
8. registry.notify
9. proc.kill                       (idempotent)
10. writeRecord + writeFinal       (parallel)
11. opts.onComplete                (push notification)
12. donePromiseResolve
```

We insert **between (2) and (3)** when the natural close was exit-0
AND a hook is resolved:

```
2.a runHookIfNeeded(run, resolvedHook)
    - If resolved is undefined OR exitCode !== 0 OR run already terminal: no-op
    - Spawn child_process.spawn(command, {shell: true, cwd, env, detached: true})
    - Stream stdout/stderr to runDir(id)/hook.log
    - Await with timeout (SIGTERM → 2s → SIGKILL on overshoot)
    - On exit:
        - exitCode 0  → set run.hookResult = { passed: true, command, exitCode, duration, logPath, tail }
        - non-zero    → set run.hookResult = { passed: false, ... }; flip the terminal we'll request below from "completed" to "hook_failed"
2.b If hookResult.passed === false:
    - Override the `terminal` argument to applyCloseHandlerTerminal: pass "hook_failed" instead of the natural-close terminal.
    - Set run.errorMessage from §4.4 table.
```

The `applyCloseHandlerTerminal` guard (`src/runs.ts:1100`) still
arbitrates the race with `forceTerminate`. If `forceTerminate` settled
the run while the hook was still spawning, two things must hold:

1. **`forceTerminate` kills the hook.** The new `run.hookProc`
   branch (§4.4a) inside `forceTerminate` SIGTERMs the hook, then
   SIGKILLs after 2s. No orphaned hook subprocess.
2. **Hook-exit idempotency.** When the killed hook's `close`/`error`
   callback fires (post-SIGTERM/SIGKILL), `isTerminal(run.status)` is
   already true. The enforcer logs a warning, clears `run.hookProc`,
   drops the hook result. It does NOT call `applyCloseHandlerTerminal`,
   does NOT call `finalize`, does NOT mutate `run.status`.

This pair is the slice-2 critic gate's headline test (§9).

`run.hookResult` is a new optional field on `Run` and `RunRecord`:

```ts
export interface HookResult {
  passed: boolean;
  command: string;
  exitCode: number | null;       // null if killed by signal or spawn error
  durationMs: number;
  logPath: string;               // absolute path to hook.log
  tailText: string;              // last 50 lines / 4 KB
  tailBytes: number;
  tailLines: number;
  failureKind?: "exited" | "timeout" | "spawn_error" | "signal";
}
```

Persisted on `RunRecord`. Read by `formatCompletionNotification`,
`history.ts`, `gc/policy.ts` (no change — `hook.log` lives under
`runDir(id)/` and is swept by the existing per-run sweep), and the
doctor surface.

### 5.1 `final.md` ordering

`writeFinal` writes the persona's last assistant text. Today it runs
in step (10) above. We must call it BEFORE the hook spawns so that
`CONDUCTOR_FINAL_TEXT_PATH` resolves to a real file when the hook
inspects it.

Move `writeFinal(run)` to step (2.a-pre): write `final.md` first,
then spawn the hook. `writeRecord` stays at step (10). This is a
small reorder; `writeFinal` is idempotent and pure.

### 5.2 GC contract

`hook.log` lives at `runDir(id)/hook.log`. `gc/inventory.ts` lists
files under each `runDir(id)/`; the cold-archive sweep already
removes the dir's transcript and log files at TTL. **No change to
the GC code.** `hook.log` is GC'd by the same TTL that governs the
rest of the run record.

Confirmed by reading `src/gc/policy.ts` — the sweep is dir-scoped, not
file-name-aware. New files under `runDir(id)/` are picked up
automatically.

## 6. UX surfaces

### 6.1 Doctor (`/conductor doctor`)

New section "Hooks":

```
hooks:
  builder:    npm test --silent && npx tsc --noEmit  (project config)
  simplifier: npm test --silent                      (user config)
  oracle:     (none)

```

Best-effort PATH check: for each resolved hook, attempt to extract
the leading binary (split on whitespace, take first token, ignore
shell builtins) and `which`-check it. Warn if missing:

```

```

Cascade source is shown in parens so users can find and edit the
right config. Mirrors how `defaultMode` and `gc` surface in doctor
today.

`hook_failed` runs render distinctly from `failed`:

```
✗ builder-7f3a    failed       2m18s   // exit 1
⊗ builder-9k2x    hook_failed  3m42s   // npm test exited 1
✓ builder-mq8c    completed    1m07s
```

The glyph (`⊗`) and color slot (yellow vs red) make the gate failure
visually distinct from a persona crash.

### 6.3 Ensemble panel widget

Brief `· hook` glyph during hook execution phase (post-`proc.close`,
pre-`finalize`). The widget already polls the registry; we add
`run.hookExecuting === true` as a render condition. Falls between
the existing live-tool-call slot and the `· STALLED Ns` glyph. When
hook exits, glyph disappears and the row settles to the terminal
form.

### 6.4 Slash commands

**No new slash command.** A new top-level `/conductor hooks status`
would duplicate doctor's "Hooks" section and add more surface to
maintain. Instead:

- `/conductor doctor` shows the resolved hook table.
- The completion envelope surfaces per-run hook results.
- We keep `/conductor watchdog status` separate — watchdog and hooks
  are orthogonal concerns.

### 6.5 Conductor-prompt §11 addendum

A new paragraph slots near §11's "Verifier briefs MUST be
self-contained" block in `src/conductor-prompt.ts:248`:

```
**`hook_failed` handling.** When a sub-agent terminates with
`<status>hook_failed</status>`, the conductor's recorded
`<hook><exit-code>` and `<tail>` carry the harness-enforced gate
result. Default routing: `ensemble_send(producer_id, "hook failed
with: <tail>; revise per these results")`, capped at the same
≤3-iteration loop semantics as `builder ⇄ critic`. Do NOT spawn a
fresh `critic` — the hook is a stronger, mechanically-grounded
signal than a critic review. After 3 hook_failed iterations,
escalate to the user with the failing command and tail, the same way
you'd escalate a stuck critic loop.
```

This is a single paragraph, additive only, no rewrite of existing
§11 content.

## 8. Open questions & risks

### Open questions

1. **Should `hook_failed` get its own GC TTL?** Default is to share
   `failedTtlDays` (60 days). Diagnostic value of preserving hook tails
   may warrant 90 days. **Punt to dogfood evidence**; ship with
   `failedTtlDays`.
2. **Should the hook see the full final.md, or just a path?** Current
   design: path only (env var). Inlining would inflate env. **Keep
   path-only.**
3. **Is there a use case for hook_passed → still-fail-the-run?** No
   today. The hook is a binary gate. Punt.
4. **Multi-step hooks: should we support a list of commands with
   per-step status?** No — `&&` and `;` already chain shell commands
   adequately. Adding structured multi-step would force us to design
   a per-step stdout capture model. Defer to a hypothetical v0.12.
5. **`ensemble_send` resumption: does the hook re-run on every
   terminal, or only on the first?** Re-runs on every terminal (§4.6).
   That's the correct behavior because `send` is the conductor saying
   "try again"; the gate should re-fire.
6. **Hooks for `inherit_context: none` personas (oracle, redteam,
   etc.):** Allowed by the cascade, but the `<hook>` block does not
   inject into the *next* persona's context — it's in the result
   envelope. `inherit_context: none` personas don't see envelopes
   either. So a hook on an oracle is purely a parent-conductor
   signal. Document this; don't restrict.

### Risks

1. **R1 — Hook + git interactions on amended commits.** If a hook
   runs `git commit --amend` (it shouldn't, but the persona's prior
   commit might), state changes between hook spawn and `finalize`.
   Mitigation: doc rule "hooks must be read-only; don't `git commit`
   inside a hook." Defense in depth: the hook's exit code is the
   only thing we read; we don't re-inspect tree state.
2. **R2 — Hook timeout + watchdog hard kill race.** Mitigated by
   `run.hookExecuting` flag (§4.5). Critic gate on slice 2 must pin
   this with a test that runs a 200s hook against a 120s soft
   threshold and asserts no advisory is emitted.
3. **R3 — Hook spawns a daemon that survives.** `detached: true` +
   process-group SIGTERM mitigates. SIGKILL fallback at 2s mirrors
   `forceTerminate`. Same risk profile as v0.10's hard-kill path.
4. **R4 — `hook.log` blows past 100 MB.** A pathological hook
   (`yes` redirected) fills the disk. Mitigation: cap hook.log writes
   at 10 MB during streaming; if exceeded, kill the hook with
   `hook_failed`/`runaway_output`. Slice 2 builds in the cap.
5. **R5 — Hook running while `ensemble_pause` was active.** Pause
   targets the pi process via SIGSTOP. After pi closes, pause is
   irrelevant. But what if `ensemble_pause` is called *during* hook
   execution? Current design: pause only affects `run.proc`, which
   is the pi process. Hook subprocess is separate. Document: pause
   does not affect hooks. (Could be revisited if dogfood shows a
   need.)
6. **R6 — Pre-commit hooks vs `on_complete_hook`.** The host repo's
   git pre-commit hook (CONTRIBUTING.md mandates one for
   pi-conductor itself) already runs `npm test` on `git commit`.
   v0.11 hook would re-run `npm test`. Net is double-run cost on
   builder runs in this repo. Mitigation: if dogfooding pi-conductor
   on itself, set the project hook to a lighter check
   (`tsc --noEmit` only) and let the git pre-commit do the test
   sweep. Documented in `personas/builder.md` recommendation
   comment.
7. **R7 — Cascade explicit-disable surprise.** A user sets
   `on_complete_hook: ""` in a project config to disable a user-level
   hook; they later want to re-enable per-call. Per-call is highest
   priority and overrides — that's what they want. But if they only
   set `on_complete_hook: ""` per-call to skip a hook for one spawn,
   the next spawn (without the per-call override) reverts to the
   project hook. Document the layered behavior clearly.
8. **R8 — Adding `RunStatus` value breaks readers.** Every consumer of
   `RunStatus` must now handle `hook_failed`. Oracle-extended audit
   list with risk classification:

   **Compile blockers (TS exhaustive switch — *must* be touched in
   slice 1a or `tsc --noEmit` fails):**
   - `src/transcript-style.ts:43` — `statusColorSlot(status)` switch
     with no `default`. New union member is a TS error.

   **Type-shape constraints (`Record<RunStatus, T>` — must add a key
   or typing fails):**
   - `src/status-glyph.ts:13` — `STATUS_GLYPH: Record<RunStatus, string>`.
     Pick a glyph for `hook_failed` (proposed `⊗`) at slice-1a time.

   **Silent miscategorization risk (literal-equality filter that
   bypasses `isTerminal` — code compiles but behaves wrong):**
   - `src/commands.ts:833–839` — watchdog-status command's
     non-terminal filter. Without `r.status !== "hook_failed"`,
     hook-failed runs are misclassified as active and would appear in
     the watchdog-status table.

   **Non-exhaustive (graceful soft-compat — old branches still work
   but should be updated for full coverage):**
   - `src/history.ts`, `src/widget.ts`, `src/notifications.ts`,
     `src/gc/policy.ts`, `src/doctor.ts`, `src/event-handler.ts`,
     `src/foreground-stream.ts`, `tests/personas.test.ts` (status
     snapshot).

   **External soft-compat (no in-repo touch, but changelog callout
   needed):**
   - External `record.json` consumers — vault dashboards, the user's
     pi-dashboard project, any CR-tracking that scrapes
     `runs/<id>/record.json`. They will encounter the new terminal
     status string. Slice 5's changelog entry calls this out.

   Slice 1a's audit run is a `tsc --noEmit` followed by a `grep -rn
   '"completed"\|"failed"\|"killed"\|"timeout"' src/` sweep to catch
   literal-equality readers the type system can't see.

9. **R9 — `final.md` mtime ordering contract change.** v0.11 moves
   `writeFinal(run)` from after the hook to before it (§5.1). The
   reorder is mechanically safe — `writeFinal` is idempotent and
   pure — but the *observable contract* changes: external watchers
   that today treat `final.md` mtime as a "run is in a terminal
   state" signal will fire prematurely under v0.11, before the run
   is actually terminal (it might still go to `hook_failed`).
   Mitigation: changelog entry stating the contract change. If
   dogfood evidence shows real consumers depend on the old ordering,
   add a `.complete` sentinel file written *after* `finalize` flips
   status to terminal — external watchers can poll for that file
   instead of `final.md` mtime. Defer the sentinel to v0.11.1 unless
   evidence demands it sooner.
