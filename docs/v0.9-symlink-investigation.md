# v0.9 — Legacy `extensions/conductor` symlink investigation

Read-only investigation into the silent persona-discovery failure observed
2026-05-18. The user's session at session-start saw `ensemble_list` return
0 personas; tracing showed pi-conductor was loaded twice (once via the
`packages[]` array, once via an `~/.pi/agent/extensions/conductor/index.js`
symlink) and the symlink-load resolved `import.meta.url` to a path under
which `../personas/` did not exist.

HEAD: `69f886c` (clean). Source-of-truth commits cited inline.

---

## 1. Findings — who created the symlink

**Smoking gun: our own `README.md:70` and `HANDOVER.md:316` document the
exact `mkdir + ln -s` install pattern that produced the broken symlink.**

```text
$ grep -n "ln -s" README.md HANDOVER.md
README.md:70:   ln -s /path/to/pi-conductor/src/index.ts ~/.pi/agent/extensions/conductor/index.ts
HANDOVER.md:316: ln -s ~/scratch/pi-conductor/src/index.ts ~/.pi/agent/extensions/conductor/index.ts
```

Verbatim README block (`README.md:65–71`):

```bash
# Or auto-load every session
mkdir -p ~/.pi/agent/extensions/conductor
ln -s /path/to/pi-conductor/src/index.ts ~/.pi/agent/extensions/conductor/index.ts
```

That predates the `packages[]`-array install path landed in v0.7 / v0.8 and
was never removed. The actual symlink we found pointed at
`/home/samfp/scratch/pi-conductor/dist/index.js` rather than `src/index.ts`
— a one-character variant of the documented incantation, almost certainly
created by an agent following the README pattern but substituting the
manifest-declared entry (`package.json` declares `"pi": { "extensions":
["./dist/index.js"] }`, so `dist/index.js` was the natural choice).

### Suspects ruled out (with evidence)

| Suspect | Why ruled out |
|---|---|
| `pi-coding-agent` host symlinking on extension load | `grep -rln 'symlink\|symlinkSync\|fs.symlink' dist/` in `@earendil-works/pi-coding-agent/dist/` returns 0 hits in any non-vendor file. Migration code in `dist/migrations.js:121–137` only `renameSync`s `commands/` → `prompts/`. |
| `pi update` for our `local`-type entry | `core/package-manager.js:944–1011` `resolveLocalExtensionSource` only adds the path to an in-memory accumulator. No symlink, no copy. |
| `pi install npm:<x>` on any package | `core/package-manager.js:870–880` `installNpmBatch` runs `npm install -g` into the global node_modules root. Doesn't touch `~/.pi/agent/extensions/`. |
| `pi-extmgr` `installPackageLocally` | `pi-extmgr/src/packages/install.ts:236–253` `cp -r`s an extracted tarball; no symlinks. Also: cache mtime `~/.pi/agent/.extmgr-cache/metadata.json` is `Feb 25 19:49` — extmgr hasn't run recently. |
| `oh-pi` postinstall | Not installed: `find / -name oh-pi -type d 2>/dev/null` returns nothing. Only present as an entry in the (stale) extmgr metadata cache. |
| Repo hooks (`hooks/pre-commit` + `scripts/install-hooks.sh`) | `grep 'ln -s\|symlink' hooks/ scripts/ tools/` → 0 hits. Hook only does `npm run build` + `npm test` + `git add dist/`. |

The symlink mtime falling between `tmux a` (18:29) and the explicit `rm`
(19:26) means whatever produced it ran from inside an interactive pi
session — most likely an agent acting on the README install snippet (the
exact text exists in two places in this repo and is copy-pasteable).

### Why the symlink was *probably already there* before today's session

The user's prior session (`/tmp/v0.8.2-handoff.md`) reported conductor
"working" — but that was before commit `3949fc4` ("build: compile to
dist/index.js to bypass jiti per-startup transpile") landed yesterday and
*first produced* `dist/index.js`. If a stale symlink targeting
`dist/index.js` existed earlier, it would have been a *broken* symlink
(target nonexistent) and pi's loader would silently skip it (the
`existsSync(realpath)` check in `package-manager.js:392`). The dist
artifact was first produced ~19:00 May 17 UTC by `3949fc4`. So the
symlink could have been created any time before that and only became
*active* once `dist/index.js` started existing — which matches the
"worked yesterday, broken today" pattern.

This makes the precise creation timestamp moot for the fix decision: the
*pattern* (cargo-culted README install) is the bug, not the specific date.

## 2. Failure mechanism

### Path 1: legitimate load (via `packages[]`)

`~/.pi/agent/settings.json:packages[]` includes
`/local/home/samfp/scratch/pi-conductor`. pi's `package-manager.js:993`
treats it as a `local`-type source, calls `resolveLocalExtensionSource`,
which walks the `pi.extensions` manifest array (`./dist/index.js`),
`existsSync`s `/local/home/samfp/scratch/pi-conductor/dist/index.js`,
loads it. `import.meta.url` resolves to
`file:///local/home/samfp/scratch/pi-conductor/dist/index.js`.

`builtinPersonasDir()` (`src/personas.ts:34–39`):

```ts
const here = fileURLToPath(import.meta.url);
return resolve(dirname(here), "..", "personas");
```

→ `dirname("/local/.../pi-conductor/dist/index.js")` = `/local/.../pi-conductor/dist`
→ `resolve(..., "..", "personas")` = `/local/.../pi-conductor/personas`
→ `existsSync` true → 16 personas loaded. ✅

### Path 2: legacy load (via `~/.pi/agent/extensions/`)

The host *also* auto-discovers any `*.{ts,js}` files and any directory
containing an `index.{ts,js}` under `~/.pi/agent/extensions/`
(`package-manager.js:388–438` `collectAutoExtensionEntries`). The
symlink at `~/.pi/agent/extensions/conductor/index.js` matched the
`indexJs` branch at `:383`, so `entries.push(indexJs)`.

When the host loads via that path, `import.meta.url` is the *symlink
path* (`file:///home/samfp/.pi/agent/extensions/conductor/index.js`)
because Node's loader does **not** canonicalize symlinks by default:

```text
fileURLToPath(import.meta.url) =
  /home/samfp/.pi/agent/extensions/conductor/index.js
dirname(...)                   =
  /home/samfp/.pi/agent/extensions/conductor
resolve(..., "..", "personas") =
  /home/samfp/.pi/agent/extensions/personas
existsSync(...)                = false → 0 personas
```

Both load instances populate the *same* tool registry / persona resolver
in the host process. Whichever is registered last appears to be the
"winning" instance for tool listing — and for at least one ordering, the
stale one (with empty personas) wins. `ensemble_list` returns the empty
list, every spawn fails with "no personas resolved".

### Would `realpath`-walking fix it?

Yes. `node --version` on this host: `v24.7.0`. `fs.realpathSync(here)`
would canonicalize the symlink to
`/local/home/samfp/scratch/pi-conductor/dist/index.js`, after which the
`..` walk lands on the package root regardless of how the host loaded
the bundle. The fix is one line:

```ts
const here = realpathSync(fileURLToPath(import.meta.url));
```

…plus an import from `node:fs`. No async migration needed (the function
is already synchronous and called from sync code paths).

## 3. Recurrence assessment

**Recurrence is likely** — the documented install pattern is in two
durable artifacts (`README.md`, `HANDOVER.md`). Anyone who:

- copy-pastes the README install instructions (humans, agents, future
  self), or
- restores a backup of `~/.pi/agent/extensions/`, or
- onboards via a bootstrap script that codifies the README pattern,

…will recreate the broken symlink. The host's auto-discovery of
`~/.pi/agent/extensions/conductor/index.js` is intentional behavior
(`package-manager.js:388–438`) and won't change downstream.

**It is not deterministic on `/reload`** (we verified: the user's
manual `rm` + `/reload` produced 16 personas, not 0). It is conditioned
on the symlink existing AND `dist/index.js` being a real file, both of
which only became simultaneously true after the `3949fc4` build-to-dist
migration landed yesterday. The window of vulnerability is ongoing;
silent "0 personas" is the failure mode for any future install that
reproduces the pattern.

## 4. Recommended fix

**Pick: (d) — `realpath`-walk `import.meta.url` in `builtinPersonasDir()`.
Plus (e) — strip the `mkdir + ln -s` snippet from README/HANDOVER and
replace with the modern `packages[]` install.** Run as one slice.

### Why (d) over the alternatives

| Option | Verdict | Reason |
|---|---|---|
| (a) Patch the auto-installer | Rejected — there is no auto-installer in our control. The smoking gun is README copy-paste. |
| (b) Doctor warning only | Insufficient as primary — the warning surfaces only on `/conductor doctor`, not at session start. The user lost the chain budget before noticing. Should still ship as defense-in-depth (see §5). |
| (c) Hardcode an additional discovery path | Rejected — uglier than (d) and entrenches the legacy install pattern. |
| **(d) `realpath`-walk** | **Selected.** One-line code change, fixes the root cause, fully back-compat: works whether the bundle is loaded via `packages[]`, via legacy symlink, via `pi -e <path>`, or via any future load mode. |
| (e) Doc-only | Insufficient alone — existing legacy installs in the wild won't self-heal, but pairs perfectly with (d). |
| (f) Combine (d) + (e) + (b) | This is the recommendation. |

## 6. Open questions for the user

1. **Upstream issue for `pi-coding-agent`?** The host's auto-discovery of
   `~/.pi/agent/extensions/conductor/index.js` *combined with* the
   simultaneous load via `packages[]` produced the dual-registration
   that confused the persona resolver. Two reasonable upstream patches:
   (a) dedupe extensions by their `realpath` so two registrations of
   the same physical bundle collapse, or (b) at least warn when the
   same canonical path appears twice in the extension list. **Should
   we file this upstream against `@earendil-works/pi-coding-agent`?**
   Recommend yes — the bug class isn't conductor-specific and other
   packages with `packages[]` entries + legacy symlinks would hit the
   same trap.
2. **Drop `HANDOVER.md` entirely?** It's a v0.7-era onboarding doc with
   stale defaults (`PI_CONDUCTOR_MODE=0` opt-out flow). The whole file
   may be retire-able; the current README + PRD cover its content.
   Confirm whether the patched HANDOVER stays in the tree or moves to
   `docs/archive/`.
3. **Should the doctor warning be enabled in test environments?** The
   audit relies on `homedir()`. If we want it stub-friendly, the
   refactor adds a tiny `homeDir?: string` parameter to
   `DoctorReportOptions`. Worth doing for testability; flag if scope
   creep is unwelcome.
