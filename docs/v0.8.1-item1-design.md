## 1. Problem restatement

When the parent conductor spawns a persona with `inherit_context: filtered`, the
parent's session messages run through `filterParentContext`
(`src/context-filter.ts:73`). For each parent assistant message, the filter
walks `content` and drops `toolCall` blocks whose `name` matches an excluded
prefix (`ensemble_*`, `subagent`). When the same assistant message also contains
a `text` block — typical when the conductor narrated the spawn ("Spawning
critic-X to gate Y. 3/4 slots in use, holding the turn") immediately before the
tool call — the rewrite path at `src/context-filter.ts:127–130` preserves the
prose and drops only the `toolCall`. The persona's seeded transcript therefore
contains a parent assistant turn whose entire content is third-person
orchestration prose *about* the spawn, often naming the persona by id.

Two dogfood failures (2026-05-15) showed that personas read this prose as a
quoted brief or a meta-conversation rather than executing their actual brief
(`critic`, `oracle`). Both recovered cleanly via `ensemble_send` corrective.
Root cause: structurally, the filter preserves this kind of prose; behaviorally,
the existing `<filtered-history>` sentinel (`src/runs.ts:358–378`) does not
warn the persona about role-identity drift specifically — it only warns about
"dangling references" in general.

## 2. Hypothesis: why did the existing sentinel under-fire?

The existing sentinel:

- **Sits at the top** of the seeded transcript (`src/runs.ts:425`,
  `seedMessages = [filteredHistorySentinel(), ...seedMessages]`). Latest-turn
  bias in chat models means the *bottom* of the transcript dominates attention.
- **Does not name the role-identity failure mode.** It tells the persona to
  treat orchestration *references* with skepticism ("if you see assistant prose
  that references prior orchestration ... treat those statements with
  skepticism"). It does not say "the most recent user message in this
  transcript is YOUR brief; earlier user messages are framing."
- **Does not anchor the persona's task deterministically.** A persona reading a
  filtered transcript has no marker telling it *where* its brief lives.
- **The leaked prose names the persona by id.** "Spawning critic-X..." reads as
  a status report *about* the persona, not as a quote of someone else's plan;
  identity confusion is more pronounced when the prose names the agent.

**Pick:** all four contribute, but the dominant single cause is **the absence
of a deterministic anchor for "where is my actual brief?"** combined with
**latest-turn salience bias making the leaked prose more visible than the
top-of-transcript sentinel**. A persona has no reliable way today to tell
"this assistant message is leftover orchestration narration" apart from
"this assistant message is conversational context I should integrate."

**Design implication for (b)-strengthen:**

- The sentinel must give the persona an unambiguous *anchor*: "the last user
  message in this transcript is your brief; earlier prose is framing."
- The sentinel should explicitly call out the role-identity failure mode
  (third-person prose about you is not your brief).
- The sentinel must remain at the top (it sets the contract for everything
  below) — but its content must compensate for latest-turn bias by giving
  a *rule the persona can apply at the bottom of the transcript*, not a
  general advisory.
- (a') reduces the surface (a leaked third-person message about the spawn no
  longer survives), so the strengthened sentinel is the second line of defense
  for the leak surfaces (a') doesn't catch — e.g. orchestration narration in a
  **separate** assistant turn that didn't co-occur with an `ensemble_spawn`
  call. This is consistent with the "belt-and-suspenders" framing locked in
  the conductor decision.

## 3. Mitigation (a') — structural drop spec

### 3.1 Trigger

In `filterParentContext` at `src/context-filter.ts:118–139` (the
`case "assistant"` branch), the *pre-pass* at `:88–103` already collects the
set of excluded `toolCall` ids. Augment it (or compute alongside) the **set of
parent message indices** whose `content` array contained at least one
excluded `toolCall`.

In the per-message dispatch, for `role === "assistant"`:

- If the message index is in that set → **drop the entire message** (do not
  `out.push`, do not rewrite).
- Else → existing behavior (filter `thinking` blocks if `dropThinking`,
  pass `toolCall` blocks not on the exclude list, etc.).

### 3.2 Action

Drop the whole `AgentMessage`. The `toolResult` for any excluded `toolCallId`
is already dropped in the existing `case "toolResult"` (no orphan results, see
test at `tests/context-filter.test.ts:178–189`). No new
toolResult-cleanup logic is needed.

### 3.3 Sentinel interaction

The `dropped` calculation in `src/runs.ts:404–414` runs *after* filtering and
checks `filtered.length !== parentMessages.length` plus a per-index identity
walk. Under (a'), an assistant turn that previously survived as a rewritten
message now disappears entirely. Both detection paths still fire:

- The length-mismatch path fires because one entire entry was removed.
- Even if length happens to match (e.g. a legitimate assistant text turn
  shifted into the same slot), the identity walk catches it because
  `filtered[i] !== parentMessages[i]`.

So the existing sentinel-fire trigger is a strict superset of pre-(a') triggers —
the sentinel will fire on every case that fired before, plus more.

### 3.4 Edge cases (decisions, with rationale)

| Assistant message content | Pre-(a') | Under (a') | Rationale |
|---|---|---|---|
| Excluded `toolCall` only | Drop (empty rewrite) | **Drop** | No change. |
| `text` + excluded `toolCall` | Rewrite to `text` only | **Drop whole** | The fix. |
| `thinking` + excluded `toolCall` | Drop (both stripped) | **Drop whole** | No change. |
| `text` + non-excluded `toolCall` | Pass with both | **Pass with both** | Not orchestration. |
| `text` + excluded `toolCall` + non-excluded `toolCall` | Rewrite (text + non-excluded) | **Drop whole** (decision) | See below. |
| `thinking` + `text` + excluded `toolCall` | Rewrite to `text` | **Drop whole** | The fix. |

**Decision for the mixed `text` + excluded + non-excluded case:** drop whole.

- Rationale: any turn that mixed `ensemble_spawn` with another tool call
  (e.g. `read` + `ensemble_spawn` in the same turn) is by definition an
  orchestration turn; the `read` was probably a quick lookup the parent did
  while drafting the spawn. Treating "any excluded toolCall ⇒ orchestration
  turn ⇒ scrub" is the simpler invariant and matches the design intent
  ("structural, no regex, no false-positive risk").
- Counter-rationale we considered and rejected: "preserve the non-excluded
  toolCall + result so file knowledge survives." Risk: the non-excluded
  toolCall result still references back to a parent assistant turn that no
  longer exists, creating a dangling-result orphan that the seeded transcript
  emits as "tool result without prior call." This is the exact problem the
  pre-pass `excludedCallIds` set was designed to prevent for excluded tools;
  re-introducing it for the mixed case to save one `read` is bad value.

### 3.5 Stale `stopReason` latent bug

`src/context-filter.ts:127–130` rewrites the assistant message via
`{ ...(msg as any), content: filtered }`. The pre-(a') rewrite path leaves
`stopReason: "toolUse"` on a message whose `toolCall` block was just removed.
No consumer in the harness reads `stopReason` from filtered output today, but
it's a latent shape-correctness bug.

**Decision: address in the same slice.** Under (a') the rewrite branch is
*deleted entirely* (we either drop or pass-through; we never rewrite for the
excluded-toolCall case). The latent bug therefore disappears as a side effect
of the (a') change. **The pass-through case (`filtered.length === content.length`)
remains unchanged**, so we are not introducing a new shape rewrite. Net: zero
new `stopReason` writes, one path that was buggy is gone.

If a `thinking`-only drop (`dropThinking`) ever rewrites a message, it's still
done via `{ ...(msg as any), content: filtered }` and would still propagate
`stopReason: "toolUse"`. We accept this remaining surface as-is — `dropThinking`
runs on every spawn and any `stopReason` fix would also need test coverage we
don't budget for here. **Open for v0.8.2.**

## 4. Mitigation (b)-strengthen — sentinel text spec

### 4.1 Verbatim replacement (target ≤300 words; actual: 178)

Replace the body of `filteredHistorySentinel()` at `src/runs.ts:358–378`. The
function signature stays identical. The new `content` string:

```
<filtered-history>
You are reading a FILTERED slice of a parent conductor's conversation. Two
things to know before you act:

1. **Your brief is the LAST user-role message in this transcript.** Earlier
   user-role messages were the parent conductor talking to itself or to its
   user; they are framing, not your task. Treat them as background context.

2. **Some assistant prose may discuss YOU in the third person** — sentences
   like "spawning critic-X to gate Y" or "holding the turn while inspector
   runs". That prose is leftover orchestration narration from the parent. It
   is NOT a quote of your brief, NOT instructions to you, and NOT a
   conversation you are part of. Ignore it; do not meta-comment on it.

The following entry types were dropped before you saw this transcript:
  - Orchestration tool calls (ensemble_*, subagent) and their results
  - Sub-agent completion notifications (`<sub-agent-completed>` cards)
  - The conductor's internal reasoning (`thinking` blocks)
  - Bash commands marked with the `!!` excludeFromContext flag

If you see a dangling reference to prior orchestration ("the inspector said
X", "as oracle noted") and you do not see a matching tool result above, that
reference is from a dropped turn — treat the claim with skepticism.

Now: read your brief (the last user message), do the work, return your result.
</filtered-history>
```

Word count: 178. Character count: ~1.3 KB. Token cost: ~280 tokens. Sentinel
fires only on filtered spawns where `dropped === true` (per `src/runs.ts:425`),
so the cost is paid per-persona-spawn, not per-turn — well within budget.

### 4.2 Fallback shorter version (≤120 words; actual: 96)

Reserved if the verbatim fails the test-budget for sentinel size or if the
oracle gate flags verbosity:

```
<filtered-history>
You are reading a FILTERED slice of a parent conductor's conversation.

Your brief is the LAST user-role message in this transcript. Earlier user
messages are framing, not your task.

Some assistant prose may discuss YOU in the third person ("spawning
critic-X", "holding the turn"). That is leftover orchestration narration,
not your instructions. Ignore it.

Dropped: orchestration tool calls (ensemble_*, subagent), completion cards,
thinking blocks, and bash flagged !!. Dangling references to dropped turns
should be treated with skepticism.

Read the last user message, do the work, return your result.
</filtered-history>
```

### 4.3 Why this text addresses the hypothesis

| Hypothesis cause | Verbatim text addressing it |
|---|---|
| No deterministic anchor | "Your brief is the LAST user-role message" + "Read the last user message" closer |
| Latest-turn salience bias | The closer ("Now: read your brief... return your result") rephrases the anchor at the end of the sentinel, increasing the chance the persona retains it through the rest of the seeded transcript |
| No role-identity failure-mode warning | Rule 2 names third-person prose about the persona explicitly |
| Generic "dangling reference" advisory | Kept (rule 3) but downgraded to secondary |

### 4.4 What stays out of the sentinel

- We do **not** add an explicit example with concrete persona names ("e.g.
  'spawning oracle-7f3a'") — risk of priming role confusion or wasted tokens.
  The two example fragments inline ("spawning critic-X", "holding the turn")
  are short and behaviorally illustrative without naming a real persona.
- We do **not** instruct the persona on how to disambiguate `ensemble_send`
  follow-ups vs the original brief. `ensemble_send` injects a new user-role
  message at the bottom; the "last user message" rule already covers this case.
  (Verify in §6 tests.)

## 5. Q#16 per-persona audit

**Premise.** Inheriting filtered context costs tokens; the v0.8 dogfood showed
it can also cost reliability via Item 1's failure mode. The flip-side is that
some personas legitimately benefit from inherited context (file reads, prior
user prose, branch summaries). The audit below asks: for each persona, is the
parent's filtered context worth the cost?

**Heuristic.**
- `none` if the persona's job is well-defined by `default_reads` (`context.md`,
  `design.md`, `plan.md`) plus its task brief — i.e. it should ground only in
  artifacts, not in conversation history.
- `filtered` if the persona benefits from seeing the parent's prior file ops,
  user prose, and branch summaries — typically because it's investigating
  something contextual or producing a brief based on conversation history.
- `none` is also preferable when the persona is a **gate** that should boot
  fresh and re-derive judgment from artifacts (oracle's "decision-consistency"
  framing); inheriting parent prose risks anchoring it to the parent's
  hypothesis.

| # | Persona | Current | Recommended (v0.8.1) | Rationale (one sentence) | Risk if flipped to `none` |
|---|---|---|---|---|---|
| 1 | `oracle` | filtered | **none** | Gate persona; should re-derive from artifacts (`context.md`, `design.md`, `plan.md`) and the brief — inheriting parent prose risks anchoring to the parent's framing. | Loses parent's articulated worry; mitigation: cartographer/parent encodes worry in `context.md` per chain shape. |
| 2 | `redteam` | filtered | **none** | Adversarial gate; same reasoning as oracle — should boot fresh and try to break the artifact, not the parent's interpretation of the artifact. | Same as oracle; mitigation: pass diff/PR explicitly in brief. |
| 3 | `inspector` | filtered | **none** | Read-only recon over a fresh codebase area; the brief tells it what to map. Filtered context is mostly noise (the parent already saw enough to know recon is needed). | Loses signal about what the parent already knows; mitigation: parent encodes "I already read X, focus on Y" in brief. |
| 4 | `analyst` | filtered | **none** | Deep-dive over an area; produces structured suggestions. Same reasoning as inspector — brief + artifacts are sufficient. | Same as inspector. |
| 5 | `profiler` | filtered | **none** | Performance baseline measurement; takes a target file/scope from the brief. Conversation history is irrelevant. | Loses parent's prior bench commands; mitigation: include in brief. |
| 6 | `investigator` | filtered | **filtered** (KEEP) | Bug root-cause hunt; benefits from seeing what the parent already tried (file reads, bash commands, error reproductions). The "Phase 0 reframe → Phase 1 reproduce + trace" loop in the persona description explicitly leans on seeing what's been tried. | n/a (not flipping). |
| 7 | `scribe` | filtered | **none** | Documentation drafting; produces from artifacts and brief. Needs the artifact (e.g. design.md), not the conversation. | Loses paraphrasable parent prose; mitigation: parent quotes the prose into the brief if scribe needs it. |
| 8 | `clarifier` | filtered | **filtered** (KEEP) | Turns vague request into concrete acceptance criteria; **needs to see what the user actually said** in their own words, including back-and-forth. Cannot work from artifacts alone. | n/a (not flipping). |
| 9 | `cartographer` | filtered | **filtered** (KEEP) | Job is literally to package up parent context; needs the most context, not the least. | n/a (not flipping). |
| 10 | `critic` | filtered | **filtered** (KEEP) | Per-slice review; benefits from seeing the slice in the context of prior commits, file ops, and the parent's articulated acceptance criteria. Default-rejection persona benefits from skepticism material. **However, this is the persona that confused itself in the dogfood failure** (`critic-X` leak). The sentinel + (a') fix should address that — but if the gated oracle disagrees, this is the row to reconsider first. | Loses commit + file-op history; mitigation: encode in brief. |
| 11 | `finalizer` | filtered | **filtered** (KEEP) | Whole-task completeness gate; needs to see the *trajectory* (prior slices, user concerns raised mid-task, file ops) — that's the entire point of being stricter than critic. | n/a (not flipping). |
| 12 | `verifier` | filtered | **none** | Independent verification of a claimed change; **independence is the value**. Inheriting parent context risks the verifier just confirming what the parent already concluded. | Loses claimed-change description; mitigation: pass it explicitly in brief (`builder` returns it as evidence anyway). |
| 13 | `designer` | filtered | **filtered** (KEEP) | Designs from inherited context — the v0.8 design itself was produced this way. Filtered slice is the input; flipping would force the parent to re-encode all framing into the brief. | n/a (not flipping). |
| 14 | `planner` | filtered | **filtered** (KEEP) | Decomposes an *approved* design; benefits from seeing the design's discussion history (alternatives weighed, decisions locked). | n/a (not flipping). |
| 15 | `simplifier` | filtered | **filtered** (KEEP) | Identifies safe simplifications; needs to see what the parent considered out-of-scope vs in-scope to avoid touching wrong code. Write-capable; flip would cost more than it gains. | n/a (not flipping). |
| 16 | `builder` | filtered | **filtered** (KEEP) | Implements a slice; benefits from prior file knowledge and any dogfood-witnessed problems. Write-capable; flip would force re-reading. | n/a (not flipping). |

**Summary:** **9 personas keep `filtered`** (`investigator`, `clarifier`,
`cartographer`, `critic`, `finalizer`, `designer`, `planner`, `simplifier`,
`builder`) and **7 flip to `none`** (`oracle`, `redteam`, `inspector`,
`analyst`, `profiler`, `scribe`, `verifier`). 9 + 7 = 16. ✓

### 5.1 Risk callout for the flips

The 7 flips collectively reduce the leak surface for Item 1's failure mode by a
lot — the dogfood failures hit `critic` and `oracle`. `oracle` flipping to
`none` removes one entire failure path; `critic` stays `filtered` but is now
protected by (a') + the strengthened sentinel.

Highest-regret flip: `oracle` losing parent worry. Mitigation: in §11 of the
prompt addendum, `oracle` is opener and gate; openers always have a
cartographer-style brief, gates always reference `design.md` / `plan.md`. The
cost of the flip is one extra sentence in oracle briefs ("the parent worry is:
X"). Acceptable.

Second-highest regret: `inspector` losing prior file knowledge. Same
mitigation — briefs already need to specify scope.

### 5.2 What does NOT flip — and why explicitly

- `cartographer` / `clarifier`: their job is the prose itself.
- `designer` / `planner` / `builder` / `simplifier` / `critic` / `finalizer` /
  `investigator`: producers and producers'-gates that need trajectory.

## 7. Risks & open questions for the design's oracle gate

### 7.1 Risks

- **R1 (a') drops legitimate prose.** A turn where the parent legitimately
  reasoned in prose alongside an `ensemble_spawn` call loses that reasoning
  from the persona's view. Surface in dogfood; reverse if seen. Mitigation:
  the strengthened sentinel anchors the persona to its brief regardless,
  and the conductor's §11 chain shapes already discourage prose-heavy spawn
  turns ("don't keep heavy orchestration narration near a `ensemble_spawn`
  call when the spawn uses `inherit_context: filtered`").

- **R2 sentinel size.** Adding ~280 tokens per filtered spawn. At ~50 spawns
  per medium task (as observed in v0.8 dogfood), that's ~14k tokens of
  sentinel cost per task. Acceptable; well below the cost of a single
  `ensemble_send` corrective ($1+ per recovery).

- **R3 Q#16 flips create regret tail.** A persona flipped to `none` losing
  parent-prose context that *only manifests as a problem in some future task
  shape*. Mitigation: the audit is reversible per-persona via frontmatter; the
  test in §6.1 D pins the audit so any flip-back is intentional and reviewed.

- **R4 sentinel + (a') interaction is not exhaustively tested.** We test that
  (a') drops whole messages; we test that the sentinel content is right.
  We do **not** test that a persona spawned with (a')+sentinel actually
  behaves differently in the dogfood failure case — that's a live-run signal,
  not a unit test. Manual smoke per §8.3 covers this; live integration test
  (`spawn.integration.test.ts`) gated on `CONDUCTOR_LIVE_TESTS=1` would be
  the harder signal but is out of scope.

- **R5 latent stale-`stopReason` case left.** The `dropThinking` rewrite
  path still propagates `stopReason: "toolUse"` on a message whose toolCall
  block was dropped. Documented in §3.5 as v0.8.2.

- **R6 The test in §6.1 D depends on a persona-frontmatter loader.** If
  `src/personas.ts` doesn't expose a function that returns
  `(name, frontmatter)`, the test will need a tiny local YAML reader.
  Builder should choose; both options are sub-10 LoC.

### 7.2 Open questions for oracle gate

- **OQ-1 — Should `clarifier` flip to `none`?** Argued KEEP filtered above
  (needs to see what the user said in their own words). Counter: when
  spawned by the parent with the user's prose forwarded into the brief,
  filtered context is redundant. Decide: KEEP for now, revisit if dogfood
  shows clarifier is robust without it.

- **OQ-2 — Do we add a leak-detection logging hook now?** Item 1's design
  decision deferred (a) regex stripping. To make the v0.8.2 design possible,
  we need to instrument leak surfaces in (b)-strengthened production. Proposal:
  emit a debug-level structured log when (a') drops a message
  (`{ run_id, persona, dropped_count, dropped_names: [...] }` — `dropped_names`
  is the list of toolCall names that triggered the drop). Stays internal to
  `src/runs.ts`; no user-visible surface; gated on `process.env.DEBUG`.
  Does **not** add new visible cost. **Recommendation: include in the slice.**
  Incremental code: ~10 LoC.

- **OQ-3 — Symmetry with non-`ensemble_*` excluded prefixes.** §6.1 A test "E"
  pins that `subagent`-prefix tool calls also trigger (a'). Should the
  symmetry be enforced for any future-added prefix? **Recommendation: yes**
  — the (a') logic doesn't special-case `ensemble_`; it dispatches on the
  configured `excludeToolPrefixes` set. The test pinning this is the contract.

- **OQ-4 — Rollback story.** If (a') over-fires in dogfood, what's the
  rollback? Single-commit: revert the slice. The 7 persona flips revert with
  it. Alternative: feature-flag (a') with an opt-out env (`PI_CONDUCTOR_PRESERVE_PROSE=1`).
  **Recommendation: no flag.** The codebase has no comparable env-flag
  precedent and the fix is small enough to revert cleanly. If oracle disagrees,
  add a flag; cost is ~5 LoC.

- **OQ-5 — Should the strengthened sentinel be reused for `inherit_context: full`?**
  Currently `full` doesn't seed a sentinel because nothing was dropped. With
  (a') landing, `full` is the only path that preserves the orchestration
  prose verbatim. **Recommendation: leave `full` alone.** Personas using
  `full` are explicitly opting into raw inheritance; no v0.8 persona uses
  `full`; this is yagni.

### 8.3 Manual smoke gate (post-Step-7, pre-finalizer)

The unit + integration tests confirm structural correctness; they do **not**
confirm that a persona reading the strengthened sentinel actually anchors to
the last user message under filtered context. That's a live-model signal.
Before handing off to `finalizer`, the builder (or a verifier) must run the
following two scenarios end-to-end and record evidence in the slice's
completion notes.

**Scenario A — `critic` re-creation of the witnessed leak.**

1. With conductor mode ON and a parent session that has at least one
   prior `ensemble_spawn` turn co-occurring with prose, spawn a `critic`
   gate on a small mock diff via `ensemble_spawn`.
2. Pre-condition the parent's last assistant turn before the critic spawn
   to include narration like
   `"Spawning critic-<id> to gate the diff. N/4 slots in use, holding the turn."`
3. Open the critic's focused stream (Ctrl+G) once it starts.
4. **Expected:** critic produces an actual review of the diff. No
   meta-commentary on the brief, no "I don't see a diff to review," no
   self-referential prose about being spawned.
5. **Failure signal:** any of (3-line orient-yourself stalls, requests for
   the brief, comments like "I see I was spawned as a critic").

**Scenario B — `oracle` re-creation of the witnessed leak.**

1. Same setup with an `oracle` gate spawned to review a small mock plan.
2. Pre-condition the parent's last assistant turn to include narration like
   `"Holding the turn while oracle-<id> reviews. 2/4 slots in use."`
3. **Expected:** oracle produces a structured review of the plan against
   inherited decisions.
4. **Failure signal:** oracle quotes the orchestration narration back, asks
   what plan it should review, or describes its role in the third person.

**Recording.** The slice commit message (or a follow-up comment on the
finalizer's gate) should include one line per scenario:
`smoke A (critic): pass | evidence: <transcript path or quote>` and same for B.
**`finalizer` rejects the slice if §8.3 is not recorded.**

If either scenario fails despite green unit tests, the (b)-strengthen text in
§4.1 is the first thing to revisit (try the §4.2 fallback, or rephrase the
anchor rule). Do **not** silently revert (a'); the structural drop is
independently valuable.
