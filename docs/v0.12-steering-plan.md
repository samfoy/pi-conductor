## 1. Decomposition philosophy

Each slice is independently green (`npm test` + `npx tsc --noEmit`), single Conventional Commit, vertical and verifiable, **test-first** per AGENTS.md / `CONTRIBUTING.md`. Per-slice critic gates are scoped — only slices touching pure resolvers (1, 4) or load-bearing prose (7) get them; mechanical wiring slices (2, 3, 5, 6) stay un-gated. Vertical means each slice produces an observable check beyond pure-refactor (test that fires, argv that flips, reconcile branch that reclassifies). Slices that would be horizontal (types-only, header-only) are merged into the slice that consumes them.

### Slice 5 — `fix(watchdog,reconcile): RPC bump asymmetry + orphan reclassify + forceTerminate cleanup`

**Dependencies.** Slice 4 (`Run.pendingAcks`, `RpcStdinQueue` is live, `streamingMode` is stamped at spawn).

**Files.**

## 3. Slice ordering rationale

Strict serial. Out-of-order builds must be impossible:

- **Slice 2 depends on slice 1's** `Run.steerable?` and `Run.streamingMode?` types because `RpcStdinQueue` is keyed off the run lifecycle (slice 5's `forceTerminate.destroy("force-terminate")` will read `run.streamingMode`).
- **Slice 3 depends on slice 2's** `RpcStdinQueue.enqueue()` because the auto-cancel handler synchronously enqueues the cancellation envelope. Without the queue, slice 3 would have to fall back to raw `proc.stdin.write` and break the single-writer invariant (Risk #1).
- **Slice 4 depends on slice 3's** `routeRpcResponse` stub being upgradable to a real correlation router. Slice 4 cannot wire the ack-envelope timeout without a queue that forwards `response` lines to a per-run handler — and that's exactly what slice 3 ships as a stub.
- **Slice 5 depends on slice 4's** `Run.pendingAcks` Map because `forceTerminate.destroy()` rejects pending ack promises. Building slice 5 before slice 4 would require either a dummy pendingAcks Map (slice creep) or skipping the ack-rejection path entirely (regression risk).
- **Slice 5 split-on-demand note.** The three sub-fixes inside slice 5 (bump asymmetry, reconcile-orphan classify+executor, forceTerminate cleanup) are NOT test-coupled — each lands in its own test file with its own production-code surface. Builder MAY split slice 5 into 5a/5b/5c if (a) the cumulative diff exceeds the size/blast-radius the M tier comfortably absorbs, or (b) merge-conflict surface against `master` forces it during review. Each split must be independently green and ship as its own Conventional Commit. Critic gate scope (the three-pronged review enumerated in slice 5's body) must be applied to whichever split owns each concern. Default plan: ship as one slice.
- **Slice 6 depends on slice 5's** clean `forceTerminate` for the live integration test ("forceTerminate mid-turn dies cleanly"). Without slice 5's `RpcStdinQueue.destroy` preamble, the live integration test would observe a dangling stdin-promise rejection that doesn't propagate cleanly.
- **Slice 7 depends on slice 6's** slash command + live test having landed so the §10 prompt addendum can describe a working surface. Drafting §10 before slice 6 risks describing functionality that doesn't ship.

**Config-tree placement note (§3 reference).** Oracle review 2 §6 flagged that `cfg.watchdog.defaultKillOnStall` lives under `cfg.watchdog.*` whereas `defaultSteerable` is at top-level `ConductorConfig`. **Planner discretion: keep at top-level for v0.12.** v0.10 placed `defaultKillOnStall` under `cfg.watchdog.*` because there are 4 watchdog knobs that share the namespace. Steering currently has 1 knob (`defaultSteerable`); pulling it under `cfg.steering.*` for symmetry would create a single-occupant config sub-tree. If v0.13+ adds further steering knobs (e.g. `defaultStreamingBehavior`, `ackTimeoutSeconds`), pull them all under `cfg.steering.*` together. Trade-off is one-line; no migration cost since all v0.12 fields are optional.
