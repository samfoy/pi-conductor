# v0.12 — Steering running sub-agents — inspector map

Survey only. No design, no fixes. The next persona (planner / designer)
takes the deep-dive. Citations are repo-relative unless prefixed with
`[pi-dist]` (= `/home/samfp/.local/share/mise/installs/node/24.7.0/lib/node_modules/@earendil-works/pi-coding-agent/dist/`).

The architectural pivot the milestone implies: today the conductor
spawns each sub-agent under `pi --mode json -p` (print-mode JSON) which
is one-shot and inherently unsteerable. Pi exposes a separate
`--mode rpc` headless mode whose stdin protocol includes a `steer`
command. Adding steering means migrating (or branching) the spawn
pipeline onto that RPC protocol while keeping our existing event-state
plumbing.

## 1. Current sub-agent spawn pipeline (print-mode)

### Tool entrypoint → spawn

- `src/tools.ts:441`–`:625` — `registerSendTool` (the `ensemble_send` LLM tool); `sendToRun` call at `tools.ts:520`.
- `src/tools.ts` ~`registerSpawnTool` (LLM `ensemble_spawn`) calls `spawnRun`. Foreground/background fan-out and the inline-stream throttle live in this file.
- `src/runs.ts:745` — `spawnRun(opts)`. Allocates the run dir, flips `status="running"`, calls `planSpawnPiArgs`, then `runPiSubprocess`.
- `src/runs.ts:503` — `planSpawnPiArgs(opts)`: branches between fresh spawn (`--session-dir`) and seeded-resume (`--session <path>`) depending on `inherit_context`.

### Argv composition — `buildPiArgs` (`src/runs.ts:333`–`:362`)

- Always emits `["--mode", "json", "-p", ...]` (`runs.ts:334`).
- Branches on `kind`: `"fresh"` → `--session-dir <dir>` (`:336`); `"resume"` → `--session <path>` (`:338`).
- Then `--model`, `--thinking`, `--append-system-prompt <persona body>` (`:345`/`:351`).
- Zero-or-more `--skill <abs-path>` flags (`:357`–`:359`), user-then-project order, computed by `collectInheritedSkillPaths` (`runs.ts:90`); only emitted when `persona.inheritSkills`.
- Trailing positional is the prompt text. Fresh: `buildSubAgentPrompt(persona, task)` (`runs.ts:273`); resume: raw send message (`buildResumePiArgs`, `runs.ts:378`).

### Subprocess plumbing — `runPiSubprocess` (`src/runs.ts:925`–`~1100`)

Single helper used by both `spawnRun` and `sendToRun`.

- `child_process.spawn(invocation.command, invocation.args, {stdio: ["ignore","pipe","pipe"], shell:false, env: buildSubagentEnv()})` at `runs.ts:935`. **stdin is `ignore`d** — no upstream channel into the running subprocess today. Single biggest delta for v0.12.
- `attachSpawnedProc(run, proc)` at `:949` records `proc` and `pid` and flushes record.json so post-startup reconcile can liveness-probe.
- Hard timeout via `setTimeout` → `forceTerminate(run, "timeout", …)` at `:953`–`:957`.
- `proc.stdout.on("data", ...)` (`:1085`) buffers, splits on `\n`, hands each line to `processLine` (`:1068`–`:1083`).
- `processLine` appends raw line to `transcript.jsonl`, `JSON.parse`s (non-JSON silently dropped), hands the parsed event to `applyEvent` (`event-handler.ts:38`) which returns `{kind:"none"|"updated"|"finalize"}`. On `finalize` calls the local `finalize` closure (`:973`).
- `proc.on("close", ...)` (`:1097`) routes natural exit to `finalize("completed", 0)` or `finalize("failed", code)` via the W7 idempotency guard `applyCloseHandlerTerminal` (`:1296`).

### JSON event handler — `src/event-handler.ts:38`–`:99` `applyEvent`

Listens for: `agent_end` → finalize completed; `turn_end` (no toolcall & not errored) → finalize completed; `message_end` → accumulate usage / messages / lastToolCall, return `updated`; `tool_result_end` → push to messages. All other types silently ignored. **Important for v0.12:** RPC mode emits `response` and `extension_ui_request` lines the print-mode handler has never seen — multiplexer or explicit drop required. See §2. `formatToolCallShort` (`:113`) — inline widget summary.

### `sendToRun` / `validateSendable` (resume-via-`pi --session`)

- `src/runs.ts:1158`–`:1193` — `validateSendable(run)` pure pre-check. **Hard reject `running`** (`:1162`–`:1167`): *"wait for it to finish before sending."* This is the gate the milestone has to revise. Also rejects `paused` (`:1168`–`:1172`) and `queued` (`:1174`–`:1178`); rejects on missing `run.sessionPath` or missing session file.
- `src/runs.ts:1210`–`:1267` — `sendToRun(run, message, opts)`. Gates via `validateSendable`, resets terminal fields, flips `run.status = "running"` (`:1232`–`:1245`). **Resume = a brand-new subprocess** with `pi --mode json -p --session <path>`. Previous subprocess is long-dead. Builds argv via `buildResumePiArgs` (`:1252`) and reuses `runPiSubprocess` (`:1254`–`:1266`).

### `forceTerminate` (`src/runs.ts:1403`–`:1500`)

Idempotent (`isTerminal(run.status)` bails). Foreign-pid guard refuses runs whose `parentPid !== process.pid` (`:1421`). `run.proc.kill("SIGTERM")` at `:1438`; 2s SIGKILL fallback (`:1444`). v0.11 also SIGTERMs the on-complete-hook process group if a hook is in flight (`:1462`–`:1486`). Sets `run.status` to `timeout` / `killed`; v0.10 watchdog reason `"stalled"` lands as `killed` with distinguishing `errorMessage`.

### `pauseRun` / `resumeRun` — SIGSTOP / SIGCONT (`src/runs.ts:1513`–`:1551`)

`pauseRun` requires `status === "running"` and a live `run.proc.pid`; `signaler(pid, "SIGSTOP")` flips to `paused`. `resumeRun` is the mirror. Both use a swappable `Signaler` for tests. **Implication for steering:** SIGSTOP suspends the entire pi subprocess including its stdin reader. A queued steer written to stdin while paused sits in the kernel pipe buffer until SIGCONT, then is read in arrival order. See §3 / §5 q7.

### `seedSessionFile` — pre-spawn bootstrap (`src/session-seed.ts:55`–`:88`)

Writes a pi-format JSONL session file ahead of `child_process.spawn`. Header `{type:"session", version:3, id, timestamp, cwd}`; subsequent lines `{type:"message", id, parentId, timestamp, message}` with linear `parentId` chain (first = `null`). Used by `planSpawnPiArgs` (`runs.ts:503`) when the persona has `inherit_context: filtered` or `full` AND a parent-message snapshot exists. Boot mode flips to `kind: "resume"` so pi's own session-resume code reads the seeded file.

**Compatibility for v0.12:** RPC mode also accepts `--session` (parsed before mode dispatch — `[pi-dist] main.js:386`–`:497`). Seeded-resume should keep working under RPC, but no test in this repo proves that yet.

### Conductor-host env

`src/runs.ts:222`–`:268` — `buildSubagentEnv()` strips `ENV_SESSION_DIR` etc. so a re-spawned `pi --session …` child doesn't inherit the parent's session-dir.

## 2. RPC mode surface (pi-coding-agent)

Read from `[pi-dist] modes/rpc/*.d.ts` and `rpc-mode.js`.

### `RpcCommand` union — `[pi-dist] modes/rpc/rpc-types.d.ts:13`–`:113`

Each command is a JSON line on stdin with optional `id` for
correlation.

- `{type:"prompt", message, images?, streamingBehavior?:
  "steer"|"followUp"}` (`rpc-types.d.ts:14`–`:21`). The CLIENT
  decides up front whether a prompt sent during streaming lands in
  the steer or follow-up queue. If the agent is idle, the prompt
  starts a new turn directly. Server: `rpc-mode.js:288`–`:309`.
- `{type:"steer", message, images?}` (`rpc-types.d.ts:22`) — pure
  steering queue add. Server: `await session.steer(...)`
  (`rpc-mode.js:311`–`:314`). Behaviour: delivered after the
  current assistant turn finishes its tool calls, before the next
  LLM call (`[pi-dist] core/agent-session.js:869`–`:887`).
- `{type:"follow_up", message, images?}` (`rpc-types.d.ts:28`).
  Queued; fires only when the agent settles (no more tool calls
  AND no pending steers). `[pi-dist] core/agent-session.js:888`–
  `:902`.
- `{type:"abort"}` (`rpc-types.d.ts:34`) — cancels current
  operation. Server: `await session.abort()`
  (`rpc-mode.js:319`–`:322`). Note: the conductor's
  `forceTerminate` SIGTERMs the whole subprocess (`runs.ts:1438`),
  so RPC `abort` is finer-grained but does NOT replace SIGTERM
  (the subprocess survives). See §5 q6.
- `{type:"get_state"}` → returns `RpcSessionState`
  (`rpc-types.d.ts:120`) including `isStreaming`,
  `pendingMessageCount`, `messageCount`, `sessionFile`.
  **`isStreaming` is the explicit "is there a turn running right
  now?" signal** the conductor's state machine doesn't currently
  track.
- `{type:"set_steering_mode", mode:"all"|"one-at-a-time"}`
  (`rpc-types.d.ts:60`) and `set_follow_up_mode` (`:64`) — the
  agent's per-turn delivery policy; affects how multiple queued
  steers flush.
- Adjacent / probably out-of-scope: `new_session`, `compact`,
  `set_auto_compaction`, `set_auto_retry`, `abort_retry`, `bash`,
  `abort_bash`, `switch_session`, `fork`, `clone`,
  `get_fork_messages`, `get_messages`, `set_session_name`,
  `set_thinking_level`, `cycle_thinking_level`, `set_model`,
  `cycle_model`, `get_available_models`, `export_html`,
  `get_last_assistant_text`, `get_commands`. The full table is in
  `rpc-types.d.ts`.

### `RpcResponse` — `[pi-dist] modes/rpc/rpc-types.d.ts:135`–`:325`

- Every command returns a `{type:"response", command,
  success:true, data?}` line. `id` is echoed back for correlation.
- `steer` response: `{command:"steer", success:true}` — empty
  payload; the actual delivery confirmation lands on the in-band
  event stream.
- `prompt` response is special — it returns success only after
  *preflight* succeeds (`rpc-mode.js:288`–`:309`); subsequent
  stream events flow as agent events.
- Failure shape: `{type:"response", command:string,
  success:false, error:string}` (`rpc-types.d.ts:323`).

### `RpcExtensionUIRequest` — `[pi-dist] modes/rpc/rpc-types.d.ts:329`–`:374`

- The RPC server emits `extension_ui_request` lines for any
  extension UI affordance: `select`, `confirm`, `input`, `editor`,
  `notify`, `setStatus`, `setWidget`, `setTitle`,
  `set_editor_text`. Host responds via `extension_ui_response`
  (`:376`–`:383`).
- **Conductor implication:** under print-mode today, sub-agent
  extensions that block on `ctx.ui.input(...)` simply error out
  (no TTY). Under RPC mode those requests flow back over stdout
  and the host MUST answer them or the sub-agent stalls forever.
  The conductor either auto-cancels them (forwarding
  `cancelled: true`) or proxies them to the user. Either way it's
  new wiring.

### `rpc-client.d.ts` — programmatic API

- The persona task explicitly says **don't use `RpcClient`**.
  Confirmed: it's a helper that owns its own subprocess
  (`process: ChildProcess`, `start()`, `stop()`,
  `onEvent(listener)`, `waitForIdle()` etc.). We already manage
  `proc` on `Run` (`runs.ts:953`) and have our own event plumbing
  (`event-handler.ts`). Adopting `RpcClient` would fork ownership
  of the subprocess and break `forceTerminate`'s SIGTERM path,
  the watchdog's `lastEventAt` updates (`event-handler.ts:65`),
  and the foreground-stream throttle (`tools.ts`
  ~`createUpdateThrottle`).
- The `RpcClient` source is still useful as a **reference
  implementation** of line framing, request/response correlation
  by `id`, and `extension_ui_request` round-tripping — the
  conductor will need to re-derive most of that.

### `jsonl.d.ts` — line framing

- `serializeJsonLine(value)` and `attachJsonlLineReader(stream,
  onLine)` (`[pi-dist] modes/rpc/jsonl.d.ts`).
- **Critical:** "Framing is LF-only. Payload strings may contain
  other Unicode separators such as U+2028 and U+2029. Clients
  must split records on `\n` only." Today's print-mode reader at
  `runs.ts:1085`–`:1090` already splits on `\n` only — existing
  framing is compatible.
- **The serialise direction is new:** print mode never wrote to
  the child's stdin. RPC mode requires
  `proc.stdin.write(JSON.stringify(cmd) + "\n")` per command.
  We'd want our own `serializeJsonLine` (importing pi's
  non-public dist file is a runtime-dep we should avoid).

### CLI flag coexistence

- `[pi-dist] main.js:76` resolves the mode: `--mode rpc` →
  `appMode="rpc"`, `--mode json` → `"json"`. Both go through the
  same `createSessionManager` (`main.js:386`), `createRuntime`
  (`main.js:467`), `--append-system-prompt` (read via
  `parsed.appendSystemPrompt` in `buildSessionOptions`), and
  `--skill` / `resolveCliPaths` (`main.js:407`). Then
  `appMode === "rpc"` hands off to `runRpcMode(runtime)`
  (`main.js:529`).
- **All four flags we use today (`--session-dir`, `--session`,
  `--append-system-prompt`, `--skill`) are mode-agnostic** —
  consumed before mode dispatch.
- Incompatibility: `[pi-dist] main.js:367` — RPC mode rejects
  `@file` arguments. We don't use those.
- Semantic difference: in RPC mode the trailing positional prompt
  is NOT auto-prompted; the host must send a `{type:"prompt",
  message:...}` command. The current "trailing positional = first
  user message" pattern in `buildPiArgs` (`runs.ts:359`) must
  change — either inject the prompt via initial RPC `prompt`
  command, or stays in argv but is ignored.

## 3. State-machine implications

`Run.status` taxonomy (`src/types.ts:227`–`:235`):
`queued | running | paused | completed | failed | killed | timeout
| hook_failed`.

### "Steerable" subset of `running`

Today there is no internal substate inside `running`. Print-mode
runs are either streaming events or about to terminate; the
conductor cannot tell the difference, but it doesn't matter (no
upstream channel). Under RPC, "running" splits into:

1. **`isStreaming: true`** — agent mid-turn. A `prompt` command
   requires `streamingBehavior` (`[pi-dist]
   core/agent-session.js:727`: *"Agent is already processing.
   Specify streamingBehavior ('steer' or 'followUp') to queue
   the message."*). `steer` / `follow_up` always queue.
2. **`isStreaming: false`** — turn finished, awaiting next
   prompt. A `prompt` would start a fresh turn directly.

Derivable from existing events (`agent_end` flips streaming false;
new `message_end` implies streaming true) but we don't track it.
`Run.lastEventAt` (`event-handler.ts:65`) is the closest existing
signal.

### In-flight steer with previous turn still running

The pi server queues. From `[pi-dist]
core/agent-session.js:907`–`:920`: `_queueSteer` pushes onto
`_steeringMessages` and awaits the agent's deliver-on-next-turn
boundary. There's no "steer arrived but not yet delivered" event
to surface. The conductor would need to either track
`pendingSteers` itself OR poll `get_state.pendingMessageCount`.

### Pause / SIGSTOP × stdin queue

`pauseRun` (`runs.ts:1513`) sends SIGSTOP — the subprocess's
stdin reader is suspended. A steer command written while paused
sits in the kernel pipe buffer (default 64KB on Linux). Many
stacked steers could fill the buffer and `proc.stdin.write` will
back-pressure / return false. On SIGCONT all buffered stdin is
read in order; no hook to re-validate "do you still want these?"

### Watchdog interaction

- `src/watchdog.ts:240`–`:280` `effectiveConfig` and `:303`–`:480`
  `Watchdog`. Stall is computed off `Run.lastEventAt`, which is
  bumped only by `message_end` / `tool_result_end`
  (`event-handler.ts:65`/`:96`).
- A queued steer that hasn't been delivered yet does NOT bump
  `lastEventAt`. So a sub-agent stuck waiting on a tool call
  could receive a steer and still trip the soft-stall threshold
  before the agent acknowledges it.
- v0.10 §"Q5" (`PRD.md:616`) deferred a `watchdog_silence` knob;
  v0.12 may want a "steer in flight" signal that defers stall
  classification by some grace window.
- The PRD-locked `kill_on_stall` semantics (`PRD.md:612`) include
  a pre-kill recheck (slice 2, `b7a36a9`). Adding a
  "pending-steer resets stall freshness" rule needs to compose
  with that recheck.

## 5. Coexistence questions for the designer

1. **Mode opt-in granularity.** All sub-agents become RPC, or
   per-persona frontmatter (`steerable: true`), or per-spawn LLM
   arg? Cost: every RPC sub-agent holds an open subprocess longer
   (idle-after-turn instead of exit-after-turn) and the conductor
   must answer `extension_ui_request` lines. PRD.md:262 already
   notes "ensemble_send bypasses the concurrency cap" — RPC mode
   amplifies that because the subprocess is genuinely live.
2. **Tool surface — overload `ensemble_send` or split.** (a)
   Overload with optional `streaming_behavior:
   "auto"|"steer"|"follow_up"|"resume"`; (b) two new tools
   `ensemble_steer` + `ensemble_follow_up` alongside
   `ensemble_send`; (c) auto-detect: running → steer, terminal →
   resume. Affects conductor-prompt (`conductor-prompt.ts:103`,
   PRD.md:297-block) and observability.
3. **Steer-into-finishing race.** A steer at T+0 may arrive at
   the agent at T+10, but the agent may have reached `agent_end`
   at T+5 and exited. The conductor's `proc.stdin.write` either
   buffers into a closed pipe (EPIPE on next tick) or fails. We
   need a "steer dropped because sub-agent finished" path.
4. **Acknowledgement envelope.** Surface RPC `response` lines
   (e.g. `{command:"steer", success:true}`) to the LLM as a new
   envelope, or stay silent and let the next `message_end` serve
   as ack? Silence keeps the conductor prompt simpler;
   ack-envelopes give the LLM something concrete to wait on
   inside a tool call.
5. **Print-mode session files vs RPC-mode session files.** Both
   modes use the same `SessionManager` (`[pi-dist] main.js:386`)
   and write the same JSONL format. `seedSessionFile` should
   keep working. But RPC mode supports `switch_session` / `fork`
   / `clone` / mid-process compaction; print-mode doesn't. Do we
   expose those, or hard-bind one subprocess = one session for
   v0.12?
6. **`forceTerminate` vs RPC `abort`.** Today `forceTerminate`
   SIGTERMs (`runs.ts:1438`); RPC `abort` cancels current turn
   but leaves the subprocess alive. Add a soft-cancel verb
   (`ensemble_abort`)? Map `ensemble_kill` to RPC `abort` first
   then SIGTERM? Adding the mid-step changes the LLM's mental
   model of "kill = gone."
7. **Pause/resume × steer queue.** Forbid steers while paused
   (clean), buffer them in conductor-side state and flush on
   `resumeRun` (UX nice, more code), or just write them to stdin
   and trust the kernel pipe (simplest, but unbounded).
8. **Slash command surface.** `/conductor send` does not exist.
   Adding one alongside the overlay 's' key gives the user a
   non-overlay path to steer. Worth it, or rely on the overlay?
9. **Watchdog and pending steers.** When `kill_on_stall: true`
   and a steer is in flight, extend thresholds, or stay strict
   (the steer might be the *cause* of the stall — a malformed
   message — and we want to kill anyway)? PRD §"Q3" / §"Q5"
   (`PRD.md:614`/`:616`) deferred analogous opt-out knobs.
10. **Backward compat with archived runs.** v0.9 cold-archive
    (`PRD.md:553`, decision D3) keeps `record.json` / `final.md`
    / `session/<...>.jsonl` for `ensemble_send` resumption via
    `pi --session`. Post-archive runs have `proc === undefined`
    and `pid === undefined`, so a "send to a terminal run" still
    spawns a fresh subprocess — nothing to steer. The two paths
    legitimately diverge here; worth pinning in the design.

## 6. Risk callouts

1. **Stdin race / partial-event tearing.** `proc.stdin.write` is
   not awaited synchronously with stdout reads. A steer command
   line written while a `message_end` line is being consumed on
   stdout could interleave at the wire level if we don't go
   through a single writer queue. Pi's own `RpcClient.send` is
   the reference; we'd need our own. Risk: half-written JSON
   line on stdin → pi's `attachJsonlLineReader` drops the
   record, the steer is lost, no error surfaced.
2. **`extension_ui_request` blocking the sub-agent.** RPC mode
   emits these synchronously from inside extension callbacks
   (`[pi-dist] modes/rpc/rpc-mode.js:75`–`:200`). If the
   conductor doesn't respond, the sub-agent's extension code
   blocks (`createDialogPromise`, no default timeout when
   `opts?.timeout` is undefined). A persona using a
   `ctx.ui.confirm`-calling extension could wedge forever. The
   conductor's pre-RPC era never had this failure mode.
3. **Watchdog confounding.** `lastEventAt` is bumped only by
   `message_end` / `tool_result_end`. RPC `response` lines do
   NOT bump it. A steer the agent acknowledges but sits on for
   130s before producing a `message_end` will trip soft stall.
   With `kill_on_stall: true`, the conductor's *own steer*
   could trigger a hard kill. PRD §"Q3" reversal
   (`PRD.md:614`) shows the team has a low tolerance for
   spurious stall events.
4. **Forced-terminate path tangles.** `forceTerminate` SIGTERMs
   on `run.proc` (`runs.ts:1438`). For an RPC sub-agent we'd
   probably want a graceful `abort` first, drained event reads,
   then SIGTERM. The current `applyCloseHandlerTerminal`
   race-fix (`runs.ts:1296`) and W7 idempotency guard
   (`runs.ts:1386`/`:1454`) are tightly woven; adding a new
   "graceful close requested but not yet observed" state risks
   regressing one of those.
5. **Session-file divergence between modes.** Both modes use
   the same on-disk schema (verified — same `SessionManager`),
   but RPC `set_session_name`, `fork`, `clone`, and
   compaction-on-RPC commands can mutate the session file in
   ways `findSessionFile` / `discoverSessionPathIfMissing`
   (`runs.ts:129`/`:170`) were never tested against. Latent
   risk of a stale `run.sessionPath` after a `switch_session` or
   `fork`. Out of scope if v0.12 doesn't expose those verbs —
   but worth flagging.

## Open questions

- **What's pi-coding-agent's RPC stability story?** The dist
  files are versioned with the rest of pi-coding-agent — no
  separate semver. PRD.md doesn't say which pi version we pin
  against; if pi 0.x bumps the RPC schema (added/removed
  commands, renamed events) the conductor breaks silently. The
  designer should decide whether to vendor a copy of
  `rpc-types.d.ts` and assert at runtime.
- **Does `RpcSessionState.isStreaming` flip synchronously with
  `agent_end`?** Empirically untested in this repo. The
  state-machine design hinges on it.
- **Are there events RPC mode emits that print mode doesn't?**
  Scan of `[pi-dist] modes/rpc/rpc-mode.js` shows the same
  `AgentSessionEvent` stream is forwarded via `output(...)`
  plus `extension_ui_request` and `response` lines — but a
  unit-level diff between the two event streams is not
  documented. The designer should reproduce one persona under
  both modes and diff transcripts.
