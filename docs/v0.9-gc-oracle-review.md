## 4. PRD consistency

### Closes PRD #12?

✓ Cleanly. `PRD.md:533` asks "keep finished sub-agents' sessions alive forever (disk grows) or GC after N days?" Design picks 30 days (completed) / 60 days (failed) for full-delete, with cold-archive as the bytes-reclaim tier between active and delete. `D7` gives an explicit user contract: 30-day-resumable.

### Decision-log collisions

Reviewed `PRD.md:560-585` decision log. No collisions. Specifically:

- **Foreground-default + auto-downgrade** (`PRD.md:14, 19`) — orthogonal; GC operates on terminal records only, never touches queued/foreground in-flight.
- **`ensemble_send` bypasses concurrency cap** (`PRD.md:262`) — orthogonal.
- **`ensemble_send` rejects running/paused/queued, accepts terminal** (`PRD.md:262`) — D8's two-gate invariant is consistent: GC only reclaims terminal-on-disk + registry-not-running. Send still works for terminal sub-agents that haven't been GC'd.
- **`/reload` preserves runs** (`PRD.md` decision 2026-05-XX, `shutdown.ts:67`) — design respects this. GC isn't triggered by reload (auto-trigger is on `session_start`, not `session_shutdown`).
- **Persona `inherit_context` audit** (`PRD.md:534`, question #16) — orthogonal.

### Resume-from-finished semantics — does PRD lock anything?

**No explicit lock.** PRD documents resume mechanics (`PRD.md:223-262`) — "send to a finished sub-agent resumes via `pi --session`" — but never specifies a retention contract. PRD #12 is precisely the gap, and the design closes it. Therefore D7's "30-day-resumable" introduces a **new contract** rather than colliding with an existing one. Slice 9 must update PRD §"Open questions" to mark #12 resolved and add to the decision log.

## 5. Hidden-assumption audit

### A. "Cold-archive preserves resume" (D7)

**Verifiable: ✓ partially.** The chain `runDir/session/<ts>_<uuid>.jsonl` → `validateSendable(run.sessionPath)` → `pi --session <path>` is grounded by `map.md:79-100` and `src/runs.ts:291-299`. **However:** D7's claim that "Pi will then re-create a new transcript.jsonl on the resumed run via the close-handler write path" (`design.md:194`) needs one more step. After archive:

- `run.transcriptPath` is still set (in-memory). After cold-archive removes the file, `processLine` continues `void appendFile(run.transcriptPath, ...)` (`runs.ts:770`). On Linux, `appendFile` opens with `O_CREAT|O_APPEND` so the file is recreated. **OK.**
- But the run dir itself must still exist. Cold-archive only `unlink`s `transcript.jsonl` (`design.md:69`); the dir survives. **OK.**
- **However** — once a run is GC'd from the in-memory registry (cross-session), the user types `ensemble_send <id>` and `validateSendable` runs `existsSync(run.sessionPath)`. If `session/<ts>.jsonl` is intact, the send succeeds; pi opens, conductor passes `--session-dir` pointing to the (still-existing) `<runDir>/session/`, and a new transcript starts. Contract met — but the **user-visible UX** isn't documented anywhere in the design. This is **A2**.

### B. First GC run on 11 GB tree < 500 ms (`design.md:225`)

Sanity check the syscall budget:

- 228 runs × `readdir(runsRoot)` = 1 readdir.
- 228 × `readFile(record.json)` ≈ 1.8 MB total (avg 7.9 KB; 219 actual + 9 spawn-failed without). Page-cache friendly.
- 219 × `stat(transcript.jsonl)` for size.
- 218 × `stat(session/)` (or readdir if needed for sidecar checks); cold-archive needs to know `.archived` presence so add 228 `existsSync(.archived)` and 228 `existsSync(.pinned)` = 456 `stat`s.

Total ≈ 1 + 228 + 219 + 228 + 456 ≈ **1130 syscalls + 1.8 MB read**. On warm cache, comfortably < 100 ms. On cold cache, dominated by ext4 readdir + file open latency, ~1–3 ms per dir-walk = 200–700 ms. **Claim "< 500 ms" is plausible-ish but not bulletproof on cold cache.** Acceptable for a startup-time cost; perf-test in slice 1 should set the bar at < 1 s cold.

### C. D2 default values

| Knob | Value | Defensible? |
|---|---|---|
| `completedTtlDays=30` | 2 sprints | ✓ defensible; matches typical post-mortem window |
| `failedTtlDays=60` | 2× completed | ✓ defensible |
| `totalSizeBudgetBytes=5 GB` | "half SSD slop" | weak rationale — depends on user's free space; **acceptable as default**, doctor surface lets users tune |
| `transcriptSizeCapBytes=100 MB` | "designer/planner top out 200–850" | ✓ in spirit; **A4** corrects the count claim |
| `orphanReconcileAfterHours=24` | "older than a day with no live process" | ✓ — well above any legitimate long-running timeout |
| `autoOnSessionStart=true` | conservative defaults make this safe | ✓ |

**Verdict:** all 7 defaults are defensible. The `5 GB` budget rationale is the weakest but the dual-with-age policy means it rarely fires alone; it's the safety net.

### D. Sidecar `.pinned` / `.archived` tolerance

Spot-checked `src/history.ts:15-80`. History uses injected `listRunIds()` which is just `readdirSync(runsRoot)` (returns dir names, not file names). **Sidecars inside a runDir don't affect the listing.** ✓

`statMtime(id)` uses "most recent mtime across record.json + final.md" — `touch .pinned` will NOT update those, so pin-state changes won't reorder history. ✓ But `touch .archived` happens *during* the GC pass, which also writes the archive marker just after unlinking transcript.jsonl. If `statMtime` is later refactored to glob the dir, it would see `.archived` mtime and reorder archived runs to the top. **Slice 8 acceptance must pin `statMtime` to record.json + final.md only; do not let sidecars influence ordering.** Worth a one-line note in slice 8 acceptance criteria.

`src/doctor.ts` reads no run records (`map.md:163`) — sidecars are invisible to it until slice 7 explicitly teaches it. ✓

### E. Allocator id reuse (inspector #6)

`src/runs.ts:147` — `allocateRunId` checks `!registry.has(id) && !existsSync(runDir(id))`. After full-delete, the dir is gone → id is reusable. Design defers hardening to v1.0 (`design.md:307`). **Acceptable.** The window for collision is `persona-<4hex>` = 65K per persona; on 228 runs over 5 days, P(birthday) ≈ 0.4% but rises to ~5% at 1000 runs. **Worth flagging in slice 5** as a follow-up: log `gc.id_reused` when a freshly-allocated id matches a previously-deleted one.

### Overall hidden-assumption finding

**1 explicit drift (A4)** + **1 underspecified UX (A2 — post-archive send)**. Cold-archive resume IS verifiable from sources, but the user-facing UX gap (what the resume's first card says, whether the partial new transcript confuses `/conductor history`) is not pinned by the design.

## 6. Slice-plan sanity

### Slice 1 — Inventory + policy (M)

Files: `src/gc/{inventory,policy}.ts`, `src/types.ts` (config), tests. **M is honest.** Inventory walks ~228 dirs; policy is pure but the table-driven test fixtures will be substantial (≥8 cases per `design.md:254-264`). **No split needed.**

**One concern:** the design has policy depend on `RunRegistry` (`design.md:148-152`) for the two-gate. That couples the supposedly-pure policy module to a runtime singleton. Design's resolution is "GC engine takes `RunRegistry` as a constructor arg" but the slice-1 file list doesn't include `runs.ts` (where `RunRegistry` lives). Either inventory or policy must accept the registry at call time. **Recommend slice 1 acceptance pin the seam: `walkInventory(runsRoot, registry, now)` returns `InventoryEntry { ..., inMemory: Run | undefined }` so policy stays purely a function of `(InventoryEntry[], Config, NowMs)`.** This matches the §3 architecture diagram's intent.

### Slice 5 — Orchestrator + auto trigger (M, deps 1+2+3+4)

✓ deps correct. **Could be split** into 5a (orchestrator `runGc(opts)`) and 5b (auto-trigger wiring in `src/index.ts`'s `session_start` handler + 6 h debounce + `last-gc` mtime). Splitting would let critic gate the orchestrator engine separately from the lifecycle wiring. Designer flagged this as a critic-gated slice (`design.md:296`). **Recommend planner consider 5a/5b split if critic flags blast risk.** Not required — M is plausible if the wiring stays mechanical.

### Slice 8 — History display (S, deps 4+3)

Acceptance per `design.md:182`: "mark archived/pinned in `/conductor history`". **Risk of UI bleed:** `/conductor history` currently outputs a single line + optional excerpt per run (`history.ts:60-78`). Slice 8 needs to *append* a marker glyph (e.g. `📌` for pinned, `📦` for archived) to the existing line — no schema change, no new section. **Acceptance must explicitly forbid:** new sections, sort-order changes, summary footers, or any structural reflow. ## 7. Risks the design under-states

Design §5 enumerates 6 risks (`design.md:191-227`). Adding:

| New risk | Severity | Mitigation |
|---|---|---|
| **R7. Quit-vs-next-start orphan window.** `session_shutdown` reason="quit" SIGTERMs but never `writeRecord`s (`shutdown.ts:74-82`). Until next `session_start` runs reconcile, on-disk says `running`. If user starts a *different* tool against that record (vault note citing the id, dashboard scrape) they see stale state. | Low | **A1** — flip running→killed in `handleSessionShutdown`; one extra `writeRecord` per running child. Bounded loop, best-effort consistent with everything else in `runs.ts`. |
| **R8. ensemble_send to archived run UX.** D7 says "session/ preserved → send works" but never specifies what the user sees. Does the new turn's transcript card show "(transcript starts here; prior history archived)"? Does it silently start a fresh-looking transcript? | Medium | **A2** — pin in design and slice 3 acceptance. Probably: send works, new transcript created, doctor surface notes "post-archive resume" in history annotation. |
| **R9. Dry-run output not actionable enough.** D6 documents `--dry-run` exists; doesn't say *what* it prints. Just "would delete X" is weak; users need bytes-reclaimed totals to decide. | Low | **A3** — slice 6 acceptance pins `gc --dry-run` output to include `bytes_to_reclaim`, `runs_to_archive`, `runs_to_delete`, `runs_lose_resume`, plus per-entry list. |
| **R10. id-reuse-after-GC tooling confusion.** Inspector #6 deferred to v1.0; risk is real today since dogfood vault notes already cite `inspector-*` ids. | Low | Add a `gc.id_reused` log line in slice 5 when the allocator picks a freshly-deleted id. Not a blocker; planner can defer. |
| **R11. `last-gc` debounce-marker placement.** Design says "6 h debounce on `session_start`" (`design.md:299`); doesn't specify where the marker lives. `~/.pi/agent/conductor/runs/.last-gc` collides with the runs root being walked. | Low | Slice 5 acceptance: marker lives at `~/.pi/agent/conductor/.last-gc` (root, not under runs/). |

**Of these:** A1 (R7) and A2 (R8) are the substantive amendments; A3 (R9) is acceptance-criteria polish; R10/R11 are flags for the planner to consider but don't gate.
