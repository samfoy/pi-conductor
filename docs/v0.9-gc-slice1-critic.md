## 5. Forward-looking concerns for Slice 2/3

### Slice 2 (orphan reconciliation + A1 shutdown patch)

**Can Slice 2 trust the `reconcile-orphan` action descriptor?** Yes, with one caveat:
- `ReclaimAction` for `reconcile-orphan` carries only `id` + `reason`. Slice 2's `reconcileOrphans` will need the `runDir` to re-read `record.json` for the writeRecord. That's available on the `InventoryEntry` (which Slice 2 already has access to via the orchestrator), so the action descriptor is sufficient — Slice 2 should join action ↔ entry by `id`. Good.
- The orphan TTL gate at `policy.ts:178` uses `transcriptMtime ?? startTime`. Slice 2's tests will need to mirror this fallback (matters when a run never wrote a transcript).
- `decideForEntry` returns `reconcile-orphan` for `status === "running" && !inMemory && stale`. Slice 2's executor must check `entry.inMemory === undefined` *again at execution time*, not just trust the policy snapshot — there's a TOCTOU window where a previously-orphaned entry could come back to life (resumed via `ensemble_send`) between policy and execution. Plan §A2 already mentions this; pin in Slice 2 acceptance.

### Slice 3 (cold-archive + delete executor)

**Can Slice 3 trust the action descriptors?**
- `delete` carries `id`, `reason`, `bytesReclaimed`, `losesResume`. Slice 3 only needs `id` to find the runDir; `bytesReclaimed` is reporting metadata, `losesResume` is for doctor surface. **Sufficient.**
- `cold-archive` carries `id`, `reason`, `bytesReclaimed`. Slice 3 will need to write `<runDir>/.archived` (sidecar mtime = archive timestamp per D5) + truncate or unlink `transcript.jsonl`. The action does NOT specify which transcript files exist; Slice 3 must re-stat. Plan §3 acceptance covers this.
- D8's two-gate invariant: Slice 3 MUST re-check `entry.inMemory === undefined && isTerminal(status)` against the live registry at execution time (not just the policy snapshot). Otherwise a fast-completing run could be archived while still being written. Plan §3 risks already calls this out.

### Cross-cutting

- `InventoryEntry.totalSizeBytes` excludes everything except `transcript.jsonl` + `record.json` + `final.md`. The `session/` dir bytes are NOT counted. On a run where pi has cached >100 MB of session JSONL, GC budget math under-counts. Document or extend in Slice 3 (when reclaim accuracy matters). For Slice 1 — acceptable; the bias is "GC reclaims less than promised," not "GC is unsafe." Flag in design follow-ups.
