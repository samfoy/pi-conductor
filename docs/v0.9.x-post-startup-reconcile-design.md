## 1. Goals & non-goals

### Goals

- **Eliminate the orphaned-`running` blind spot.** When a host event
  destroys the pi-conductor extension's in-memory state (pi-dashboard
  systemd restart, OS reboot, OOM kill of the parent pi process)
  while sub-agent run dirs survive on disk with `status: "running"`,
  the new pi runtime today returns "No sub-agents" and the
  records sit forever. Fix: at extension load, scan `runsRoot/*/record.json`,
  classify each `running` record as **alive** (re-adopt) or **dead**
  (reclassify), and ensure both states are visible to `ensemble_status`,
  `ensemble_send`, `/conductor history`, and the GC.
- **Distinguish the two restart paths.** The `/reload` case (commit
  `1c72856`) intentionally spares running children; the pi-dashboard
  restart case kills them. Both wipe the in-memory registry. The new
  reconcile MUST honor the `/reload` invariant ("don't murder live
  children just because the runtime restarted") while still reclaiming
  the genuinely-dead orphans.
- **Make orphans resumable.** A reclassified-as-`crashed` record with
  a valid `sessionPath` should accept `ensemble_send <id> "<message>"`,
  spawning a fresh `pi --session <path>` and re-attaching to the same
  conversation. The user-visible promise: "you can pick up where the
  dead critic left off."
- **Coexist with v0.9 GC.** The post-startup reconcile must run
  **before** the v0.9 GC sweep on the same `session_start`, so GC sees
  the reclassified records (not orphan-`running` ones) and applies
  its size/age/pin rules cleanly.

### Non-goals

- **Multi-host coordination.** Two pi-conductor extensions sharing the
  same `runsRoot` is out-of-scope (Q9 below). Single-host, single-runtime.
- **Replacing the v0.9 GC orphan reconcile.** The GC's `reconcile-orphan`
  rule (`docs/v0.9-gc-design.md` D5, `src/gc/policy.ts:165ŌĆō188`) keeps
  doing what it does today: reclaim `running` records older than
  `orphanReconcileAfterHours` (default 24h) during the regular GC pass.
  This design adds a **second, earlier** reclassifier that fires at
  startup and uses a stronger signal (process liveness) so we don't have
  to wait 24h for orphans to clear.
- **Daemonizing.** The reconcile is one-shot at extension load. No
  background timer, no periodic re-scan. (The watchdog already covers
  mid-session liveness.)
- **Auto-resume.** Reclassified records are visible and `ensemble_send`-able,
  but the conductor does not automatically re-spawn the dead sub-agent.
  Resume is user-initiated (`ensemble_send <id>` with whatever the user
  wants to say next).
- **Cross-process resumption while the original pi is still alive.**
  If a record is `running` AND the original pi is alive AND the registry
  is empty (i.e. `/reload` with surviving children), we re-adopt the
  child as a `Run` in the registry but **do not** re-establish the
  event stream. The original pi's events will land on disk
  (`record.json` updates, `final.md`); the conductor sees the
  re-adopted run as `running` until it terminates naturally and
  the next reconcile or GC pass picks up its terminal state. (Q5
  below ŌĆö "stream re-attach" is a v2 problem.)

## 3. Detection algorithm

```ts
// src/reconcile-startup.ts (new module)

export interface PostStartupReconcileResult {
  readopted: string[];          // running children of /reload preserved
  reclassified: string[];       // dead orphans flipped to killed/failed
  unresumable: string[];        // killed but sessionPath missing on disk
  errors: Array<{ id: string; error: string }>;
}

export interface PostStartupReconcileDeps {
  runsRoot: string;
  registry: RunRegistry;
  /**
   * Liveness probe. Returns true iff a process with this pid is alive
   * AND looks like a pi sub-agent (heuristic: `kill(pid, 0)` plus
   * optional /proc/<pid>/cmdline contains "--session"). Defaults to
   * `defaultLivenessProbe` which uses process.kill(pid, 0).
   */
  isAlive: (pid: number) => boolean;
  /** ms epoch; injected for tests. */
  now: number;
}

export async function reconcileOrphansAtStartup(
  deps: PostStartupReconcileDeps,
): Promise<PostStartupReconcileResult>;
```

### Per-record decision tree

```
for each <runsRoot>/<id>/record.json:
  raw ŌåÉ readFile()                           // skip on ENOENT
  record ŌåÉ JSON.parse(raw)                   // collect to errors on parse fail
  if registry.has(record.id):                // shouldn't happen at session_start
    continue                                 // (registry is freshly empty)
  if record.status Ōłł TERMINAL_STATUSES:
    continue                                 // GC handles these; not our job
  if record.status === "queued":
    flip to failed; errorMessage = "orphaned: queue entry abandonedŌĆ”"
    persist; do NOT re-adopt
    add to reclassified
    continue
  if record.status === "paused":
    // paused without a live process is a contradiction we resolve
    // toward dead (paused is in-memory-only state, never persisted
    // intentionally). Treat like running.
  // status is now running (or paused-as-running):
  if record.pid === undefined:
    flip to killed; errorMessage = "orphaned: pre-pid-schema recordŌĆ”"
    if sessionPath exists on disk:
      registry.register(buildOrphanRun(record, status=killed))
    else:
      add to unresumable
    add to reclassified
    continue
  if isAlive(record.pid):
    // /reload case: live child, runtime wiped. Re-adopt.
    registry.register(buildOrphanRun(record, status=running, proc=undefined))
    add to readopted
  else:
    // pi-dashboard restart / OOM / OS reboot. Dead orphan.
    flip to killed
    errorMessage = "orphaned: process gone (post-startup reconcile)"
    finishedAt = now
    persist
    if sessionPath exists on disk:
      registry.register(buildOrphanRun(record, status=killed))
    else:
      add to unresumable    // still register but mark sessionPath unset
    add to reclassified
```

### `buildOrphanRun(record, status, proc?)` ŌĆö partial Run construction

The re-adopted/reclassified `Run` is a degraded view of the on-disk
record:

- `messages: []` ŌĆö full history is in `transcript.jsonl` on disk; the
  conductor doesn't re-stream it into memory (would require a parser
  for the JSONL stream format and would inflate RSS for large
  transcripts). The first `ensemble_send` resume populates a fresh
  message stream.
- `usage: emptyUsage()` ŌĆö same reasoning; cumulative usage is preserved
  in `record.usage` on disk and re-read on next `writeRecord` after a
  resume.
- `proc: undefined` ŌĆö even in the re-adopt case. We don't have a handle
  on the original ChildProcess.
- `lastEventAt: record.finishedAt ?? record.startTime` ŌĆö best
  available signal.
- All event-stream-driven fields (`stalledSince`, `nonSubstantiveFinal`,
  `lastToolCall`) start undefined.

The re-adopted `Run` is honest about its degraded state: `validateSendable`
will accept it once it reaches a terminal status; `ensemble_kill` can
SIGTERM via `record.pid` if we re-stash that on the Run (yes, slice 1
adds `Run.pid?: number` too).

### Liveness probe

```ts
export function defaultLivenessProbe(pid: number): boolean {
  try {
    process.kill(pid, 0);    // signal 0 = check existence + permission
    return true;
  } catch (e: unknown) {
    const err = e as NodeJS.ErrnoException;
    if (err.code === "ESRCH") return false;
    if (err.code === "EPERM") return true;  // process exists, we can't signal
    return false;             // unknown errno; assume dead
  }
}
```

Rationale for skipping `/proc/<pid>/cmdline` cross-check:
- **False-negative risk:** macOS, BSD, containers without `/proc`.
- **False-positive risk:** the original pid was reused by another
  process. PID reuse on Linux requires `pid_max` (~32K) overflow; with
  modern kernels (`pid_max = 2^22 = 4M`), reuse within a single startup
  cycle is functionally impossible. Documented in ┬¦6 R2.
- **If false-positive ever bites:** the worst case is we re-adopt a
  ghost; the next `ensemble_send` will fail when the orphan run
  doesn't reach a terminal state. User runs `/conductor reconcile
  --force` or kills the run manually. Acceptable v1 behavior.

## 4. Lifecycle integration

### Where it fires

In `src/index.ts:289` (`pi.on("session_start", ŌĆ”)`), insert reconcile
**before** the existing `setImmediate(maybeAutoRunGc)` block, but
inside its own `setImmediate` so it doesn't block the bootstrap path:

```ts
pi.on("session_start", async (_event, ctx) => {
  cwd = ctx.cwd;
  ctxRef = ctx;
  // ŌĆ” widget mount, focused-overlay shortcut install ŌĆ”

  // v0.9.x: post-startup reconcile of orphaned running records.
  // Runs BEFORE the v0.9 GC sweep so reclassified records are
  // visible to GC's age/budget rules, not the orphan-TTL path.
  // Pure registry-mutation + on-disk record flips; no expensive I/O.
  // Failure here MUST NOT break session bootstrap ŌĆö best-effort only.
  setImmediate(() => {
    void reconcileOrphansAtStartup({
      runsRoot: runsRoot(),
      registry,
      isAlive: defaultLivenessProbe,
      now: Date.now(),
    }).catch((err: unknown) => {
      console.error(`reconcile-startup: failed: ${(err as Error)?.message ?? String(err)}`);
    });
  });

  setImmediate(() => {                  // v0.9 GC ŌĆö UNCHANGED
    const cfg = loadConfig(cwd);
    void maybeAutoRunGc({
      runsRoot: runsRoot(),
      config: cfg.gc,
      registry,                         // now contains the re-adopted/reclassified entries
    }).catch((err: unknown) => { ŌĆ” });
  });
  // ŌĆ” watchdog start ŌĆ”
});
```

### Ordering within session_start

```
t0  session_start fires
t0  widget mounted, focused-overlay shortcut installed
t0+ setImmediate #1: reconcileOrphansAtStartup
       reads runsRoot/*/record.json (~10ŌĆō50 files typical, ~500 max)
       calls isAlive() per pid (cheap; signal 0 syscall)
       writes back N records (only the dead ones)
       registers M Run entries in registry
       returns
t0+ setImmediate #2: maybeAutoRunGc
       reads inventory (sees re-adopted entries via registry, sees
         reclassified records' new statuses on disk)
       runs policy ŌåÆ reclaim
t0+ watchdog starts (subscribes to registry; sees re-adopted entries)
```

The two `setImmediate`s are sequential because the v8 microtask queue
is FIFO; the reconcile completes before GC's first await in the same
tick batch. (Verifier should pin this with a test using awaited
`setImmediate`-equivalent fakes.)

### Why not block on `await` in the `session_start` handler

Pi's bootstrap latency is user-visible. Even 200ms of disk I/O on
session_start would feel sluggish. `setImmediate` defers the work
to the next event-loop iteration ŌĆö UI stays responsive, the user
doesn't wait. Best-effort: if the disk is slow or the `runsRoot`
is on NFS, the reconcile catches up by the user's first turn.

### Interaction with v0.9 GC (`docs/v0.9-gc-design.md` D5)

After post-startup reconcile, the GC's `decideForEntry` (`src/gc/policy.ts:158`)
sees:

- Re-adopted runs (`/reload` survivors): `inventory.inMemory` is set
  (registry has the entry), so GC's Rule 1 ("active in registry")
  applies: `keep`. Good.
- Reclassified runs (genuinely dead): `inventory.status` is `killed`
  (or `failed` for queued orphans). `inventory.inMemory` is set
  (registry has the entry). GC's Rule 1 fires on the active-in-memory
  gate, but the gate has a sub-condition: `proc !== undefined ||
  !isTerminal(mem.status)`. Re-adopted-as-`killed` runs are terminal
  AND `proc === undefined`, so Rule 1 does NOT fire ŌĆö GC falls through
  to its standard age/size/transcript-cap rules. Correct.

This is the load-bearing interaction. Slice 2 must pin a test:
"after post-startup reconcile, GC's policy treats reclassified-killed
runs identically to records that died and were never re-adopted."

### Interaction with the existing GC orphan reconciler (`src/gc/reconcile.ts`)

Different. The GC orphan reconciler (`reconcileOrphans` in
`src/gc/reconcile.ts`) only fires from `gc/index.ts` AFTER the
inventoryŌåÆpolicy phase, and it's gated by the orphan-TTL
(`orphanReconcileAfterHours`, default 24h). With post-startup reconcile
landing first, the GC orphan reconciler sees no `running` records
(they're already reclassified or re-adopted). It becomes a fallback
for runs that orphaned mid-session (e.g. a sub-agent whose pi crashed
without firing a close handler).

### Schema bump: `pid` field

### `ensemble_send` resume contract

`validateSendable` (`src/runs.ts:1095`) ŌĆö already accepts terminal
states with `sessionPath`. Verified against today's code:

```ts
if (run.status === "running") return { ok: false, reason: "currently running" };
if (run.status === "paused") return { ok: false, reason: "paused" };
if (run.status === "queued") return { ok: false, reason: "queued" };
if (!run.sessionPath) return { ok: false, reason: "no resumable session" };
if (!existsSync(run.sessionPath)) return { ok: false, reason: "session file missing" };
return { ok: true };
```

A reclassified-as-`killed` run with `sessionPath` set passes all five
checks. `sendToRun` (`src/runs.ts:1147`) re-spawns `pi` with
`--session <path>`. **No code changes needed** in `validateSendable`
or `sendToRun` ŌĆö slice 3's "wiring" is just verification + a
regression test that `ensemble_send <reclassified-id>` works end-to-end.

For unresumable orphans (sessionPath missing): `validateSendable` rejects
with "session file is missing on disk." User sees a clear error. No
silent failure.

## 8. Failure modes

| # | Failure | Surfaced where | Recovery |
|---|---|---|---|
| F1 | `runsRoot` doesn't exist (fresh install). | `readdir` rejects with ENOENT. | Reconcile catches at the top level, logs once, returns empty result. Bootstrap continues. |
| F2 | One `record.json` is corrupt (truncated write from a prior crash). | Per-record try/catch in the scanner. | Add to `errors[]` with parse-error message; continue with others. Surface count in doctor. |
| F3 | `kill(pid, 0)` raises an unexpected errno (not ESRCH/EPERM). | `defaultLivenessProbe` catches, returns `false` (treat as dead). | Conservative ŌĆö false-positive reclassify is recoverable; false-negative re-adopt of a dead process is the same as today's "no reconcile at all" baseline. |
| F4 | Pid was reused by an unrelated process. | False-positive readopt; ensemble_send to it fails when `pi --session` doesn't recognize the orphan as resumable. | `validateSendable` rejects on missing/invalid sessionPath; user runs `/conductor reconcile --force`. Documented in ┬¦6 R2. |
| F5 | Concurrent GC pass writes the same record while reconcile is mid-flight. | Both writes are full-file `JSON.stringify`. Last-write-wins, no merge. | Reconcile fires from `setImmediate #1`; GC fires from `setImmediate #2`. Sequential by virtue of ordering, not locking. Document the load-bearing ordering. |
| F6 | Reconcile takes too long (e.g. 10K records on slow disk). | `setImmediate` defers; user can use the UI while it runs. | If reconcile is mid-pass when the user spawns a new sub-agent, the new spawn's `registry.register` doesn't conflict (different id). |
| F7 | `pid` field exists in record but is wildly wrong (e.g. a stored `0`). | `defaultLivenessProbe(0)` raises EINVAL on most kernels ŌåÆ caught, returns `false` ŌåÆ reclassify. | Conservative; matches "garbage pid = dead" intuition. |
| F8 | `/reload` re-fires `session_start` immediately after a reconcile from the same runtime. | Reconcile re-runs; sees the just-registered Run; classifier returns `skip-already-in-registry`; no-op. | Idempotency is a slice 2 invariant (W4 above). |

## 9. Risks & open questions

### Risks

- **R1. Multi-host conflict.** Two pi-conductor extensions on the same
  `runsRoot` could both run reconcile concurrently and each see the
  other's pids as "alive in this pid namespace" ŌĆö leading to mutual
  re-adoption. Out-of-scope for v0.9.x; mitigation: documented in ┬¦1
  non-goals. Future v2 could add a `<runsRoot>/.reconcile-lock` file,
  same shape as `.last-gc`.
- **R2. PID reuse across reboots.** A run from yesterday's session has
  pid 12345; today's bootstrap has a fresh process at the same pid 12345
  (rare but possible: the kernel's `pid_max` is bounded and after a
  reboot it resets). Mitigation: if `pid_max` is small enough that this
  bites in practice, slice 1's classifier could also persist the pi
  process's start-time (read from `/proc/<pid>/stat` `starttime` field)
  and compare on reconcile. Punt to dogfood evidence; classifier API
  shape supports adding the start-time arg later without breaking
  existing callers.
- **R3. Re-adopted runs are read-only (no event-stream re-attachment).**
  The `/reload` case re-adopts a live child but doesn't re-stream its
  events. The user sees `lastEventAt` frozen; widget shows it as
  "running but stale." Not a regression (today: the `/reload` case is
  silently broken; the user sees nothing at all about surviving
  children). Mitigation: doctor surface explicitly notes "re-adopted
  runs are read-only; they will eventually terminate and write final.md
  on disk." Future v2 could add a transcript-tail watcher that streams
  new transcript.jsonl appends back into `Run.messages`.
- **R4. Schema bump risk.** Adding `pid?: number` to `RunRecord` is a
  forward-compatible change (older readers ignore unknown fields, JSON
  is permissive). Older WRITERS will produce records without the field;
  reconcile handles those via `reclassify-pre-schema`. No migration
  needed. Documented in Q11.

### Open questions for the planner

- **OQ1. Should the conductor re-attach the transcript stream?** Per Q6
  / R3 above, re-adopted runs are read-only. Is a transcript-tail
  watcher (read appended bytes from `transcript.jsonl`, parse, push to
  `Run.messages`) worth a v0.9.x slice 5? My recommendation: defer to
  v2; the `/reload` case is rare relative to dashboard restarts, and
  read-only re-adoption is strictly better than today. Surface in the
  design's "Open questions" so the planner can ask the user.
- **OQ2. Should reconcile fire on every `session_start` or only on the
  *first*?** The first session_start of a given runtime is the only
  one that finds wiped state; subsequent `session_start`s within the
  same runtime have a populated registry, so reconcile is a no-op.
  Cheap to fire every time (idempotent), but could be guarded by a
  module-scoped `let firstStart = true; if (!firstStart) return;`
  flag for clarity. Recommendation: ship "every session_start"; the
  no-op path is free.
- **OQ3. Should `ensemble_kill` on a re-adopted run send SIGTERM via the
  persisted pid?** Today's `forceTerminate` requires `run.proc`. For
  re-adopted runs, `proc === undefined`. Two choices: (a) extend
  `forceTerminate` to fall back to `process.kill(run.pid, "SIGTERM")`
  when `proc === undefined` but `pid` is set; (b) reject `ensemble_kill`
  on re-adopted runs with "killed externally ŌĆö wait for natural exit."
  Recommendation: (a). A few-line patch in `src/runs.ts:forceTerminate`,
  pinned by a slice-2 test. List as in-scope for slice 2 in the
  planner's expansion.

## Findings for planner

- **OQ3 is in-scope for slice 2.** `forceTerminate` falling back to
  `process.kill(run.pid, sig)` when `run.proc === undefined` is a
  small patch but it's required for the user's mental model
  (`ensemble_kill <re-adopted-id>` should "just work"). The planner
  should add a sub-bullet to slice 2's spec: "extend `forceTerminate`
  with a pid-fallback when `run.proc` is unset." Pin a W7 mutation
  witness that drops the fallback branch.
- **R2 (PID reuse) is dogfood-driven.** The classifier API supports
  adding a `procStartedAt` parameter without breaking; the planner
  should treat this as a "future expansion" annotation, not a slice
  in v0.9.x.
- **OQ1 (transcript-tail re-attach) is v2.** Document explicitly in
  the planner's "deferred" list.
- **No oracle pre-pass requested.** This is a tight, mechanical
  mirror of v0.9 D5 with one schema bump. The builder Ōćä critic loop
  on slices 1 and 2 should catch any ambiguity surfaced during
  implementation. If the builder finds a load-bearing surprise, send
  back to the designer.
