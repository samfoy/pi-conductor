## 5. Two-gate safety (design D8)

Both gates exercised:

- **Gate 1 (registry):** `tests/gc-executor.test.ts:131` "skips active
  runs (re-checks registry) — LOAD-BEARING witness target" passes
  `registryActive = new Set(["inspector-cccc"])` and asserts `archived =
  []`, `failed[0].error =~ /active during reclaim/i`, **and** the
  on-disk `transcript.jsonl` is intact post-execute. Witness A (above)
  proved this gate has teeth.

- **Gate 2 (disk):** `tests/gc-executor.test.ts:156` "skips runs whose
  status changed to non-terminal since plan" creates a fresh runDir
  with `record.json` containing `status: "running"`, runs cold-archive,
  asserts `archived = []`, `failed[0].error =~ /non-terminal|status
  changed|running/i`, **and** the on-disk transcript is intact.
  Witness B proved this gate has teeth.

- **Disagreement direction:** D8 specifies "skipping unless both agree
  is the safe direction." `src/gc/executor.ts:79-130` short-circuits on
  *either* gate failing — registry-active OR disk-non-terminal both
  classify as `failed[]`, neither path proceeds. The `||`-style
  semantics asked for by D8 are honored ("skipped unless both gates
  agree").

- **Reordering risk:** Gate 1 fires before the record read; Gate 2
  fires after the record read but before any disk mutation. No fs writes
  happen on the failure paths. Verified by reading the executor
  control-flow.

Two-gate safety: ✅ encoded and witnessed.

The contract is correctly partitioned: policy filters by pinning,
executor preserves the sidecar bytes. Slice 4 only needs to write
`.pinned` and rely on the existing `inventory.pinned` field — no
