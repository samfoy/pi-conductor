## P0 — workflow defects

### Parent-reload contract for context-filter / runs changes

The pi-conductor extension loads from `./src/index.ts` at parent-session
startup with no live-reload mechanism. This means iterating on
`src/context-filter.ts`, `src/runs.ts`, or any other extension surface
requires restarting the parent pi session — losing scratchpad state,
in-flight sub-agent records, and the active conductor brief. v0.8.1 Item 1's
§8.3 manual smoke gate could not run in-session for this reason; the slice
shipped on static evidence (mutation test + byte-identical sentinel + 9 new
green tests + out-of-band `tsx` smoke) instead.

**2026-05-15 corroboration.** The §8.3 close required a parent pi session
restart between commit `423f500` and the smoke run; verifier-xszh confirmed
that all four post-`423f500` runs from the originating session
(critic-w1dj, builder-axqy, builder-s3d8, finalizer-7794) carried the OLD
sentinel, while the post-restart session's spawns (critic-tppr,
clarifier-q5h2) carry the NEW §4.1 sentinel. The reload tax is empirically
load-bearing for any context-filter or runs.ts iteration.

Decide one of:

- (a) Document the reload contract for contributors (acceptable but a tax).
- (b) Build a reload-on-disk-change mechanism for the extension.
- (c) Provide a `pnpm dev`-style watch flow that auto-restarts the parent.

Triage in early v0.8.2.
