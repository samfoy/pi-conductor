## P0 — workflow defects

### Parent-reload contract for context-filter / runs changes

The pi-conductor extension loads from `./src/index.ts` at parent-session
startup with no live-reload mechanism. This means iterating on
`src/context-filter.ts`, `src/runs.ts`, or any other extension surface
requires restarting the parent pi session — losing scratchpad state,
in-flight sub-agent records, and the active conductor brief. v0.8.1 Item 1's
§8.3 manual smoke gate could not run in-session for this reason; the slice
shipped on static evidence (mutation test + byte-identical sentinel + 9 new
green tests + out-of-band `tsx` smoke) instead.

**2026-05-15 corroboration.** The §8.3 close required a parent pi session
restart between commit `423f500` and the smoke run; verifier-xszh confirmed
that all four post-`423f500` runs from the originating session
(critic-w1dj, builder-axqy, builder-s3d8, finalizer-7794) carried the OLD
sentinel, while the post-restart session's spawns (critic-tppr,
clarifier-q5h2) carry the NEW §4.1 sentinel. The reload tax is empirically
load-bearing for any context-filter or runs.ts iteration.

Decide one of:

- (a) Document the reload contract for contributors (acceptable but a tax).
- (b) Build a reload-on-disk-change mechanism for the extension.
- (c) Provide a `pnpm dev`-style watch flow that auto-restarts the parent.

Triage in early v0.8.2.

### dist/ rebuild lifecycle gap

Commit `3949fc4` switched `pi.extensions` from `./src/index.ts` to
`./dist/index.js` to bypass jiti's per-startup TypeScript transpile. The
unit suite still runs against `src/` via `tsx`, so a `src/` change can
land green-tested without the corresponding `dist/index.js` rebuild — and
the runtime extension would silently no-op the new code.

This nearly happened in the v0.8.2 (B-4) sanitizer + (A) `ensemble_kill`
work this session: both slices committed `src/` changes that left
`dist/index.js` stale. The conductor caught the gap at finalizer-gate
time and routed the rebuild as commits `ff9a40e` + `cfeffb5`. Mid-flight
discovery is fragile — the gap should be enforceable.

Decide one of:

- (a) Wire `npm run build` into the `pre-commit` hook so any `src/`
  change automatically bundles `dist/`.
- (b) Add a pre-commit gate that fails if `git diff --cached` touches
  `src/` without also touching `dist/index.js`.
- (c) Add a separate `npm run check:bundle` that asserts
  `dist/index.js` was generated from current `src/` (e.g. timestamp
  comparison or content hash).

Triage in early v0.8.2.

### Stale SHA citations in backlog after `git commit --amend`

Three slices this session had their backlog CLOSED entries cite the
pre-amend SHA instead of the on-branch SHA:

- §11 verifier-brief sub-rule: `d9846ed` → actual `1f70d08`. Caught by
  `critic-rgbf`, fixed in `e2ac8cf`.
- (B-4) sanitizer: `2a37f6b` → actual `e9c0c1e`. Caught by
  `finalizer-gkke`, fixed in [this commit's SHA].
- (A) `ensemble_kill` tool: `045f0be` → actual `24b6e97`. Caught by
  `finalizer-gkke`, fixed in [this commit's SHA].

Each amend was triggered by the steering hook rejecting `§` in commit
subjects (separate session-observation item above) — the builder /
scribe edited the backlog with the pre-amend SHA, then the steering
hook forced an amend on the subject swap, leaving the backlog citation
pointing at an orphan.

Pattern recurred 3× in one session. Stale SHAs resolve today via
reflog (~30 day window) and silently break thereafter. Two mitigations:

- (a) Builder/scribe convention: update backlog SHA citations *after*
  the final amend, not before.
- (b) Reference commits in backlog by descriptive title or
  date-anchored phrase ("§11 verifier-brief slice, 2026-05-15") and
  derive SHA at lookup time via `git log --grep`.

(a) is convention-only; (b) is harder to enforce but more robust.

**Suggested slice:** edit `personas/builder.md` and `personas/scribe.md`
to add a "verify SHAs against final amend" step after commit; or add a
prepare-commit-msg hook that scans the staged backlog diff for stale
SHA references.
