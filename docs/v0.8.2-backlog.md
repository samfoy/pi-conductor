### 1. Zombie sub-process detection â€” independent of `kill_on_stall`

**Witnessed:** `builder-mtpt` pi subprocess died mid-first-bash-call. Transcript ended with one `tool_use` and zero `tool_result`. The pi process disappeared from `ps -ef`. The run record stayed `status: running` until manual `ensemble_kill` 12 minutes later. The v0.10 watchdog correctly emitted soft (132s) and hard (612s) advisories, but `kill_on_stall: false` (the default) meant the hard advisory was advisory-only â€” the run held its `m_w_c=1` write slot until a human (the orchestrator) killed it.

**Second witness 2026-05-21 (`builder-shzs`, slice 2):** same shape. Cause attributed by the user to a **pi-dashboard server restart** â€” the host's pi infra restarted out from under the running subagent subprocess. Not an internal pi/conductor failure mode; an external host event. The watchdog's behavior was correct (silent transcript â†’ soft + hard advisories). `kill_on_stall: true` was set on this spawn, so the hard threshold should have escalated to `forceTerminate`; verify the enforcer kicks for hard-when-pid-already-gone (probably no-op safe but worth a test).

**Third witness 2026-05-21 (`builder-utrr`, slice 2 retry):** same shape, ~39m duration. Investigation pinned the actual root cause to **pi-dashboard's `_healthCheck` idle-reaper at `backend/pi-manager.ts:_healthCheck`** â€” reaps slots that are `!pi.running && idle > 30 min`. Background sub-agents (`ensemble_spawn` with `foreground: false`) end the parent's turn immediately on tool return; the parent slot then looks idle from pi-dashboard's POV while a background sub-agent is still doing real work. After 30 min the parent slot was getting `gracefulShutdown()`-ed, killing the conductor extension and orphaning all its sub-agents. Patched 2026-05-21 by disabling the reap block (preserved as commented-out code with a 2026-05-21 dated rationale). Witnessed dirty tree salvaged via `stash@{0}`.

### Proper fix (separate from the zombie-detection backlog item)

**Conductor should publish run-state into pi-dashboard so background sub-agents count as parent activity.** Sketch: when the conductor extension has at least one `Run.status === "running"` in the registry, emit a heartbeat that bumps the parent slot's `_lastActivity` (via a synthetic event the pi-manager's `tool_execution_update` handler already updates on, or a new `agent_busy` channel). The reaper can then be re-enabled safely with the same 30-min default; idle slots that *truly* have no background work will still be reaped, but slots actively orchestrating sub-agents won't.

Until this lands, the reaper is off in pi-dashboard. Memory cost: idle slots hold RSS until manually closed. Acceptable trade-off given the bug.

### 3. Watchdog hostility to long quiet bash tool calls (`builder-ccl8`, fourth slice-2 witness)

**Witnessed:** `builder-ccl8` was killed by its own watchdog at 622s silent. Cause: a single bash call ran `bash /tmp/wdd-verify-slice2.sh 2>&1 | tail -90` â€” a multi-witness WDD verification script piped through `tail -90`. `tail -N` buffers until the upstream pipe closes, so 8 sequential `npx tsx --test` invocations (~30â€“60s each cold-start) emitted no output to the parent for ~10 minutes. The `kill_on_stall: true` setting on the spawn caused the watchdog to escalate the hard threshold to `forceTerminate`. Working as designed; bad outcome.

**Question for design:**

- Should `tool_execution_update` heartbeats from a still-running bash call count as activity for the watchdog? Today the watchdog observes `tool_execution_start` / `_end`; a single multi-minute bash call has only the `_start` and the eventual `_end`, with no heartbeat between. Pi may or may not surface partial-output deltas as `tool_execution_update` events on long bash; verify.
- Alternative: surface `last_tool_call_at` in addition to `last_message_at`, and treat a tool-call still in flight as "agent is alive" with a much higher threshold (e.g. 30 min) than the no-tool-call silence threshold (120s).
- Either way, the current per-spawn `kill_on_stall: true` semantics is too coarse. A spawn might want "kill if NO tool call has been started in 600s" but "don't kill mid-tool-call regardless of silence."

**Builder-prompt rule learned (orchestrator-side mitigation, applied to slice-2 retry brief):**

- Never pipe a long-running script through `| tail -N` or `| head -N` â€” those swallow streaming output.
- Break long verification into one bash call per logical unit (one witness, one test file, one mutation) so each call ends and surfaces a `tool_execution_end`.
- Use `tee /tmp/log` or `set -x` if a multi-step bash call genuinely must run together.
- Use `git checkout <file>` to revert mutations â€” fast and uniform.

May also be worth pinning these into `personas/builder.md` if the failure recurs.

### 5. Builder anti-pattern: real-subprocess smoke tests with `--test-timeout=0` (`builder-ccl8` / `builder-ew9e` triage) â€” CLOSED 2026-05-21 (`85ee77f`)

**Witnessed:** `tests/hook-runner.test.ts:418` shipped a single "real bash -c \"exit 0\"" smoke test with `runHook(... timeoutSeconds: 5 ...)`. When `runHook` had a residual W1 mutation in the predecessor's stash (close handler dropped instead of calling `finalize(code, signal)`), the smoke test hung indefinitely â€” because tsx defaults `--test-timeout=0` (no timeout). Combined with `| tail -15` swallowing intermediate output, the failure mode was invisible: the test runner was idle in `ep_poll` waiting for a child that would never resolve.

**Builder-prompt rule learned (orchestrator-side mitigation):**

- When a test forks a real subprocess, ALWAYS pass an explicit `--test-timeout=<N>ms` to `npx tsx --test`. The default (`0`, no timeout) turns a `runHook`-style bug into an unbounded hang.
- Real-subprocess smoke tests should be in their own file (`tests/*-smoke.test.ts`) so they can be run separately from the deterministic-stub unit tests, and the unit-test path doesn't get held hostage by smoke flakes.
- Pre-commit hooks running `npm test` with no per-test timeout are also vulnerable; a global `NODE_TEST_TIMEOUT_MS` env var or a wrapper that injects `--test-timeout=30000` would be defense in depth.

May also be worth pinning into `personas/builder.md`.

### 7. Conductor run-state registry doesn't survive pi-dashboard restart (`critic-my33` orphaning) â€” CLOSED 2026-05-21 (`d5583be..1f9604a`, see v0.9.x closure block above)

**Witnessed:** during slice-2 critic gate, pi-dashboard exited with status 143 (SIGTERM) and was systemd-restarted. The new dashboard came up cleanly with the idle-reaper patch intact, but **the conductor extension's in-memory run-state registry was wiped** â€” `ensemble_status` returned "No sub-agents," and `ensemble_kill critic-my33` returned `agent_id "critic-my33" not found`. The on-disk artifacts (`runs/critic-my33/record.json`, `transcript.jsonl`, `session/seeded.jsonl`) all survived intact, but the runtime had no handle on the run, so `ensemble_send` was unable to resume the prior critic's session even though `pi --session` could have done so given the session file path.

**Question for design:**

- Should the conductor extension reconcile `runs/<id>/record.json` files into the in-memory registry on extension load? Today's only path is via `ensemble_spawn` populating a new entry. A reconcile step on startup would surface orphaned `running` records as `crashed` (mirroring v0.9's pre-shutdown reconcile of orphaned-running records, but on the *startup* side instead of shutdown).
- Once reconciled, `ensemble_send` could work on the orphaned record (tagged `crashed` post-reconcile) since the spec says it works on terminal states. Effective behavior: "resume my dead critic from where it left off, preserving session."
- Risk: a reconciled `crashed` record's pi process is gone; `ensemble_send` would have to spawn a fresh pi with `--session <path>` to actually resume. The spec says it does this already on terminal-state agents â€” verify the wiring still works after reconcile.

**Workaround used today:** spawn a fresh sub-agent and inline the prior agent's findings as evidence in the new brief. Works for verification gates where the prior work is documented in the transcript, but loses the model's loaded context. Not a substitute for proper resume.

**Adjacent v0.9 connection:** v0.9 already implemented "pre-shutdown reconcile of orphaned `running` records" (PRD #12 closure). The complementary case â€” *post-startup* reconcile after a hard restart â€” wasn't part of v0.9's scope. Worth treating as a v0.9.x patch rather than v0.10.x backlog.

### 8. WDD revert hazards (`builder-7e93` slice-1 finding) â€” OPEN, deferred to milestone-close cleanup commit (post-v0.9.x)

### 9. `child.killed` / `child.exitCode` are unreliable liveness signals (`builder-7e93` slice-1 finding) â€” OPEN, deferred to milestone-close cleanup commit (post-v0.9.x)

**Witnessed:** during v0.9.x slice-1 W1 test (real-subprocess liveness probe), the initial test asserted on `child.killed` and `child.exitCode` to confirm the child survived the probe. Both are unsafe in this scenario:

- `child.killed` is set only when `subprocess.kill()` is called on the local Node handle â€” it stays `false` even if the child died from an external `SIGKILL` or its own exit.
- `child.exitCode` stays `null` after a signal-based termination; only graceful exits populate it.

Correct liveness signal: `child.signalCode` (set when the child died from a signal) AND attaching an `exit` listener that sets a flag.

Same hazard family as commit `85ee77f` (`docs(builder): warn against --test-timeout=0 with real-subprocess tests`). Both are about real-subprocess test discipline.

**Builder-prompt rule learned:**

- For any test that forks a real subprocess and asserts on liveness or exit state: prefer `child.signalCode` + an `exit`-listener flag over `child.killed` / `child.exitCode`.
- Combine with the existing rule from `personas/builder.md` (commit `85ee77f`): explicit `--test-timeout=<ms>` to bound real-subprocess hangs.
- Worth folding into `personas/builder.md`'s real-subprocess section as a follow-up at milestone close.

### 10. WDD witness brittleness via tsx loose-transpile (`critic-2kx5` slice-1 finding) â€” OPEN, deferred to milestone-close cleanup commit (post-v0.9.x)

**Witnessed:** during v0.9.x slice-1 critic gate (W3 reproduction), the named mutation `record.pid === undefined` â†’ `record.pid === null` produced a TypeScript compile error (`number | undefined` not assignable to `number`). The strict path would reject the mutation as not-buildable. But `tsx` runs tests with loose transpile, so the test still executed and the assertion still fired correctly.

**Implication:** the witness IS valid â€” the mutation DID kill the test â€” but the catch is partly a TypeScript-level rejection rather than a pure runtime-behavior failure. A more subtle mutation that compiles cleanly under strict TS (e.g. `record.pid === 0` or `record.pid > 0`) might evade the witness.

**Worth pinning into `docs/wdd.md`:**

- WDD witnesses should ideally name mutations that compile cleanly under strict TypeScript so they exercise the runtime check rather than relying on tsx's loose-transpile fall-through. When a witness's mutation is type-rejected by `tsc --noEmit`, the witness is still meaningful but should be paired with a sibling mutation that compiles â€” or the design should explicitly note the witness is a TS-level catch.
- Not blocking for slice-1 since the witness DOES kill the test. Filed for posterity.

**Question for design:** is *zombie detection* (the underlying pi process exited or vanished â€” distinct from "pi is alive but silent") a separate failure mode that warrants a forced-kill path independent of `kill_on_stall`? Plausible primitives:

- Periodic `kill(pid, 0)` liveness check on `Run.proc.pid`; if ESRCH, transition to `crashed` immediately without consulting `kill_on_stall`.
- Surface pi subprocess exit code on the `Run` record so the close handler can flip terminal status without waiting for a final-event message.
- A separate `forceKillOnZombie: true` default that's safe to ship ON because zombies are unrecoverable.

**Why this is its own line item:** v0.10's threshold model assumes a live process going silent. A dead process going silent is a different failure mode and arguably shouldn't need an opt-in to clean up.

### 2. `kill_on_stall` default for write-capable persona spawns

**Witnessed:** same incident. Default `kill_on_stall: false` is conservative for read-only personas (oracle/inspector/etc. â€” losing 12 minutes of advisory time is annoying but cheap). For write-capable personas (builder/simplifier) the same default is harsher: a zombie holds the `m_w_c=1` write slot, blocking the entire chain.

**Question for design:** should the cascade default for `kill_on_stall` be persona-class-aware? Sketch:

- Read-only personas: default `kill_on_stall: false` (current behavior â€” preserve transcripts for post-mortem).
- Write-capable personas: default `kill_on_stall: true` (the chain can't progress with a zombie holding the slot; cheap to respawn).

Would interact with the v0.11 cascade pattern landing in slice 4 (per-call > project > user > frontmatter > built-in default). The "built-in default" layer becomes persona-class-conditional.

**Counterargument:** a builder mid-commit that gets hard-killed leaves a half-written change. Mitigation: write-capable personas always commit before exiting per `personas/builder.md` (or hook quality gates land first via v0.11 and reject).

### Conventional Commits steering hook misfires on heredoc commit messages

`builder-mqye` (Step 1 lifecycle fix, commit `1c72856`) hit the user's
personal `~/.pi/` Conventional Commits steering hook regex while
attempting a multi-line message via `git commit -m "$(cat <<'EOF' ...`.
The regex inspects the literal `-m` arg and sees `$(cat...`, which
doesn't match the allowlist (`feat:` / `fix:` / etc.) â€” so the
attempt is rejected even though the actual subject is well-formed.

Workaround: write the message to a tempfile and use `git commit -F
/tmp/msg`. The `-F` path bypasses the `-m` regex entirely. This is a
user-environment papercut, not a pi-conductor surface â€” the hook
lives outside the repo. But the friction is reproducible across
sub-agents.

**Suggested mitigation (in pi-conductor scope):** one-line nudge in
`personas/builder.md` and `personas/scribe.md` â€” *"For multi-line
commit messages, prefer `git commit -F /tmp/msg` over heredoc-style
`git commit -m \"$(cat <<EOF ... EOF)\"` â€” some user-side steering
hooks reject the heredoc form."* Pure prompt nudge, no code change.

### Mutation testing substitutes for live smoke when smoke is structurally impossible

`critic-8bv5` (Step 1 lifecycle fix gate) and `verifier-zfqv`
(end-of-slice verification) both used mutation testing on
`src/shutdown.ts` as their novel verification: drop the reload guard,
flip SIGTERMâ†’SIGKILL, etc. Each mutation reproduces a unit-test
failure, proving the tests pin the new behavior independently.

This was the strongest available runtime check because live

**Suggested addition to `personas/critic.md` and `personas/verifier.md`:**
