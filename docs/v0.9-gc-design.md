## 1. Goals & non-goals

### Goals

- **Reclaim disk.** 11 GB on this machine after 5 days of dogfood; transcripts are 98% of bytes. Stop the monotonic growth without losing the metadata that makes `/conductor history` useful.
- **Preserve resumability.** `ensemble_send` to a recently-finished agent must keep working. The pi `session/<...>.jsonl` is the only file required for resume (verified via `validateSendable` reading `run.sessionPath`); GC must distinguish "drop transcript" (safe) from "drop session/" (breaks resume).
- **Reconcile stale `running` records.** 3 orphans observed today; will keep accruing as users hard-kill the parent. Crash-tolerant lifecycle requires explicit reconciliation, not just GC.
- **Never break an in-flight run.** Fire-and-forget `appendFile` on `transcript.jsonl` (`runs.ts:770`) means GC unlinking an active transcript silently corrupts state. Active-run safety is a hard invariant.
- **Make policy observable.** `/conductor doctor` should show current usage + what the next GC pass would do, so users can tune defaults before they reclaim anything.

### Non-goals

- **Pi's own session cache** (`~/.pi/agent/sessions/`). Pi owns its lifecycle for that dir. Conductor only manages `~/.pi/agent/conductor/runs/`.
- **Within-run transcript compaction.** Item 5 (envelope compaction) handled the parent-context analog; reclaiming bytes inside a single transcript.jsonl (per-line summarization) is a separate problem and not in scope.
- **Cross-machine sync / cloud archive.** Local-disk only. If a user wants to keep history forever, they back up the dir themselves.
- **History UI changes.** `/conductor history` keeps reading what's there; it just sees fewer / smaller entries after GC.
- **Allocator fix for id reuse** (inspector surprise #6). Out of scope; tracked as a follow-up.

## 2. Policy decisions

### D1. Retention model — **Dual (age TTL + size cap), per-persona overrides for transcript size**

**Decision.** Two independent thresholds; whichever fires first triggers reclaim.
- **Age TTL** caps how long a record can live, per status tier.
- **Size cap** triggers when total `runs/` disk usage exceeds a budget. Eviction order: largest-bytes-first, oldest-finishedAt-first as tiebreak. Per-persona size overrides apply *first* (one designer transcript >500 MB triggers immediately even if total budget isn't exhausted).

**Alt considered — pure age TTL.** Rejected: 30-day-keep-everything would be ~60 GB on this dogfood; doesn't address the "one designer run = 8.5% of disk" problem.

**Alt considered — pure size cap (LRU).** Rejected: "delete the oldest until under budget" loses small-but-recent runs unnecessarily and gives the user no temporal contract. A size-only policy has surprising eviction behavior on bursty workloads.

**Alt considered — per-status-tier TTL only (failed kept longer).** Folded *into* the dual policy; not a standalone alternative. Failed/killed/timeout get the same age TTL as completed in v0.9 (simpler default; `pinned` is the escape hatch for "keep this failure for post-mortem").

### D2. Default values — **30d / 60d / 5GB / 100MB-per-transcript**

| Knob | Default | Rationale |
|---|---|---|
| `gc.completedTtlDays` | **30** | Two sprints. Long enough for ensemble_send post-mortem, short enough that one bad week doesn't accumulate. |
| `gc.failedTtlDays` | **60** | Failed runs are diagnostic gold; keep ~2× longer. |
| `gc.totalSizeBudgetBytes` | **5 GB** | Roughly half a typical SSD's free-space slop budget. Below this, age TTL is the only signal. Above, size eviction kicks in. |
| `gc.transcriptSizeCapBytes` | **100 MB** | Per-transcript cap. Designer/planner top-out at 200–850 MB; ~100 MB cap triggers cold-archive on the outliers without touching ordinary runs. (Inspector inventory: ≥10 transcripts on this dogfood machine exceed 168 MB — see `docs/v0.9-gc-inspector-map.md:30-41` for the top-10 table; the original "~6 transcripts > 100 MB" estimate in earlier design drafts was corrected per oracle review A4.) |
| `gc.orphanReconcileAfterHours` | **24** | A `running` record older than a day with no live process is orphaned. |
| `gc.autoOnSessionStart` | **true** | Idle, debounced, dry-run-logged before reclaiming. |
| `gc.perPersonaTtlDays` | `{}` | Optional per-persona override map; empty by default. Users can lower designer to 14d if they want. |

**Alt considered — make defaults more aggressive (7d / 1GB).** Rejected: dogfood values that delete history before the user has even noticed there's a tradeoff to tune are worse than visible bloat. Conservative defaults + a doctor surface that shows "you'd reclaim X if you ran GC now" lets users opt into aggression.

**Alt considered — match pi-coding-agent's session retention.** Pi has no published default (it's user-config); cribbing wouldn't add signal.

### D3. Reclaim strategy — **Tier ladder: cold-archive first, then full-delete**

**Decision.** Two-tier reclaim, applied in order:

| Tier | Action | Reclaims | When triggered |
|---|---|---|---|
| **Cold-archive** | `unlink transcript.jsonl`. Keep `record.json`, `final.md`, `session/`, `.pinned` if present. Touch `.archived` sidecar with timestamp. | ~98% of bytes per run | Transcript size > `transcriptSizeCapBytes` **OR** run age > 50% of TTL with size > 10× median |
| **Full-delete** | `rm -rf <runDir>`. Frees the id for reuse. | 100% | Run age > full TTL **AND** already cold-archived **AND** unpinned |

**Alt considered — Option A (full delete only).** Rejected: kills resumability and history with one knob. The inspector inventory shows record + final = ~0.02% of transcript bytes; cold-archive is nearly free metadata for ~100% of the user-facing reclaim.

**Alt considered — Option C (gzip transcript in place).** Rejected for v0.9. Pros: transcripts are highly compressible (~70% reclaim). Cons: requires read-side decompression in `/conductor history` (currently doesn't read transcript.jsonl) and breaks the "transcript path is what I tell you" contract — users who `tail` the file post-GC get a binary blob. Folded as v1.0 candidate ("hot-archive" tier between active and cold-archive).

**Cold-archive preserves resumability** because `ensemble_send` only needs `session/<...>.jsonl` (verified at `runs.ts:842-878`). After cold-archive a run is still resumable via `ensemble_send` AND still listable in `/conductor history` — only the verbatim event-stream replay is gone.

### D4. Pinning — **Sidecar `.pinned` file**

**Decision.** Pinning is a sidecar `<runDir>/.pinned` (empty file, presence = pinned). No schema bump on `record.json`.

**API.** Two slash subcommands:
- `/conductor pin <agent_id>` — `touch <runDir>/.pinned`
- `/conductor unpin <agent_id>` — `unlink <runDir>/.pinned`

**Pinned semantics.** Skip both cold-archive and full-delete. Pinned records still count against disk budget for *reporting* (doctor) but not for *eviction*. If user pins more than the budget, GC logs "pinned-exceeds-budget" but does not reclaim.

**Alt considered — schema bump (`pinned: boolean` on RunRecord).** Rejected per inspector surprise #5: forward-compat is fine, backward-compat is ⚠ (old conductor silently drops the field on next `writeRecord`). Sidecar file survives any rollback to a v0.8 conductor.

**Alt considered — pin via persona / status pattern.** Rejected: too magic; users can pin explicitly once they see the doctor's "you'd reclaim X" preview.

### D5. Stale `running` records — **Reconcile to `killed` with marker, never delete in same pass**

**Decision.**
- A run with `status === "running"` AND `isStale(run)` AND no live `Run` in registry is reconciled to `status: "killed"` with `errorMessage: "orphaned: process gone (reconciled by GC)"` and `finishedAt: <reconcile-time>`.
- **`isStale`**: `mtime(transcript.jsonl) < now - gc.orphanReconcileAfterHours*1h` (transcript mtime is the best proxy since `lastEventAt` is in-memory only, see inspector §7).
- **Never delete in same pass.** Reconciliation flips status; standard GC math then runs against the now-`killed` record on the *next* pass. Two-pass design avoids combining "this looks dead" with "delete this" in one heuristic.

**Implementation.** Reconciliation runs *before* the inventory→policy→reclaim pipeline. Output is a (small) batch of `writeRecord` calls flipping status. Logged to doctor under "Reconciled this session."

**Alt considered — flip to `failed` instead of `killed`.** Rejected: status semantics already pin `killed = SIGTERM/SIGKILL`, `failed = exitCode != 0`. Orphans match `killed` more cleanly (the host *did* terminate them, just abnormally). The `errorMessage` distinguishes orphan-killed from user-killed.

**Alt considered — auto-delete orphans.** Rejected: user observability matters more than fast reclaim. Reconcile + report; the user opts into delete via the standard TTL on the now-killed record.

### D6. Trigger — **Both: auto on `session_start` (debounced), manual `/conductor gc`**

**Decision.**
- **Auto on `session_start`.** Once per process start; runs in a `setImmediate` so it doesn't block UI. Debounce key: a `<runsRoot>/.last-gc` file with epoch ms. If `now - last-gc < 6h`, skip. Auto runs are always **dry-run-logged** with action plan (counts + bytes) before reclaiming; `gc.autoOnSessionStart: false` disables.
- **Manual `/conductor gc [--dry-run] [--force] [--persona=<name>]`.** Subcommands:
  - default: same dual policy as auto, applies reclaim.
  - `--dry-run`: prints the action plan, does not reclaim.
  - `--force`: ignore the 6h debounce.
  - `--persona=<name>`: scope to that persona only.
- **Reconcile-on-start always runs** (regardless of `autoOnSessionStart` setting). It's cheap and prevents orphan accumulation in low-disk-pressure setups.

**Alt considered — `session_shutdown` instead of `session_start`.** Rejected: reload-vs-quit ambiguity (`shutdown.ts:73` has the discriminator) plus reclaim-during-shutdown is worse UX (user pressed `/quit` and now waits on rm).

**Alt considered — periodic timer mid-session.** Rejected: idle-time reclaim is fine for a daemon, but conductor is interactive. Per-session-start + manual is enough; periodic adds complexity.

### D7. ensemble_send safety — **Cold-archive preserves resume; full-delete breaks it (by design)**

**Decision.**
- Cold-archive **does not** delete `session/<...>.jsonl`. `ensemble_send` keeps working post-archive.
- Full-delete **does** kill resumability — that's the whole point. Gate full-delete on `age > completedTtlDays` (default 30); 30-day-resumable is a long enough contract.
- A finished run still in the **in-memory registry** is never reclaimed (cold-archive or delete) regardless of age. Same-session sends always work. (Registry only persists for the current session's process; cross-process resume is the disk-only path that the TTL contract covers.)
- `/conductor gc --dry-run` output explicitly flags "X runs will lose ensemble_send-resume" so the user can pin first.

**Alt considered — never full-delete; cold-archive is terminal.** Rejected: 1.8 MB/228 = 7.9 KB record.json average × ~46 runs/day = 110 KB/day of records that never go away. Tolerable for years but compounds; full-delete after 30 days frees the metadata too.

**Alt considered — keep session/ forever, only cold-archive transcripts.** Same problem as above; pi sessions are 0.1 GB / 218 today, but they grow. 30-day delete bounds it.

### D8. Active-run safety — **Two-gate invariant**

**Decision.** A run is reclaim-eligible **only** when both:
1. `record.status` is in the terminal set (`completed`, `failed`, `killed`, `timeout`) — never `running`, `paused`, `queued`.
2. **No in-memory `Run` for that id exists in the registry**, OR the in-memory `Run.status` is also terminal.

If either gate fails, the run is skipped with reason `"active"` in the dry-run plan.

**Why two gates.** Disk state lags in-memory state (writes are best-effort, see inspector §2 note). A run mid-reconciliation could have on-disk `running` and in-memory `killed`, or vice versa. Skipping unless *both* agree is the safe direction; a healthy run gets reclaimed one cycle later, an unhealthy run never gets corrupted.

**Implementation.** GC engine takes `RunRegistry` as a constructor arg. Inventory pass attaches `inMemory: Run | undefined`. Policy phase filters out anything failing either gate.

**Alt considered — file lock on transcript.jsonl.** Rejected: cross-platform pain, doesn't survive process death (so doesn't fix the orphan case anyway).

## 3. Architecture

```
┌─────────────────────┐
│ Inventory           │  read every <runDir>/record.json + stat tree
│ (pure: dir → entry) │  → InventoryEntry { id, status, finishedAt, sizeBytes,
└────────┬────────────┘                       persona, sessionPathPresent,
         ▼                                    pinned, archived, inMemory }
┌─────────────────────┐
│ Reconcile orphans   │  flip stale `running` → `killed`
│ (writes record.json)│  produces ReconcileResult { reconciledIds[] }
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ Policy engine       │  pure: (Inventory[], Config, NowMs) → ReclaimPlan
│ (pure)              │  ReclaimPlan { entries: [{ id, action, reason }] }
└────────┬────────────┘    action ∈ "keep" | "cold-archive" | "delete"
         ▼
┌─────────────────────┐
│ Reclaim executor    │  applies plan
│ (writes; effectful) │  - cold-archive: unlink transcript.jsonl, touch .archived
└─────────────────────┘  - delete:       rm -rf <runDir>
```

The split lets us test the policy engine with table-driven fixtures and the executor with `mkdtempSync` fake trees. The two effectful boundaries (`reconcileOrphans`, `executePlan`) are thin and dispatch-only.

### Module shape

- `src/gc/inventory.ts` — `walkInventory(runsRoot, registry, now)` → `InventoryEntry[]`. Pure aside from `readdir`/`stat`.
- `src/gc/policy.ts` — `planReclaim(inventory, config, now)` → `ReclaimPlan`. Fully pure.
- `src/gc/reconcile.ts` — `reconcileOrphans(inventory, config, now)` → `{ reconciledIds: string[] }`. Effectful (writeRecord).
- `src/gc/executor.ts` — `executePlan(plan, runsRoot)` → `{ reclaimedBytes, reclaimedIds }`. Effectful.
- `src/gc/index.ts` — `runGc(opts)` orchestrator. Used by both auto trigger and slash command.
- `src/gc/pinning.ts` — `pinRun(id) / unpinRun(id) / isPinned(runDir)`. Sidecar primitives.

All under `src/gc/` to keep blast radius small and importable as one namespace.

### Config additions to `ConductorConfig`

```ts
gc: {
  completedTtlDays: number;          // default 30
  failedTtlDays: number;              // default 60
  totalSizeBudgetBytes: number;       // default 5 * 1024**3
  transcriptSizeCapBytes: number;     // default 100 * 1024**2
  orphanReconcileAfterHours: number;  // default 24
  autoOnSessionStart: boolean;        // default true
  perPersonaTtlDays: Record<string, number>; // default {}
};
```

Single nested object so existing `validateConfig` only adds one `gc?:` branch.

## 5. Risks & open questions

### Risks

1. **Cold-archive vs ensemble_send race.** A finished run is cold-archived on auto-GC; user immediately runs `ensemble_send` against it. The session/ file is intact so the resume succeeds. Pi will then re-create a new transcript.jsonl on the resumed run via the close-handler write path. Risk: the new transcript is a *partial* record (from resume forward only). Mitigation: explicit. The "archived" state means "transcript was reclaimed; future events will re-populate." Document. Test: cold-archive a run, ensemble_send, assert new transcript.jsonl exists with only post-resume events.

2. **GC concurrent with active-run writes (inspector surprise #3).** If GC selects a run for reclaim while an event is in flight from `processLine`, the unlink wins and the appendFile lands on a deleted inode (Linux: file disappears at writer close). Mitigation: D8's two-gate invariant — terminal-on-disk **and** registry-not-running. Test: registry hands GC a list with a fake `Run.proc` set; assert that entry is in `keep` plan even though disk says terminal.

3. **Allocator id reuse after delete (inspector surprise #6).** Vault notes citing `inspector-2h68` could collide with a future spawn picking the same id. Mitigation: out of scope for v0.9 (acceptable risk; collision space is `persona-<4hex>` = 65 K per persona). Tracked as v1.0 follow-up: longer suffix or recently-deleted-id memory.

4. **`pushCompletionNotification` post-archive.** A foreground run's completion notification embeds the transcript path. If GC archives the transcript before the user reads the card, the path resolves to a missing file. Mitigation: notification cards are advisory; nothing breaks. But: doctor/history should explicitly mark `archived` so users know not to try `tail`-ing the path. Slice 8 covers this.

5. **`6h auto-debounce` makes test cycles painful.** Mitigation: `--force` flag bypasses; tests inject their own clock.

6. **First GC run on a 11 GB tree could take seconds.** Mitigation: `walkInventory` reads only `record.json` (1.8 MB total) and `stat`s `transcript.jsonl` for size — no full reads. Estimated < 500 ms for 228 runs.

### Open questions for the user

1. **Should builder runs that committed > 0 lines be auto-pinned?** Argument: those are evidence trail for shipped features. Argument against: noise; user can pin manually. **Recommend: no, but doctor flags "high-value runs unpinned" as an advisory.** This keeps the policy mechanical and the user explicit.

2. **`/conductor gc --dry-run` should it be the default?** Conservative argument: yes, force users to opt in to reclaim. Aggressive argument: no, the auto-on-session-start already reclaims silently (with logged plan), so manual `gc` matching that behavior is consistent. **Recommend: opt-in to dry-run via `--dry-run`; manual `gc` reclaims by default.** Matches `git gc`.

3. **Per-persona overrides — should designer/planner have `transcriptSizeCapBytes` set lower (e.g. 50 MB) by default?** Argument for: those are the only personas regularly exceeding 100 MB. Argument against: default-bias; some users like keeping every designer trace. **Recommend: leave per-persona overrides empty in defaults; document the recipe in `docs/v0.9-gc-design.md` for users who want it.** Inspector-driven defaults already handle the long tail.

4. **Should pinning prevent cold-archive too, or only full-delete?** **Recommend: prevent both.** A pinned run keeps its transcript. The 99% reclaim story is pinning's tradeoff to opt out of.

5. **Should `record.json` carry a `gc.archivedAt` timestamp post-archive?** Helps "when was this archived" forensics. Sidecar file already has mtime; record.json gets a small schema addition. **Recommend: sidecar is sufficient (`<runDir>/.archived` empty file with mtime as timestamp).** No schema bump.

## 6. Test strategy

### Unit (pure)

- `gc-inventory.test.ts` — table-driven against fixtures of run dirs with various states (terminal, running, archived, pinned, missing-fields). Asserts the entry shape, size math, age math.
- `gc-policy.test.ts` — table-driven (config × inventory × now) → expected plan. Cases:
  - Empty inventory → empty plan.
  - One terminal run < TTL → keep.
  - One terminal run > TTL, unpinned → delete (or cold-archive if not yet).
  - One huge transcript < TTL → cold-archive (size cap).
  - One pinned run > TTL > size budget → keep.
  - One running run > TTL → keep (active-run gate).
  - Total size > budget → largest-bytes-first cold-archive until under budget.
  - Mixed: prove ordering of cold-archive-then-delete tier transitions.
- `gc-reconcile.test.ts` — stale `running` reconciliation. Mutation-tested: removing the `< now - threshold` check fails the "fresh running stays running" case.

### Integration (mkdtempSync fixtures)

- `gc-executor.test.ts` — full disk roundtrip. Build a fake runs root with mkdtempSync, run executor, assert files unlinked / dirs removed / sidecars created.
- `gc-orchestrator.test.ts` — auto-GC-on-session-start with debounce. Inject `now` and `last-gc` mtime; assert skip vs run.
- `gc-pinning.test.ts` — pin/unpin/isPinned roundtrip; pinned run survives a TTL-expired GC pass.

### Smoke (manual, post-finalizer)

- `/conductor gc --dry-run` on the dogfood machine — assert plan reports ~10 GB reclaimable (mostly cold-archive of designer/planner) and ~3 orphans reconciled.
- `/conductor pin <id>` then `/conductor gc` — assert pinned run untouched.
- After GC, `/conductor history` lists archived runs with marker; `ensemble_send` to a recently-archived run still resumes.

### Mutation (per critic playbook)

- Slice 1 policy: drop the active-run gate → assert "running run gets reclaimed" test fails.
- Slice 2 reconcile: drop the staleness threshold → "fresh running gets killed" fails.
- Slice 3 executor: swap unlink for `rm -rf` in cold-archive path → "session/ survives cold-archive" fails.

## 7. Recommended chain shape

**Critic on every slice is overkill.** The pure slices (1 inventory/policy, 3 executor) carry the most blast risk and **should** get a critic; the schema/sidecar/orchestrator slices are mechanical and can ship with builder + verifier only.

Recommended:

| Slice | Builder | Critic | Notes |
|---|---|---|---|
| 1 inventory + policy | yes | **yes** | Policy logic is the design's correctness core. Critic must table-test it. |
| 2 orphan reconcile | yes | builder-only OK | Single-purpose, easy to mutation-test inline. |
| 3 executor | yes | **yes** | Effectful boundary; race with active runs is the highest disaster risk. Critic should run a manual smoke against a temp tree. |
| 4 pinning | yes | builder-only OK | Two-line sidecar functions; verifier catches drift. |
| 5 orchestrator + auto trigger | yes | **yes** | Lifecycle wiring; debounce semantics matter. Critic gates the `last-gc` and `setImmediate` integration. |
| 6 manual `gc` slash | yes | builder-only OK | Wiring; the engine is already tested. |
| 7 doctor surface | yes | builder-only OK | Output formatting; existing doctor tests cover the shape. |
| 8 history display | yes | builder-only OK | Annotation; trivial. |
| 9 backlog + PRD | scribe | builder-only OK | Docs. |

**Verifier** runs as the whole-task gate after slice 9. **Finalizer** closes.

Cost ceiling estimate: ~$45–55 for the chain, dominated by slices 1 + 3 (each with builder + critic ≈ $8–10) and slice 5 (orchestrator integration ≈ $7). Below the v0.8.3 chain ($62) because the surface is smaller and only 3 of 9 slices need critic gating.

**Decisions taken (one phrase each):**
1. **D1 retention:** dual policy (age TTL + size cap), per-persona transcript-size overrides.
2. **D2 defaults:** 30d completed / 60d failed / 5 GB total / 100 MB transcript / 24h orphan.
3. **D3 reclaim:** tier ladder — cold-archive (drop transcript, keep metadata + session/) then full-delete.
4. **D4 pinning:** sidecar `<runDir>/.pinned` file; `/conductor pin/unpin`.
5. **D5 orphans:** reconcile stale `running` to `killed` with marker; never delete in same pass.
6. **D6 trigger:** auto on `session_start` (debounced 6h) + manual `/conductor gc [--dry-run|--force|--persona]`.
7. **D7 ensemble_send safety:** cold-archive preserves session/ → resume works; full-delete only after TTL.
8. **D8 active-run safety:** two-gate invariant — terminal-on-disk AND registry-not-running.

Slice count: **9** (1 inventory+policy, 2 reconcile, 3 executor, 4 pinning, 5 orchestrator+auto, 6 manual slash, 7 doctor, 8 history, 9 docs).

Recommended chain: **critic on slices 1, 3, 5; builder-only on 2, 4, 6, 7, 8; scribe on 9; verifier + finalizer at the end.** Three critic gates instead of nine because only the policy engine, the effectful executor, and the lifecycle orchestrator carry the architectural risk that warrants a fresh-eyes pass.
