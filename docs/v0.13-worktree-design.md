# Design: v0.13 — Worktree-per-persona

**Status: DRAFT.** A future closer flips this banner once v0.13 ships.

**Revision log:**
- 2026-05-29 — initial designer output.
- 2026-05-29 — oracle gate **ADJUST** applied (oracle-jf2i). Three required changes: (A) non-git fallback in `resolveWorktreeSpec` — returns `null`, spawn degrades gracefully; (B) `ensureWorktreeGitignore` moved from `resolveWorktreeSpec` to `createWorktree`; (C) `worktreePath` cleared only on successful `removeWorktree` in `finalize`. All applied inline.

When `worktree: true` is set in persona frontmatter, the sub-agent runs
in an isolated git worktree instead of sharing the parent's working tree.
The primary beneficiary is `builder` in a chain with a second `simplifier`
or a parallel `critic` that also writes — today the `m_w_c=1` cap prevents
concurrent write-capable spawns entirely; worktree isolation allows raising
that cap safely for worktree-enabled personas.

The `worktree` field is already parsed on `Persona` (v0.1 forward-compat,
always `false` in production). This design activates it.

---

## Approach

### Core idea

At `spawnRun` time, when `persona.worktree === true`:

1. Detect whether `opts.cwd` is inside a Brazil workspace by walking up
   for a `.brazil` marker.
2. Resolve the git root for the per-package repo (`git rev-parse
   --show-toplevel`).
3. Place the worktree **inside the workspace root** at
   `<gitRoot>/.worktrees/conductor-wt/<run-id>/`. This satisfies the
   Brazil `find_workspace_directory_up` constraint (proven by direct test
   today) and the AIVirtuoso P0 lesson (same-filesystem-volume colocation).
4. Create the worktree: `git worktree add <path> -b conductor-wt/<run-id>
   HEAD` from `<gitRoot>`.
5. Record `run.worktreePath` and `run.worktreeBranch` for cleanup.
6. Override `SpawnOptions.cwd` to the worktree path when passing to
   `runPiSubprocess`.
7. On terminal (inside `finalize`), remove the worktree and branch.
8. On GC `delete`, check for orphaned worktrees and clean them up.

### What does NOT change

- The `m_w_c=1` cap. Raising it is a v0.13 follow-on, not a precondition.
  Worktree isolation is a prerequisite for safely raising the cap; v0.13
  ships the isolation, the cap relaxation is a separate config option
  unlocked in v0.14.
- `sendToRun` / `ensemble_send` resume behaviour. Resume always runs in
  the same worktree path stored on the Run; if the worktree was cleaned
  up before resume, the spawn path re-creates it.
- Merge-back. V0.13 does NOT auto-merge. The builder commits to the
  worktree branch; the worktree branch survives until the worktree is
  removed. Manual merge by the conductor or user is the v0.13 model.
  Auto-merge is a v0.14 item.

### New module: `src/worktree.ts`

```typescript
export interface WorktreeSpec {
  gitRoot: string;      // absolute git root of the package
  worktreePath: string; // <gitRoot>/.worktrees/conductor-wt/<run-id>
  branch: string;       // "conductor-wt/<run-id>"
}

/** Resolve where to create the worktree and what branch to use.
 * Makes one I/O call (`git rev-parse --show-toplevel`) to find the git root.
 * Returns `null` when cwd is not inside a git repo (bare clone, no .git, /tmp) —
 * callers must fall back to non-worktree spawn in that case.
 * Does NOT mutate .gitignore (that is `ensureWorktreeGitignore`, called from `createWorktree`). */
export function resolveWorktreeSpec(cwd: string, runId: string): WorktreeSpec | null

/** Create the worktree. Calls `ensureWorktreeGitignore` first (idempotent). Throws on failure. */
export function createWorktree(spec: WorktreeSpec, opts?: { skipGitignore?: boolean }): void

/** Idempotent: appends `.worktrees/` to `<gitRoot>/.gitignore` if not already present. */
export function ensureWorktreeGitignore(gitRoot: string): void

/** Remove worktree and branch. Best-effort (swallows ENOENT). */
export function removeWorktree(spec: WorktreeSpec): void

/** Given a run record, infer the WorktreeSpec for orphan cleanup. Pure. */
export function worktreeSpecFromRun(run: RunRecord): WorktreeSpec | undefined

/** Probe for the Brazil workspace root above cwd. Returns null if not in a Brazil workspace. */
export function detectBrazilWorkspaceRoot(cwd: string): string | null
```

## Data flow

### Spawn path

```
ensemble_spawn (tools.ts)
  → SpawnQueue.enqueueOrSpawn
      → spawnRun(opts: SpawnOptions)
          if opts.worktree:
            spec = resolveWorktreeSpec(opts.cwd, run.id)
            if spec === null:                            ← non-git cwd fallback
              logWarn("worktree: true but cwd is not in a git repo; falling back to shared cwd")
              effectiveCwd = opts.cwd
            else:
              createWorktree(spec)             ← git worktree add + gitignore bootstrap
              run.worktreePath = spec.worktreePath
              run.worktreeBranch = spec.branch
              effectiveCwd = spec.worktreePath
          else:
            effectiveCwd = opts.cwd
          writeRecord(run)                   ← persists worktreePath/Branch
          runPiSubprocess(run, piArgs, {cwd: effectiveCwd, ...})
```

### Terminal / cleanup path

```
finalize(terminal, exitCode) inside runPiSubprocess:
  ... existing hook logic ...
  applyCloseHandlerTerminal(run, terminal, exitCode)
  if run.worktreePath:
    try:
      removeWorktree({gitRoot, worktreePath, branch})
      run.worktreePath = undefined  ← cleared only on success; GC handles failures
      run.worktreeBranch = undefined
    catch (e):
      logWarn("worktree removal failed; GC will clean up", e)
      // worktreePath left set so GC delete can find and remove it
  writeRecord(run)
  onComplete?.(run)
  donePromiseResolve(run)
```

The `removeWorktree` call is best-effort inside `finalize`. If it throws,
log a warning but do NOT abort the finalize path — the run must reach a
terminal status regardless.

### GC orphan path

When `executeReclaim` runs a `delete` action on a run that has
`record.worktreePath` set and the path still exists:

```
fullDelete(runDir):
  if record.worktreePath && existsSync(record.worktreePath):
    removeWorktree({gitRoot: resolveGitRoot(record.worktreePath),
                    worktreePath: record.worktreePath,
                    branch: record.worktreeBranch})
  rm(runDir, { recursive: true })
```

`cold-archive` does NOT clean up the worktree (the run is still resumable
after cold-archive). Worktree cleanup on cold-archive would make resume
impossible. The worktree will be cleaned up when the run eventually hits
`delete`.

**In steady state, `finalize` already removes the worktree before `cold-archive`
ever runs.** Terminal runs with successful `removeWorktree` have `worktreePath`
cleared in `RunRecord`. Worktrees only survive into cold-archive when `finalize`'s
`removeWorktree` call failed — an edge case, not the common path. The cold-archive
path doesn't need to remove worktrees as a space concern; GC `delete` handles orphans.

### Resume path (`ensemble_send` → `sendToRun`)

If `run.worktreePath` is set and `existsSync(run.worktreePath)` is true:
use `run.worktreePath` as cwd for the resumed subprocess.

If `run.worktreePath` is set but the path is **missing** (worktree was
removed or a crash left the run in a weird state): recreate the worktree.
```
if run.worktreePath && !existsSync(run.worktreePath) && run.worktreeBranch:
  // Recreate: new worktree at same path, same branch name
  // (branch may still exist from the prior run — remove it first if so)
  createWorktree({gitRoot, worktreePath, branch})
```

## Placement strategy

### Non-Brazil repos (pi-conductor itself)

```
cwd = /local/home/samfp/scratch/pi-conductor
gitRoot = /local/home/samfp/scratch/pi-conductor  (same, single repo)
worktreePath = /local/home/samfp/scratch/pi-conductor/.worktrees/conductor-wt/<run-id>/
```

`.worktrees/` is gitignored automatically by the setup step (see below).

### Brazil repos

```
cwd = /workplace/samfp/Rosie/src/CSDefectResolutionExecutionService
gitRoot = /workplace/samfp/Rosie/src/CSDefectResolutionExecutionService
  (git rev-parse --show-toplevel from cwd)
brazilWorkspaceRoot = /workplace/samfp/Rosie
  (walk up from gitRoot looking for .brazil marker)
worktreePath = /workplace/samfp/Rosie/src/CSDefectResolutionExecutionService/.worktrees/conductor-wt/<run-id>/
  (inside gitRoot, which is inside workspaceRoot — Brazil happy)
```

The worktree is colocated inside the **package**'s git root (not the
workspace root). This is the same as the AGT-Superpowers convention and
avoids the cross-device link failure (same filesystem volume guaranteed
since the worktree is a child of the package dir).

### `.worktrees/` gitignore bootstrap

On first use, `createWorktree` calls `ensureWorktreeGitignore(gitRoot)` which
appends `.worktrees/` to `<gitRoot>/.gitignore` if not already present (idempotent).
`resolveWorktreeSpec` does NOT mutate `.gitignore`.
Pass `skipGitignore: true` to `createWorktree` in tests.

### Brazil multi-package caveat

Each Brazil package has its own `.git`. A builder whose task spans two
packages (e.g. modifying both `CSDefectResolutionExecutionService` and
`CSDefectResolutionCommons`) gets ONE worktree for the package whose root
is `opts.cwd`. Files in the other package are not isolated.

**v0.13 limitation:** single-package isolation only. The builder must
commit per-package or the conductor must run separate builder instances
per package. Document this in the persona's system prompt.

## Failure modes

| Error | Surfaced how | Recovery |
|---|---|---|
| `git worktree add` fails (dirty tree, branch exists, disk full) | `spawnRun` throws before subprocess starts; run flips to `failed` with error message | Retry with clean tree; GC cleans up partial state |
| `git rev-parse --show-toplevel` fails (no git repo, bare clone, /tmp) | `resolveWorktreeSpec` returns `null`; `spawnRun` logs a warning and falls back to shared cwd; run proceeds normally | Non-blocking degradation — user gets a warning, builder works without isolation |
| Worktree removal fails at `finalize` | Warning logged; run still reaches terminal status; worktreePath left set in record | GC `delete` path cleans it up |
| Crash mid-run; worktree not removed | `worktreePath` persists in record; GC `delete` detects and cleans up | GC pass |
| Resume after worktree removal | `sendToRun` detects missing path, recreates worktree | Transparent to LLM |
| Brazil workspace not detected | Worktree placed inside gitRoot (same heuristic), which is inside workspace root — correct regardless | Not an error |
| Worktree placed on different filesystem (e.g. symlinked gitRoot) | `createWorktree` will get a cross-device link error | `resolveWorktreeSpec` resolves `gitRoot` via `fs.realpathSync` to guarantee same volume |
| Two runs use same branch name | `branchExists` check in `createWorktree` throws; `spawnRun` fails | Run ID is unique; branch is `conductor-wt/<run-id>` — collision is impossible unless run IDs collide (existing guarantee) |
| `extensions.worktreeConfig` git config leak | Conductor never calls `git config`, `git remote set-url` from inside the worktree | Non-issue by design |

### Clean working tree requirement

pi-subagents enforces a clean working tree before `git worktree add`. This
is overly strict for our use-case: the conductor's working tree may have
legitimate unstaged changes (open design doc, in-progress work) and the
builder should still be isolatable.

**v0.13 decision: no clean-tree enforcement.** `git worktree add ... HEAD`
creates the worktree from `HEAD` regardless of working-tree state. The
worktree starts from the last commit, which is correct. The parent's
unstaged changes are NOT visible in the worktree. This is the desired
behaviour.

Emit a warning (not error) if `git status --porcelain` shows unstaged
changes, so users know the worktree started from HEAD, not the dirty tree.

## Alternatives considered

### Option A: `/tmp/` placement (pi-subagents default)

**Pros:** Simple, no gitignore management, consistent location.

**Cons:** Breaks `brazil-build` (proven today). Cross-device link failure.
Not viable for the primary use-case.

**Rejected.**

### Option B: Separate full Brazil workspace per run

**Pros:** Full isolation including version set, dependency cache.

**Cons:** `brazil ws create` takes minutes; not practical for per-slice
invocations. Overkill for single-package code generation.

**Rejected** for v0.13; mentioned as a v2+ option for multi-package cross-workspace tasks.

### Option C: `<WorkspaceRoot>/worktrees/<run-id>/src/<pkg>/` (AWSLogsAutonomousAgent shadow workspace)

**Pros:** Allows multi-package isolation in one place.

**Cons:** Requires symlinking the full `src/` tree; `brazil-build` ordering
across packages still unsolved; higher complexity than needed for v0.13's
single-package scope.

**Rejected** for v0.13. Worth revisiting for v0.14 multi-package worktree design.

### Option D (chosen): `<gitRoot>/.worktrees/conductor-wt/<run-id>/`

**Pros:** Inside workspace root (Brazil happy); same filesystem volume
(no cross-device link); AGT-Superpowers convention; clean `git worktree
list` output; trivially gitignored.

**Cons:** gitRoot may be a symlink (handled via `realpathSync`). Worktrees
accumulate inside the package dir (mitigated by GC + `.gitignore`).

**Chosen.**

## Risks

| Risk | Severity | Mitigation |
|---|---|---|
| `.worktrees/` not gitignored; accidentally committed | Medium | Auto-append to `.gitignore` on first use |
| Orphaned worktrees after conductor restart | Medium | GC `delete` path reads `run.worktreePath` from persisted record |
| Worktree branches accumulate (`git branch -a`) | Low | `removeWorktree` deletes the branch; GC path also deletes |
| Builder commits to worktree branch; merge-back not done | Medium | v0.13 explicitly documents merge-back as manual; conductor prompt updated |
| Multi-package edits: second package unprotected | High (known) | Documented limitation; v0.14 scoped |
| Resume after worktree removed: branch name collision | Low | Branch re-creation checks for existing branch and removes first |
| `m_w_c=1` cap not relaxed in v0.13 | Low | Correct — cap relaxation requires separate config option + testing |

## Deferred to v0.14+

- **`worktree: true` per-call override on `ensemble_spawn`** (like `kill_on_stall`). V0.13 is frontmatter-only; per-call is the natural v0.14 add, following the `kill_on_stall` cascade pattern.
- **Auto-merge** of worktree branch back to base branch on run completion. V0.13 leaves merge manual; v0.14 designs `merge_strategy: squash|rebase|none` (autoloop's pattern).
- **Raising `m_w_c` cap** for worktree-capable personas. Safe to do once v0.13 proves isolation works in dogfood.
- **Multi-package shadow workspace** for tasks spanning >1 Brazil package. Requires the AWSLogsAutonomousAgent-style `createMultiRepoShadow` approach.
- **`worktreeSetupHook`** (pi-subagents pattern): a shell script run after `git worktree add` for package-specific initialization (e.g. `brazil-build`). Useful but not needed for initial isolation.
- **node_modules symlinking**. The CDK/TS optimization from pi-subagents. Omit from v0.13; add as a `worktreeNodeModulesLink: boolean` persona frontmatter option in v0.14.
- **`/conductor worktree` slash command** for manual merge/inspect/clean. Useful for power users; not blocking.

## Open questions

1. **Should `createWorktree` require a clean working tree and fail loudly, or warn and proceed?** Design above chooses warn-and-proceed. If the builder's behaviour with dirty-parent-tree is confusing in dogfood, upgrade to error.

2. **`personas/builder.md` default `worktree: true` or opt-in?** Design activates it in both `builder.md` and `simplifier.md`. If `git rev-parse` fails (non-git cwd), `resolveWorktreeSpec` returns `null` and the spawn falls back gracefully to shared cwd with a warning. This makes default-on safe: isolation when possible, silent degradation otherwise. **Resolved: default-on with soft fallback.**

3. **Should `run.worktreePath` persist through `cold-archive`?** Yes — cold-archive preserves `record.json` (and thus `worktreePath`). The worktree itself also persists until `delete`. This means a cold-archived run can still be resumed with its worktree intact. Acceptable.

4. **What happens when the conductor process restarts while a worktree-run is active?** The `reconcileOrphansAtStartup` path will classify the run as `killed` (orphaned). The worktree will NOT be cleaned up by reconcile itself; it will wait for GC `delete`. This is acceptable — orphaned worktrees are bounded in size and cleaned by the next GC run.

## Test strategy

**Unit (pure, no subprocess):**
- `resolveWorktreeSpec(cwd, id)` returns correct path for Brazil vs non-Brazil cwd
- `detectBrazilWorkspaceRoot` walks up correctly; returns null for non-Brazil
- `worktreeSpecFromRun` handles records with/without worktreePath

**Integration (real git, no `pi` subprocess):**
- `createWorktree` + `removeWorktree` against a `git init` temp repo
- gitignore bootstrap: `.worktrees/` appended exactly once on repeated calls
- `fullDelete` in executor removes the worktree dir and branch when `worktreePath` set
- `spawnRun` with fake subprocess (inject `runHookImpl`-style seam): assert `run.worktreePath` set and `effectiveCwd` used

**Manual smoke (post-ship):**
- Spawn a `builder` with `worktree: true` on pi-conductor itself; assert worktree created under `.worktrees/conductor-wt/`; commit lands on worktree branch; terminal removes worktree
- Spawn same on a Brazil package inside `/workplace/$USER/Rosie/`; assert `brazil workspace show` succeeds from inside the worktree

## Decision log

| Question | Decision | Rationale |
|---|---|---|
| Placement | `<gitRoot>/.worktrees/conductor-wt/<run-id>/` | Brazil-compatible (inside workspace root), same-volume, AGT-Superpowers convention |
| Default-on builder/simplifier | Yes, with soft fallback | `resolveWorktreeSpec` returns `null` on non-git cwd; spawn degrades to shared cwd with warning |
| gitignore bootstrap location | `createWorktree` (not `resolveWorktreeSpec`) | Mutate with the action, not the resolution; resolver stays honest |
| `worktreePath` clear on removal failure | Conditional (only on success) | Leaves field set so GC `delete` can find and clean orphan; resume path re-checks `existsSync` |
| Clean-tree enforcement | Warn, don't error | Parent tree state shouldn't block sub-agent isolation |
| Merge-back | Manual (v0.13) | Auto-merge needs conflict resolution design; ship isolation first |
| `m_w_c` cap | Keep at 1 (v0.13) | Isolation is prerequisite; cap relaxation is a separate risk |
| Per-call override | Frontmatter only (v0.13) | Follow `kill_on_stall` pattern; per-call in v0.14 |
| node_modules symlink | Omit (v0.13) | CDK optimization; adds complexity without blocking value |
| Setup hook | Omit (v0.13) | Useful but not blocking; v0.14 item |
| Multi-package | Single-package only (v0.13) | Shadow workspace approach is a v0.14 design |
