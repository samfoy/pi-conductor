## Preamble â€” investigations + amendments resolved up-front

### A1. Orphan reconciliation also fires on `session_shutdown` (oracle amendment 1, R7)

`shutdown.ts:67-83` SIGTERMs running children but never `writeRecord`s them
(`oracle.md:182-186`). Until next `session_start` runs reconcile, on-disk state
says `running` indefinitely. Slice 2 expands to also patch `handleSessionShutdown`:
when `event.reason !== "reload"`, after the SIGTERM loop, call a new helper that
flips each affected run's record to `status: "killed"` with
`errorMessage: "orphaned: process gone (reconciled by GC)"` and
`finishedAt: Date.now()`. Best-effort consistent with everything else in
`runs.ts:1093-1100` (`writeRecord` already swallows errors).

### A2. Post-archive `ensemble_send` UX is documented + tested (oracle amendment 2, R8)

`oracle.md:97-100` traces the chain `runDir/session/<ts>_<uuid>.jsonl` â†’
`validateSendable` â†’ `pi --session <path>` and confirms cold-archive preserves
resume *mechanically*. The user-facing UX gap is what the resume's first card
shows. **Pinned behavior** for slice 3 acceptance:

- `validateSendable` (`runs.ts:843`) succeeds because `session/` is intact.
- `processLine` (`runs.ts:770`) re-creates `transcript.jsonl` on first event
  via `appendFile(O_CREAT|O_APPEND)` â€” a fresh post-resume transcript starts.
- `Run.transcriptPath` in memory is unchanged; the path resolves to the new
  file. No in-memory mutation needed.
- Slice 8 history annotation marks the run as "post-archive resume" so users
  know the visible transcript is partial.
- Slice 3 ships an integration test that cold-archives a run, simulates an
  `ensemble_send` (synthetic event push to `processLine`), asserts a new
  `transcript.jsonl` exists with only the post-resume events.

### A3. `--dry-run` output includes 4 totals (oracle amendment 3, R9)

Slice 6 acceptance pins the `--dry-run` report to include all of:

- `bytes_to_reclaim` (sum of cold-archive transcript sizes + delete dir sizes)
- `runs_to_archive` (count of cold-archive entries)
- `runs_to_delete` (count of full-delete entries)
- `runs_lose_resume` (count of full-delete entries that had `session/<ts>.jsonl` present, since deletion kills resume)

Plus the per-entry list. Test pinned: feed a fixture inventory with mixed
actions, assert all four totals appear and add up consistently.

### A4. Stale "~6 transcripts > 100 MB" claim corrected (oracle amendment 4)

`design.md:51` says "designer/planner top-out at 200â€“850 MB; ~100 MB cap

### A5. Slice 1 registry seam pinned (oracle Â§6 slice 1)

`walkInventory(runsRoot, registry, now): InventoryEntry[]` is the only function
that consults `RunRegistry`. The seam is concretely `entry.inMemory: Run | undefined`
populated by `registry.get(id)` (or equivalent) at walk time. `planReclaim` then
checks `entry.inMemory !== undefined && !isTerminal(entry.inMemory.status)`
**OR** `entry.inMemory?.proc !== undefined` (process still alive). Either
condition forces `action: "keep"` with `reason: "active-in-memory"`. This
matches the architecture diagram (`design.md:158-172`) and keeps `planReclaim`
purely `(InventoryEntry[], Config, NowMs) â†’ ReclaimPlan`.
