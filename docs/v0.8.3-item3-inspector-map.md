# v0.8.3 Item 3 — Inspector Map

Survey of the foreground transcript view + Ctrl+G focused-stream overlay code surface
ahead of the v0.8.3 UX redesign. Read-only; descriptive, not prescriptive.

HEAD: `f38d557` (clean tree at survey time).

## 1. Per-file purpose

- **`src/transcript.ts`** (270 LoC) — Pure renderer. Exports `renderHeader(run, width)`, `renderFooter(width)`, `renderTranscript(run, opts)`. Consumes a `Run` (status + `messages: AgentMessage[]`) plus `TranscriptOptions { width, collapseToolCalls, showThinking }`. Emits `string[]`, one entry per line, hard-wrapped/truncated to `width`. No TUI imports, no theme, no I/O. The Component layer is responsible for styling. Internal helpers: `renderToolCall`, `renderThinking`, `summarizeArgs`, `wrap`, `truncateOrPad`, `padOrTruncate`, `firstLine`, `shorten`, `turnSeparator`. `STATUS_GLYPH` map duplicated locally.

- **`src/foreground-stream.ts`** (320 LoC) — Pure renderers + small async helpers for the v0.4 inline-streamed foreground spawn. Exports `renderForegroundStream(run, width)` (header + collapsed transcript, tail-truncated to `STREAM_MAX_CHARS`), `renderForegroundSummary(run)` (3-line post-completion card), `resolveStreamWidth(cols)`, `awaitOrDetach`, `renderForegroundDetachedResult`, `createUpdateThrottle`, `installPostDetachCompletionListener`. Reuses `renderHeader` + `renderTranscript` from transcript.ts so the streamed view matches Ctrl+G. Has its own `STATUS_GLYPH` copy.

- **`src/widget.ts`** (148 LoC) — Effectful TUI widget mounted `belowEditor` whenever any non-terminal sub-agent exists. `mountEnsembleWidget(registry, getCtx)` returns `{ refresh, dispose }`. Subscribes to registry changes; renders one row per active run (glyph, persona:id, elapsed, last-tool-call hint, usage). Recently-finished rows linger `FINISHED_LINGER_MS` (8000ms). Uses `theme.fg(color, text)` from the host's pi-tui theme — the only file in scope that styles output.

- **`src/focused-stream-overlay.ts`** (148 LoC) — `FocusedStreamOverlay` Component. Effectful in that it `implements Component` and dispatches keystrokes via `handleInput(data)`, but `render(width)` is a thin composition: `renderHeader` + `renderTranscript.slice(offset)` + `renderFooter`. State lives in `FocusedStreamModel` and the keybinding callbacks (`onClose`, `onKill`, `onSend`, `onChange`).

- **`src/focused-stream-model.ts`** (123 LoC) — Pure navigation/fold/scroll state for the overlay. `FocusedStreamModel` holds `_focusedId`, `_collapseToolCalls`, `_showThinking`, `_scrollPerAgent: Map<id, number>`. Methods: `refresh`, `focused`, `cycleNext/Prev`, `focus(id)`, `scrollUp/Down(n)`, `toggleCollapseToolCalls`, `toggleShowThinking`. No TUI imports. Reads from `RunRegistry`.

- **`src/focused-overlay-factory.ts`** (66 LoC) — `createFocusedOverlayComponent(deps)` builds a `FocusedStreamOverlay` from session-scoped deps (`model`, `registry`, `forceTerminate`, `promptAndSendToRun`, `done`). Documented invariant: must NOT register a `RunRegistry` listener (a previous in-line implementation leaked one no-op listener per overlay open).

- **`src/focused-overlay-shortcut.ts`** (74 LoC) — `installFocusedOverlayShortcut(ctx, options)`. Wires Ctrl+G via `ctx.ui.onTerminalInput` (pi reserves Ctrl+G as a built-in shortcut and silently drops conflicting extension bindings; same workaround used for Esc detach). Returns idempotent unsub. Headless ctx (`hasUI=false`) returns no-op unsub. Pass-through when overlay already open so the overlay's own bindings see the keystroke.

### v0.7 critic findings: confirm-or-refute

**(a) Duplicate `summarizeArgs` vs `formatToolCallShort` — STILL PRESENT.**
- `transcript.ts:193` `function summarizeArgs(name, args)` — used by collapsed renderToolCall (one chevron line) at `transcript.ts:162`.
- `event-handler.ts:110` `export function formatToolCallShort(name, args)` — populates `run.lastToolCall` for the widget at `event-handler.ts:84`.
- Both switch on `bash|read|write|edit|grep` with overlapping fallback logic; output strings differ slightly (e.g. `summarizeArgs` returns just the command, `formatToolCallShort` prepends `$ `; `summarizeArgs` calls `shorten(_, 50)` while `formatToolCallShort` inlines a 50-char cap; only `formatToolCallShort` calls `shortenPath`). They render the same conceptual data ("short tool-call summary") in two different styles. The duplication will bite any "tool-call line redesign" — both must change in lockstep.

**(b) First-line-only result truncation in `renderToolCall` — STILL PRESENT.**
- `transcript.ts:184` `out.push(truncateOrPad("  ↳ " + firstLine(text), opts.width));`
- `firstLine` defined at `transcript.ts:217–220`: `const idx = s.indexOf("\n"); return idx === -1 ? s : s.slice(0, idx);`
- Effect: a multi-line tool result (e.g. a `bash` stdout with 200 lines, an `edit` diff) collapses to a single line in the transcript. There is no "more lines hidden" indicator. Screenshot evidence in `docs/v0.8.1-handoff-context.md` line 54 sub-issue (2) confirms this is what the user is reading as "tool results are invisible".

### Chrome inventory

- **Border characters used:** only U+2500 (`─`) for top/bottom rulers + turn separators; U+2503 (`┃`) as thinking gutter; U+25B8 (`▸`) collapsed tool-call; U+25BE (`▾`) expanded tool-call; U+21B3 (`↳`) inline tool-result preview. Status glyphs: `●◌⏸✓✗■⏱`. No corners (`└┘┌┐`), no double-line, no shading.
- **Padding / indent:** thinking lines indent by 4 chars (`"  ┃ "`). Expanded JSON arg lines indent by 2 (`"  "`). Inline result indents by 2 then `↳ ` (4 visible). Empty-state placeholder uses 2-space indent. No outer left/right padding — content runs flush to columns 0 and `width`.
- **Footer:** single ruler row + a single line of dot-separated keybind hints, greedy-packed (`FOOTER_HINTS` array on transcript.ts:50). Not styled. No spacing above/below the ruler. Hard-truncated to `width` if even the first hint overflows.
- **Scroll indicator:** **none.** `model.scrollOffset()` is applied via `transcript.slice(offset)`. There is no "↑ N more lines above" or "↓ N more below" affordance, no scrollbar gutter, no "page X/Y" counter, no fade. Header and footer stay pinned.
- **Color / contrast:** **none in the pure renderers.** transcript.ts emits raw text. Coloring would only be applied if the overlay component's host wraps the rendered string[] in a styled `Text`/component — current implementation in `focused-stream-overlay.ts:43` returns the strings directly from `render(width): string[]`. (Compare: widget.ts colors via `theme.fg("accent"|"dim"|"warning"|"success"|"error")`.) The overlay is therefore monochrome by construction.
- **Header layout:** fixed three rows. Top ruler (full width `─`). Center row: `glyph persona (id) — status elapsed [usage]` left side, right-aligned `[usage]` if present, joined by space-padding via `padOrTruncate`. If `left` overflows the available budget it ellipsises. Bottom ruler. Same width as top. Header glyph from `STATUS_GLYPH` map.
- **Stream lines (assistant text):** wrapped via `wrap(text, width)` — splits on `\n` then hard-cuts at `width` with a "look back up to 20 chars for a space" heuristic. No leading indent, no quoting prefix.
- **Thinking events:** rendered as a `"  ┃ thinking"` header line followed by gutter-prefixed wrap. Hidden by default; surfaced only when `t` toggled.
- **Tool events (collapsed, default):** single line `▸ <name> <summarizeArgs(name,args)>` where `summarizeArgs` is the per-tool head-truncation (cap 50 chars, ellipsis suffix). Result is **not shown at all** in collapsed mode — only the call.
- **Tool events (expanded, after `c`):** multi-line. Header `▾ <name>`, then `JSON.stringify(args, null, 2)` line by line indented 2 (each line then individually `truncateOrPad`'d), then per-result-part one line of `"  ↳ " + firstLine(text)`. **No `✓`/`✗` outcome marker, no exit code, no result tail.**
- **Turn separator:** `"── " + "turn N" + " " + "─"*remaining` filled to `width`. Same character class as the rulers — so a casual eye sees four near-identical horizontal lines per turn (top header ruler, bottom header ruler, turn separator, footer ruler) with no colour cue distinguishing them.
- **Distinct visuals between text/thinking/tool:** lean — text is unprefixed prose, thinking has a `┃` gutter, collapsed tool has a `▸`, expanded tool has `▾`/`↳`. No color contrast, no background highlight, no outcome indicator. Three different glyphs are the only signal.

## 8. Surprises / red flags

1. **`STATUS_GLYPH` map duplicated in three places.** `transcript.ts:25–33` and `foreground-stream.ts:24–32` are byte-identical 7-key records; `widget.ts:statusGlyph` (lines 135–156) is a third copy with the same alphabet but inlined as a switch with `theme.fg` calls. Any future status added (e.g. `cancelled`, `detached`) must be propagated to all three; today nothing enforces this.

2. **`renderToolCall` collapses tool results to a single line via `firstLine`** (`transcript.ts:184`, helper at 217). Multi-line bash stdout, edit diffs, and read outputs all show as their first line only with no "+N more" affordance. Combined with **no `✓`/`✗` outcome marker** in either collapsed or expanded mode, the overlay literally cannot show whether a tool call succeeded — confirmed v0.8.1 handoff complaint sub-issue (2) "tool results are invisible".

3. **Duplicate `summarizeArgs` (transcript.ts:193) vs `formatToolCallShort` (event-handler.ts:110).** Two implementations of "short tool-call summary" with overlapping switches and slightly different output (prefix `$ ` only in event-handler; `shortenPath` only in event-handler; per-key cap 30 only in transcript). Drift risk for redesign that touches one path. Note also that `formatToolCallShort` lives in `event-handler.ts` (an out-of-scope file), so a v0.8.3 redesign limited to the 7 in-scope files cannot fix this in isolation — the widget's `lastToolCall` uses the `event-handler.ts` copy and would lag any change in `transcript.ts`.

Other findings worth surfacing to the designer:

- **`FocusedStreamOverlay.render` calls `model.refresh()` as a side effect on every paint** (focused-stream-overlay.ts:46). That re-evaluates focus against the registry inside what looks like a pure render. Test coverage exists, but a redesigner expecting `render(width)` to be a pure projection of model state will be surprised. Also forces the overlay to depend on a mutable model rather than a snapshot.

- **`transcript.ts` hand-rolls `wrap` (transcript.ts:222) ignoring ANSI escapes** while the host exports `wrapTextWithAnsi` and `truncateToWidth` from `@earendil-works/pi-tui`. Any future colorization of transcript text inside the pure layer will mis-wrap because `String.length` counts ANSI bytes. A redesign that adds color must either keep coloring in the Component layer (status quo for widget.ts) or migrate to the host helpers.

- **No scroll-position indicator in the overlay.** `scrollOffset` is per-agent state but there is zero visual cue ("scrolled N lines, M total"). A user who scrolls then cycles agents has no way to see they're not at the top of the new agent's transcript (each agent has its own offset).

- **Footer hint list (`FOOTER_HINTS`) is declared as a literal array** in transcript.ts:50 with no link to the actual key dispatch in `FocusedStreamOverlay.handleInput`. Drift risk: adding/removing a key requires a coordinated edit. Already partially drifted — `FOOTER_HINTS` lists `s send` and the dispatch handles `s`, but `onSend` is optional and silently no-ops when missing; the footer hint stays visible regardless.

- **Foreground stream ignores the model's user toggles entirely.** `renderForegroundStream` hardcodes `collapseToolCalls:true`, `showThinking:false` (foreground-stream.ts:60–62). A user who opens Ctrl+G mid-foreground-spawn and toggles `t` will see thinking in the overlay but not in the inline tool-call card — the two views can diverge.

- **`STREAM_MAX_CHARS = 32 * 1024` is a JS-string-length cap, not a byte cap** (foreground-stream.ts:42–47, comment acknowledges this). Multi-byte Unicode (CJK, emoji-heavy thinking) can produce a much smaller visible budget than ASCII. Tail-truncation snaps to next newline within 1024 chars — could leave the prefix line orphaned.

- **`renderForegroundSummary` interpolates `run.errorMessage` verbatim** (foreground-stream.ts:104) into a quoted string after `collapseWhitespace`. No escaping of internal quotes; would render `"  → "an "error" ""` if the message contains quotes. Cosmetic but visible.

- **Widget's `formatRow` is the only file in scope that knows how to color rows**, but uses theme slots `accent`/`dim`/`warning`/`muted`/`success`/`error` inconsistently with what the overlay would want — there's no shared "status → theme color" map even though both files want one.

- **No tests for `widget.ts`.** Effectful but the renderless inner `render` plus `formatRow` could be lifted to a pure helper for snapshot testing. Currently only validated end-to-end.

### Top 3 surprises for designer attention

1. **Tool-result rendering is single-line + no outcome marker** — `firstLine()` in `renderToolCall` plus the absence of any `✓`/`✗` glyph means the overlay structurally cannot communicate "did this tool call succeed". Direct cause of the v0.8.1 "tool results are invisible" complaint.
2. **Triple-duplicated `STATUS_GLYPH` + duplicate `summarizeArgs` / `formatToolCallShort`** — any redesign of glyphs or tool-summary text must touch transcript.ts, foreground-stream.ts, widget.ts, AND the out-of-scope event-handler.ts in lockstep.
3. **Pure renderers emit no color and host text-wrap utilities go unused** — transcript.ts hand-rolls width-wrapping in raw `String.length` and the overlay component returns the strings unstyled, so the entire Ctrl+G view is monochrome by construction. Adding color requires either moving styling into the Component layer (widget.ts pattern) or switching to `wrapTextWithAnsi` / `truncateToWidth` from `@earendil-works/pi-tui`.
