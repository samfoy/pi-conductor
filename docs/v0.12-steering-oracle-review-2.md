# v0.12 — Steering — Oracle gate review #2 (post-design)

Gate review of `docs/v0.12-steering-design.md` (1259 lines, revised
2026-05-27) against the four pre-design fixes in
`docs/v0.12-steering-oracle-review.md` §1 plus drift introduced by
the +129-line revision. Slice-readiness check, not redesign.

---

## 1. Verdict

**APPROVE WITH FIXES.**

The four pre-design fixes are correctly applied at the spec level.
One real drift survived the revision: the design's claim that
`resolveSteerable` is *signature-isomorphic* to today's
`resolveKillOnStall` is factually wrong — today's signature is
`(run: Run, defaultKillOnStall: boolean): boolean`, not
`(opts: { perCall, project, user, defaultValue }): { value, source }`
(`docs/v0.12-steering-design.md:292`–`:294`). One editorial fix and
two open-question triage moves and the doc is ready for `planner`.

## 2. Fix-application audit

### Fix #1 — Drop persona-frontmatter layer for `steerable`

**APPLIED CORRECTLY.**

- §3 Q1 (`docs/v0.12-steering-design.md:174`–`:204`) restates lock
  with `:517` citation; Q1 explicitly says "Persona-frontmatter
  layer DEFERRED. Mirrors v0.10 `kill_on_stall`'s deferred shape
  exactly." (`:177`–`:183`).
- §4.1 (`:286`–`:351`) drops the persona layer everywhere; the
  cascade is `perCall > project > user > defaultValue` (`:303`–`:316`).
- §4.1 explicitly forbids `Persona.steerable` even as a parsed-
  but-unused stub (`:342`–`:346`); §8 slice 1 repeats it (`:1092`);
  §9 v2 list (`:1170`–`:1173`) keeps it deferred.
- Grep confirms zero residual `persona.steerable` reads anywhere
  in the design that aren't explicit "DO NOT" callouts.

**Issue (editorial, ADJUST-level).** The "Signature isomorphic to
`resolveKillOnStall`" paragraph (`:289`–`:294`) misrepresents the
existing function. Today (`src/watchdog.ts:287`) the signature is
`resolveKillOnStall(run: Run, defaultKillOnStall: boolean): boolean`
— a 2-input final-step collapse, not a 4-layer resolver returning
`{value, source}`. The 4-layer cascade for `kill_on_stall`
*intent* is real and pinned at `PRD.md:517`, but the *current
code* collapses per-call into `Run.killOnStall` upstream of the
function. Slice 1 should either (a) downgrade `resolveSteerable`
to match the existing 2-input shape, or (b) refactor
`resolveKillOnStall` in slice 1 to match the new 4-input
provenance shape (small change; preserves "isomorphic" claim and
is independently useful for `/conductor watchdog status`
diagnostics). Pick one before slicing.

Note: `mergeConfig` in `src/config.ts:74`–`:121` does NOT currently
merge `defaultKillOnStall` from JSON — so `defaultSteerable` is
genuinely new project/user-config surface. Slice 1 must wire a
mergeConfig branch for it; design alludes (`:355`) but doesn't pin.

No drift. Witness scope matches the narrower policy.

The four pre-design fixes are applied cleanly and the +129 lines
add real content (orphan-RPC reconcile branch, asymmetry

The single real ADJUST is the §4.1 "isomorphic" wording — small,

locks held; the shape is sound. Spawn redteam later if the live
RPC integration in slice 5 surfaces unexpected pi-coding-agent
behaviour.

Open questions §10 Q1, Q2, Q4, Q5 are not blockers — designer
can pre-resolve Q1+Q2 with a one-line edit and let planner
schedule Q4+Q5. Q3 (vendor RPC types) is the only one worth
