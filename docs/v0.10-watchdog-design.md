# Design: v0.10 sub-agent watchdog

Status: design (no code). HEAD at design time: `c19950a` (clean).
Inputs: `src/runs.ts` (spawn pipeline + `RunRegistry`), `src/event-handler.ts`
(the seam where every observed sub-agent event flows), `src/types.ts:191`
(`Run` shape, including `lastEventAt: number`), `docs/wdd.md`,
the three witnessed hang transcripts (`critic-yjsn`, `builder-z98m`,
the orchestrator's own `npm test`).

---

## 1. Goals & non-goals

### Goals

- **Detect stalled sub-agents early.** A `bash` blocking on stdin (unclosed
  quote, blocking read, network stall) emits zero conductor-observable
  events while `run.status === "running"`. Today the only safety net is
  the per-spawn wall-clock `timeoutMinutes`; default 60 min wastes ~$3–5
  per stuck run before firing.
- **Surface to parent with enough context to recover.** Parent decides
  kill vs wait — same shape as `nonSubstantiveFinal` advisory.
- **Let the parent opt into auto-kill** via `kill_on_stall: true` per spawn.
- **Preserve `timeoutMinutes`** as the ultimate safety net (defense in depth).
- **Fully testable.** Detector is pure (events + clock → warning). Enforcer
  is a thin shell around `forceTerminate`/`registry.notify`. Witness-able
  per `docs/wdd.md`.

### Non-goals

- **Replacing pi's tool-execution sandbox.** Pi runs the `bash`. We observe
  pi's event stream from outside; we cannot interrupt the bash itself.
- **Static analysis of bash commands** (e.g. unclosed-quote prediction).
  Too brittle, too much surface; the runtime signal is sufficient.
- **A full process-supervision system.** No retries, no restarts, no
  circuit-breaker. Just detect-and-surface-or-kill.
- **Detecting sub-agents stuck in *thinking*.** `message_update` events
  fire during streaming, including thinking blocks. The witnessed hangs
  are inter-message (between assistant turn end and tool result end),
  not intra-streaming.
- **Replacing or weakening v0.8.2 substance-check.** Substance-check fires
  post-completion ("did the run produce a real final message?"). Watchdog
  fires mid-run ("is this run still alive?"). Orthogonal.

## 2. Detection primitives — chosen v0.10 set

The pi event protocol gives us five usable lifecycle markers (per
`src/event-handler.ts`): `message_update` (streamed, partial — does NOT
update `lastEventAt`), `message_end` (assistant turn complete — updates
`lastEventAt`), `tool_result_end` (tool completed — updates `lastEventAt`),
`turn_end`, `agent_end`. Critically, **between an assistant `message_end`
that contains a `toolCall` and the matching `tool_result_end`, no events
flow.** That window is the entire stall surface.

| # | Primitive | Signal we already have? | Complexity | False-positive risk | Decision |
|---|---|---|---|---|---|
| a | Per-tool-call timeout (e.g. 10 min hard cap on any single tool call) | Approx — derive from `lastEventAt` + outstanding-toolCall test | M | Low (if cap is generous) | **Subsumed by (b).** Same code path. |
| b | **No-output-since-N seconds while running** | ✅ direct — `Date.now() − run.lastEventAt > N` while `status === "running"` and not `paused` | S | Low if N ≥ 120s; rises sharply below 60s | **SHIP — primary signal.** |
| c | Repeat-detection (same tool args ≥3 in K turns) | Need to hash + remember args per turn | M | High — legitimate `npm test` retries, polling `git status`, etc. | **Defer to v0.11.** Not the witnessed failure shape. |
| d | Tool-call ratio (>X% of run-time in single call) | Derived from (b) but noisier | M | Medium — long runs naturally accumulate big single calls | **Defer.** Subsumed by (b)+(a). |
| e | Stack heuristic ("Now let me check..." + long-pending bash) | Need text classifier on assistant prose | L | High — orient-yourself prose is normal | **Defer.** Brittle, low signal. |

**Why (b) is enough.** All three witnessed hangs share one property:
**zero events from pi for >5 minutes while the run was active and not
paused.** That is exactly what `(b)` matches. (c)/(d)/(e) would have
fired *additionally* but would not have fired *first*; (b) is the
necessary-and-sufficient signal for the witnessed bug class.

**Two thresholds, both expressed as the same primitive** to avoid
inventing two state machines:

- **Soft threshold** — default `120s`. Crosses → emit `stalled` advisory
  to parent + flip widget to warning hue. No kill.
- **Hard threshold** — default `600s` (10 min). Crosses → if `kill_on_stall`
  is set on the spawn, call `forceTerminate(run, "watchdog: stalled
  >Ns")`. Otherwise, just escalate the advisory text.

The hard threshold *is* the per-tool-call timeout from primitive (a),
emitted via the same primitive. No separate code path.

### What `lastEventAt` already gives us

`src/event-handler.ts` updates `run.lastEventAt = Date.now()` on every
`message_end` and `tool_result_end`. That is the exact silence-window
boundary we need. **No new event-handler instrumentation required** — the
field already exists and was added for the v0.8.3 idle-activity header.

### Negative cases the detector must not fire on

- `run.status !== "running"` (queued/paused/terminal).
- `run.pausedAt !== undefined` (paused — silence is expected).
- `Date.now() - run.startTime < softThreshold` (just-spawned grace; bounded
  by max(softThreshold, 30s)).
- The detector should not double-fire — once a run has crossed soft, don't
  re-emit until either: (a) it crosses hard, or (b) `lastEventAt` advances
  (run recovered), or (c) it terminates.

## 3. Recovery primitives

| Trigger | Default action | Configurable to |
|---|---|---|
| **Soft (≥ soft threshold)** | (1) Set `run.stalledSince = lastEventAt` on the Run; (2) `registry.notify(run)` so widget redraws; (3) emit one parent-visible advisory line, deduped per-run-per-threshold-cross | Suppress with `watchdog_silence: true` |
| **Hard (≥ hard threshold)** | (1) Same advisory, escalated text; (2) **if `kill_on_stall: true`** → `forceTerminate(run, reason)`. Otherwise, no kill — `timeoutMinutes` is still the ultimate stop. | `kill_on_stall: true \| false` |
| **Recovery (lastEventAt advances after stall)** | Clear `run.stalledSince`; `registry.notify`; one-line "recovered after Ns" advisory (deduped) | — |

### `kill_on_stall` default

**OFF by default.** Rationale: a 6-minute legitimate stall (large `npm test`
suite, a slow `brazil-build`, a network-bound `aws` call against a
throttled API) should produce a warning, not a kill. Autonomous chains
(this session's pattern) opt in. The flag flows through `ensemble_spawn`
arguments analogous to `timeoutMinutes` (v0.8.3 pattern).

### Soft-advisory shape (parent-visible)

Borrows `nonSubstantiveFinal`'s shape. New `<sub-agent-stalled>` envelope
delivered via the same `pushCompletionNotification` path but as a
mid-run notification:

```xml
<sub-agent-stalled>
  <agent-id>builder-z98m</agent-id>
  <persona>builder</persona>
  <silent-for>184s</silent-for>
  <threshold>120s</threshold>
  <last-tool-call>$ npx tsx --test tests/runs-helpers.test.ts 2>&1 | grep -E "</last-tool-call>
  <hint>If kill_on_stall is desired, call ensemble_kill agent_id=builder-z98m. Otherwise the wall-clock timeout (60m) will fire.</hint>
</sub-agent-stalled>
```

The `last-tool-call` is the `run.lastToolCall` field already populated
by `event-handler.ts:formatToolCallShort`. **This is the value-add of
the watchdog: the parent gets the *exact stuck command* in the warning,
not just "something is slow."**

### Single advisory per crossing

The detector tick fires every N seconds (default 30s). Dedupe by
**state transitions**, not tick events:

```
state ∈ { fresh, soft, hard, recovered }
soft→hard       → emit hard-advisory once
soft→recovered  → emit recovery-advisory once
fresh→soft      → emit soft-advisory once
hard→recovered  → emit recovery-advisory once
```

Per `Run`, store `watchdog_state: "fresh"|"soft"|"hard"|"recovered"`.
Detector returns the *transition* (`null` if no transition); enforcer
acts on transitions only. This is the same pattern as the GC
`reconcile-orphan` tier ladder.

## 4. Architecture

### Module placement: new `src/watchdog.ts`

Three reasons not to put it in `src/runs.ts` or `src/event-handler.ts`:

- `runs.ts` is already 1273 LoC; the v0.8.1 handoff flagged it as
  needing extraction. New behavior should land in its own file.
- `event-handler.ts` is purely synchronous + per-event. The watchdog
  needs an interval tick (out-of-band of any single event).
- A separate file lets the **detector** be a clean pure export (testable
  with `git stash` mutation witnesses).

### Detector — pure

```ts
// src/watchdog.ts (sketch — design only, no commit)

export type WatchdogTransition =
  | { kind: "soft"; silentMs: number; threshold: number }
  | { kind: "hard"; silentMs: number; threshold: number }
  | { kind: "recovered"; previousState: "soft" | "hard" };

export interface WatchdogConfig {
  softThresholdSeconds: number;   // default 120
  hardThresholdSeconds: number;   // default 600
  graceSeconds: number;           // default 30; suppress fires inside grace
}

export type WatchdogState = "fresh" | "soft" | "hard" | "recovered";

export function evaluateRun(
  run: Run,
  state: WatchdogState,
  now: number,
  cfg: WatchdogConfig,
): WatchdogTransition | null;
```

**Pure.** No `fs`, no `child_process`, no `registry.notify`. Inputs:
the `Run` snapshot (we read `lastEventAt`, `status`, `pausedAt`, `startTime`,
`lastToolCall`), the previous state, the wall-clock, the config. Output:
`null` (no transition) or one of the three transition shapes.

Testable with stub clocks; mutation-witnessable per WDD (drop the
`pausedAt` guard → the killing test "paused run inside soft threshold
returns null" goes red).

### Enforcer — `Watchdog` class

```ts
// src/watchdog.ts

export class Watchdog {
  constructor(opts: {
    registry: RunRegistry;
    config: WatchdogConfig;
    forceTerminate: (id: string, reason: string) => Promise<void>;
    pushAdvisory: (run: Run, transition: WatchdogTransition) => void;
    tickIntervalMs?: number;  // default 30_000
  });
  start(): void;
  stop(): void;
}
```

Behavior:

- `start()` registers a `setInterval` callback at `tickIntervalMs`.
- On each tick, walks `registry.list()`, calls `evaluateRun` per active
  run, and on a non-null transition: updates the per-run state map,
  calls `pushAdvisory` (emits the `<sub-agent-stalled>` envelope), and
  if `transition.kind === "hard"` and `run.kill_on_stall === true`,
  calls `forceTerminate`.
- Also subscribes to `registry.onChange` to pick up status flips
  (terminal → drop the per-run state from the map; running→running with
  newer `lastEventAt` → no-op the next tick will recompute).
- `stop()` clears the interval and the state map. Called from
  `session_shutdown` (parity with widget/focus-overlay).

State for the enforcer is a single `Map<runId, WatchdogState>` —
bounded by the size of `registry.list()`, which is already bounded by
the run-record GC.

### Wiring in `src/index.ts`

```ts
// inside session_start, alongside widget + focused-overlay-shortcut wiring
const watchdog = new Watchdog({
  registry,
  config: { softThresholdSeconds: 120, hardThresholdSeconds: 600, graceSeconds: 30 },
  forceTerminate: (id, reason) => forceTerminate(id, reason),
  pushAdvisory: (run, transition) => pi.sendMessage(
    formatStallNotification(run, transition),
    { triggerTurn: true, deliverAs: "followUp" },
  ),
});
watchdog.start();
// session_shutdown: watchdog.stop();
```

The `formatStallNotification` helper lives in `src/notifications.ts`
next to `formatCompletionNotification` (one-pattern-per-file rule).

## 5. UX

### Parent-visible advisory (the `<sub-agent-stalled>` envelope)

Format pinned in §3. Header line:

```
## ⏳ `builder` stalled (silent 184s, threshold 120s) — id `builder-z98m`
```

Glyph `⏳` introduced (no collision with current STATUS_GLYPH set —
checked: `● ✓ ✗ ■ ⏱ ◌ ⏸`). The advisory is delivered as a
`triggerTurn: true` `followUp` so the conductor sees it on its next turn
(same pattern as completion envelopes).

### Ensemble panel widget glyph

Widget already shows `· idle 6s` for runs the activity-header detector
classifies as idle. Extension:

- `silentMs < softThreshold` → existing `· idle Ns` (no change).
- `silentMs ≥ softThreshold` → `· STALLED Ns` (uppercase; `theme.fg("warning", …)`).
- `silentMs ≥ hardThreshold` → `· STALLED Ns!` (trailing `!`; `theme.fg("error", …)`).

Reuses the v0.8.3 component-layer styling pattern. One snapshot test per
threshold transition pinned in `tests/widget.test.ts` (or wherever the
widget gets test coverage in v0.10 — currently uncovered per inspector
map).

### `/conductor watchdog status`

New slash subcommand. Table:

```
2 active runs

  id              persona  status   silent  state    threshold  action
  builder-z98m    builder  running  184s    soft     120s/600s  warn (kill_on_stall=false)
  inspector-q9rp  inspector running  4s      fresh    120s/600s  —
```

`/conductor doctor` gains a one-line summary at the bottom:

```
## Watchdog
   active=2  stalled=1  killed-by-watchdog-this-session=0
```

## 7. Risks & open questions

### Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | **False positives on legitimate slow operations.** `npm install`, `brazil-build`, large `npm test` suites can sit silent for 90–180s. Soft threshold of 120s would warn on all of these; users would learn to ignore the warning → cry-wolf. | High | Default soft = **120s** is the sweet spot per witnessed cases (all >5min). Document recommended overrides for known-slow workloads (`stall_threshold_seconds: 300` per spawn). The advisory is *informational*; only `kill_on_stall: true` actually escalates — and that flag is opt-in for autonomous chains. |
| R2 | **Threshold tuning is empirical.** No prior dogfood data; guessing 120s/600s. | Medium | Ship the defaults; expose via config (slice 3). Iterate in v0.11 from observed data (advisory frequency vs real stall captures). |
| R3 | **Pause/resume interaction.** Paused runs naturally have stale `lastEventAt`. | Low | Detector explicitly checks `pausedAt !== undefined` → returns `null`. Mutation-witness this. |
| R4 | **Tick overhead for many runs.** 30s tick × N active runs = O(N) per tick. With current cap of 4 concurrent, negligible. | Low | Per-run cost is `now - lastEventAt` arithmetic + a Map lookup. < 1ms for hundreds of runs. |
| R5 | **Pi's bash is the actual stuck process.** We can detect, surface, and kill the *pi sub-agent*, but the bash subprocess is a child of pi. `forceTerminate` SIGTERMs pi, which should reap its child via process-group default. Confirmed in v0.7 `ensemble_kill` work — same SIGTERM path. | Low | No new mitigation needed; same path that ships today. |
| R6 | **Recovery noise.** A run that crosses soft, then the bash unblocks at 200s, would emit two advisories (soft + recovery) in quick succession. | Low | Deduplication by state-transition (§3). At most one advisory per transition. |
| R7 | **Interaction with the v0.9 GC's "auto-skip in sub-agent context"** (commit `0dd3ec1`). The watchdog must also skip itself when the conductor is running as a sub-agent — same flag check. | Medium | Slice 2 acceptance: watchdog `start()` early-returns when `process.env.PI_AGENT_DIR` indicates a sub-agent context (or the equivalent canonical check used by the GC fix). Pin a regression test. |

### Open questions for the user (planner takes design + decisions)

1. **Default kill_on_stall: ON or OFF?** Recommended OFF (advisory-only).
   ON would close the autonomous-chain loop tighter at the cost of
   surprising kills on slow workloads. The witnessed three cases all
   would have benefited from ON; legitimate slow runs (none witnessed
   yet) would have suffered. Defer to v0.11 once we have advisory data.
2. **Tick interval default 30s OK?** 10s would shrink detection lag at
   negligible cost. 60s is what users expect from `top`-style tools.
   30s feels right for "should the conductor wake up to check on
   children?"
3. **Should the soft advisory trigger a turn (`triggerTurn: true`), or
   silently inject as system context?** Triggering a turn means the
   conductor sees it during normal flow and can act; silent injection
   only surfaces when the conductor next acts on its own. Recommend
   `triggerTurn: true` — matches completion envelopes and is the
   point of an early warning.
4. **`run.stalledSince` field — keep on `Run` (in-memory only) or also
   persist to `record.json`?** Recommend **in-memory only.** The field
   is meaningful only while the run is live. On terminal status it's
   irrelevant. Persisting would also force a `RunRecord` schema bump.
5. **Should `watchdog_silence: true` per-spawn fully suppress, or just
   suppress the parent advisory while keeping the widget glyph?**
   Recommend the latter — the user-facing widget should always show
   reality.

### Deferred from Slice 3

- **`watchdog_silence` per-spawn flag.** Listed in section 6 row 3 alongside
  `kill_on_stall` and `stall_threshold_seconds`, with proposed semantics
  in section 7 Q5 above ("suppress parent advisory while keeping widget
  glyph"). **Not shipped in Slice 3.** Rationale: zero field data on
  advisory cry-wolf yet — building a silencing knob before any false
  positives have been observed is speculation. Same shape as
  `kill_on_stall` (boolean LLM tool param → `Run` field → dispatch
  guard); drop-in once a witnessed need appears (advisory frequency
  telemetry, user complaints, or test-flake noise from real chains).
  Track-level decision, not a contract change — reopen against this
  doc when the trigger fires.
- **Persona-frontmatter watchdog overrides.** The `PersonaOverride`
  type in `src/personas.ts` does not yet carry `killOnStall` or
  `stallThresholdSeconds`. Defensible reading of the cascade in
  section 6 row 3 ("per-spawn arg > project config > user config >
  built-in default") is that persona-frontmatter is a separate axis,
  not enumerated here. Defer until a witnessed need appears (e.g. a
  persona that should default to `kill_on_stall: true` regardless of
  caller). Tracked in `docs/v0.8.2-backlog.md`.

### Hidden assumptions to verify in slice 1

- `lastEventAt` is set on EVERY meaningful event (verified: `event-handler.ts`
  updates on `message_end` + `tool_result_end`; the witnessed hangs sit
  exactly between these).
- `Run.pausedAt` is the canonical paused marker (verified: `src/types.ts:218`).
- `forceTerminate` is the canonical kill path with side-effects we want
  (record flip, registry.notify, completion envelope) — verified per
  `ensemble_kill` v0.7 work and `forceTerminate` exports.

## 8. Recommended chain shape for implementation

Slices 1 and 2 must have **critic gates** (per the design.md table above).
Slice 1 because the detector is the load-bearing logic that everything
else trusts; the WDD witness on the pause-guard mutation must be
asserted before slice 2 lands. Slice 2 because the kill path is
behavioral and one-shot — wrong implementation has runtime consequences.
Slice 3 (config), 4 (UX), 5 (docs) need only verifier coverage; they're
mechanical given the engine.

**Mutation witnesses pre-declared** for the WDD pin (slice 1):

| Mutation | Killing test (must red) |
|---|---|
| Drop the `pausedAt !== undefined` guard | `evaluateRun: paused run inside soft threshold returns null` |
| Drop the `status === "running"` guard | `evaluateRun: terminal run with stale lastEventAt returns null` |
| Flip `>` to `>=` in soft-threshold compare | `evaluateRun: silentMs exactly equal to soft threshold returns null (boundary)` |
| Drop the grace-period check | `evaluateRun: just-spawned run inside graceSeconds returns null even if lastEventAt is stale` |
| Drop the state-transition dedup | `evaluateRun: still-soft run on second tick returns null (no double-fire)` |

For slice 2 (one mutation):

| Mutation | Killing test (must red) |
|---|---|
| Drop the `kill_on_stall` gate (call `forceTerminate` unconditionally on hard) | `Watchdog: hard transition with kill_on_stall=false does NOT call forceTerminate` |

Recommended chain:

```
designer (this doc) → critic (this doc) → planner → builder ⇄ critic (slice 1+2) → builder (slice 3+4+5) → verifier → finalizer
```

Estimated total cost: $25–35.
