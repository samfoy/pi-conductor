## What pi-conductor is

A pi extension that turns the parent pi session into an **orchestrator** driving persona-based sub-agents with first-class TUI visibility. The parent is the conductor, sub-agents are pi subprocesses, and the user can drop into any sub-agent's live transcript inside pi (focused-stream overlay, Ctrl+G). Closes the gap left by `pi-essentials/subagent`, `pi-mono/team-mode`, and `pi-subagents`: **you can watch a sub-agent stream live without leaving pi**.

## v0.8 in one paragraph

**Strict-overseer mode** redefines the parent's role when conductor mode is ON: the parent is a manager, not an implementer. `src/conductor-prompt.ts` §1 declares "you are the conductor: a manager"; §1.5 lists the hands-off rules ("don't read more than ~3 source files yourself; that's `inspector`'s job"); §10 expands the delegation playbook with concrete trigger heuristics for every persona; §11 ships the canonical workflow chains — `oracle → plan loop → decompose → per-slice build loop → finisher gate` — with bounded `ensemble_send` revise loops (cap 3 iterations) and an explicit "no parallel write-capable spawns" rule. Conductor mode is now **OFF by default** with three opt-in paths: `/conductor on` (per-session), `PI_CONDUCTOR_MODE=1` (env), or `defaultMode: "on"` in `~/.pi/agent/conductor.json`. The two states are a binary (`on` / `off`); no third "strict" enum value (decision logged in `PRD.md`).

## v0.8.1 backlog (priority-ordered)

Three of the five items below are pasted **verbatim** from `design.md`'s Open Questions block (search for `2026-05-15 dogfood`). The other two are recorded only in the cartographer's session-state note and **must be folded into design.md before they're forgotten** — flagged with `[NOT-IN-DESIGN]`.

### 1. Persona role-confusion under filtered context inheritance — **High** — prompt + runtime, **M**

> **Filtered-context inheritance vs persona role-confusion.** Witnessed in 2026-05-15 dogfood: two sub-agents (`critic` and `oracle`) received the parent's "Spawning critic-X..." / "Holding for ..." orchestration narration as part of their filtered slice and produced meta-commentary on the brief instead of executing it. Each persona mis-attributed role-identification because the parent's prose talked *about* the spawn ("Spawned X to gate Y. 3/4 slots in use..."). Both recovered cleanly via `ensemble_send` corrective. Mitigations to consider for v0.8.1: (a) strip orchestration narration patterns (`/Spawned|Status:|slots? in use|Holding the turn|N\/4 slots/`) from filtered slices in `src/context-filter.ts`; (b) add a sentinel in persona system prompts re-affirming role identity ("the previous user message is YOUR task, not a quote of someone else's"); (c) both. Out of scope for v0.8; collect more data points before designing the fix.

— from `design.md` Open Questions

### 2. Parallel write-capable builder spawns share git history — **High** — runtime, **M**

> **Parallel write-capable builder spawns share git history.** Two builders on disjoint *files* still share `git log` and the working tree. Witnessed: a `git commit --amend` hit the wrong commit because a sibling builder's commit had landed between plan and execution; an accepted commit was destroyed and history surgery was needed to recover. Mitigations to consider: (a) **enforce serialization** of write-capable spawns in the prompt addendum (no parallel `builder` / `simplifier` spawns); (b) **builders must `git log -1 --oneline` before any history op** and abort if the parent SHA isn't what they expected; (c) **worktree-per-persona** (already on the v0.8+ roadmap per PRD §"Implementation phases"). For v0.8 ship, capture (a) in the prompt addendum as a §11 sub-rule. Open question: do we also gate the cap from §4 (currently 4 concurrent across all personas) to a separate "max 1 write-capable" cap?

— from `design.md` Open Questions. Note: **mitigation (a) shipped** in `0b076b5` §11. Mitigation (b) and the harness-side cap are still open.

### 3. Foreground sub-agent transcript view is visually rough — **Med** — runtime UX, **L**

> **Foreground sub-agent transcript view is visually rough.** Witnessed in 2026-05-15 dogfood (user screenshot of an `investigator` foreground spawn). Concrete problems: (1) **truncation cuts at the wrong place** — `bash aws sts get-caller-identity --profile tickety 2>/…` and `bash ada credentials update --account=$DEV_ACCOUNT_ID …` hide the meaningful argument tail, leaving only the boilerplate prefix; (2) **tool results are invisible** — every `▶ bash X` line lacks a follow-up `✓/✗` outcome, so the user can't tell at a glance whether `ada credentials update` succeeded; (3) **thinking blocks hidden by default** with no inline toggle hint (the `t` keybind exists only in the focused-stream overlay footer); (4) **flat visual hierarchy** — every turn gets the same horizontal-rule + `— turn N` divider; nothing scans; (5) **`Working...` is a constant string** with no signal about whether the persona is mid-thinking, mid-bash, mid-network call, or stuck; (6) **vertical waste** — turn dividers + rules + whitespace stack to ~5 lines of chrome per visible turn; (7) **monochrome** — no color hierarchy distinguishing tool name / args / status / persona; (8) **header-line disconnect** — the rich status line (`investigator-d4hc — running 34s [4t ↑9 ↓839 $0.341]`) is visually separated from the body it describes. Where to fix: `src/transcript.ts` (`renderTranscript`, `renderToolCall`, `summarizeArgs`, the magic-number `20` wrap lookback) and `src/foreground-stream.ts` (the wrapper). v0.7 critic review already flagged adjacent code-quality issues in these files (duplicate `summarizeArgs` vs `formatToolCallShort`, first-line-only result truncation in `renderToolCall`). Out of scope for v0.8; this is a v0.8.1+ UX slice that needs its own design pass — chain shape would be `inspector` (map current rendering) → `designer` (sketch the new view) → `oracle` (consistency review) → `builder ⇄ critic` (implementation). Worth thinking about: argument-tail truncation (head + ellipsis + tail rather than head + ellipsis); a single-line tool-result preview under each tool-call line; thinking-block summary by default with `t` to expand; a dedicated `Working: <current activity>` status that pulls from the latest message_update event; tighter visual hierarchy (fold `— turn N` into a single subdued column, drop the horizontal rules, color the tool name).

— from `design.md` Open Questions

### 4. Sub-agent completes without substantive final message — **Med** — harness, **S** `[NOT-IN-DESIGN]`

Witnessed: `oracle-9k7r` stalled at `"Now let me check \`any\` leakage..."` after 43 turns / ~$2.90 spent on a v0.7 architecture review. Final assistant prose was a 60-character orient-yourself sentence; the actual report never materialized. Recovered via `ensemble_send` corrective. **Harness should detect "no substantive final message after N turns" and either warn or refuse to mark `completed`.** Heuristic candidates: final-text length < 200 chars, OR final block is a `thinking_*` rather than `text`, OR final-text matches `/let me|now I'll|next I'll/`. Lives wherever termination state flips in the run lifecycle (`src/runs.ts`, `src/event-handler.ts`).

### 5. Conductor context inflation — **Med** — runtime, **M** `[NOT-IN-DESIGN]`

Witnessed in this session: 12+ `<sub-agent-completed>` envelopes accumulated in the parent's context, each carrying the full `<result>` body (the largest were designer outputs at ~27KB). After the parent has synthesized findings into committed code, the verbatim envelope is dead weight. **Mitigation: collapse old envelopes** after the parent acts on them — replace `<result>` body with a 200-char summary, keep `agent_id` / `status` / `cost` / transcript path. Either automatic (after N envelopes have arrived since, OR after the parent has issued a `commit` / `ensemble_send` / new `ensemble_spawn`) or a `/conductor compact` slash command. Lives in `src/notifications.ts` (where the envelope is formatted) plus a new compaction policy in `src/index.ts`'s lifecycle hooks.

## Workflow validation observations

What we learned about §11's canonical chains by running them on real work this session:

- **Revise loops always converged ≤ iter 1.** Every `ensemble_send` corrective in this session resolved on the first revision. The cap of 3 iterations in §11 is well above the observed median.
- **Every reviewer gate caught a real bug the producer didn't see.** `critic-4zl9` caught a Med regression in `b3a50bd` (skipped `findSessionFile` discovery breaking `ensemble_send` to force-killed runs). `critic-ke7w` caught a Med test gap on §10 closer-trigger rows. `oracle-f0vt` caught 6 F-edits + 4 A-additions on `design.md` before any builder touched it.
- **The two failure modes that did hit were detected by the workflow itself.** Filtered-context confusion (items 1) and parallel-builder git collision (item 2) both surfaced via the build → review loop and were recovered before they corrupted ship state. Without the critic gate, both would have shipped silently.
- **Cost ratio favors gates.** Critic spawns averaged $1–3 each. Each one prevented a defect that would have cost $3–7 to fix downstream (revise loop) or much more (history surgery, in one case). Gates pay for themselves.
- **Oracle gating the design itself was high-value.** Running `oracle` on `design.md` before any builder saw it caught 16 issues. The pattern "design → oracle → designer revises → builder sees clean spec" cost ~$8 and saved the implementation slice from at least three iterations.
