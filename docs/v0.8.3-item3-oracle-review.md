The design does not enumerate which keys live in `FOOTER_BINDINGS` — Esc/Tab/↑↓/s/c/t/k all currently flow through `handleInput`. Planner must verify Esc → close-overlay survives Slice 9's refactor. **Recommendation: pin in Slice 9 acceptance test that Esc still calls `onClose`.**

The factory's documented invariant (`src/focused-overlay-factory.ts`, map.md per-file purpose) is **must NOT register a RunRegistry listener** (one-leak-per-overlay-open bug). Slice 11 sidesteps by registering the listener in `installFocusedOverlayShortcut` (session-scoped, idempotent unsub) — that's the right place. ✅ But planner should pin a test that re-opening the overlay does not stack listeners.

## 7. Risks the design under-states

The design's own §8 covers: result-flag heuristic, Slice 11 cycling responsiveness, theme-slot portability, `keyHint` signature gap, throttle vs outcome line. Generally agree. Adding:

- **D4 step 3 vs §8 Box deferral is an internal contradiction.** D4 says "replace [empty-state] with a `border:borderMuted` framed `Box` from pi-tui" (`design.md:73`). §8's "Defer to v0.9" says "`Box`/`DynamicBorder`-based overlay frame" deferred. Slice 10 (O4 empty-state) needs to follow §8's deferral path (rulers + Spacer + dim) and drop the Box prescription, OR the deferral needs to be narrowed to "Box for the *transcript viewport*, not the empty state." **Planner cannot ship Slice 10 until this is resolved.**
- **Throttle interaction (§7 / §8) — half-second flash is borderline acceptable.** Designed claim "users see call-without-outcome for ≤500ms then outcome appears" depends on the trailing edge actually firing. If a turn ends *exactly* on the leading edge, the outcome may not paint until the next event. Lightweight mitigation: bypass the throttle for `toolResult` events specifically (event-handler classifies; the foreground throttle wrapper can pass-through on `is_error || hasResult`). Not a blocker but worth a one-line mention in Slice 2 acceptance.
- **Slice 0 blast radius into event-handler.ts.** Design correctly bounds it to the `formatToolCallShort` consolidation; `event-handler.ts:84` (where `lastToolCall` is set) keeps using the wrapper that re-adds `$ ` + `shortenPath`. The widget's `lastToolCall` text shape is unchanged. Bounded correctly. ✅
- **Snapshot thrash across slices 1–5.** Design's Option A claim is "no ANSI in renderTranscript snapshots" — true. But the *plain text* shape changes in Slices 1, 2, 3, 4, 5 — every existing snapshot covering tool-call lines, thinking gutter, turn separators, and header status will need updating. Inspector counts: 19 in `tests/transcript.test.ts` + 15 in `foreground-stream.test.ts`. Realistically 8–12 of those need touch-up across the 5 slices. Not catastrophic but worth flagging that "snapshots stay passing" ≠ "snapshots untouched."
- **Slice 7 component-layer test stub.** Design proposes stubbing `theme.fg` with sentinel `[ACCENT]…[/]`. Inspector did not confirm whether `theme.fg` is monkeypatchable or whether `Theme` is a class with private state. Planner should verify before opening Slice 7 (5-min spike).

## 8. Recommendation to planner

### Amendments to apply (planner takes design + these)

1. **Resolve D4 #3 / §8 Box contradiction.** Pick: (a) drop Box from D4 #3, use rulers + Spacer + `theme.fg("dim")` for empty state (consistent with §8 deferral), or (b) narrow §8 deferral to "Box for the transcript viewport only." Recommend (a) — it's the smaller, less-spike-required path and the design already prescribes the components for it.
2. **Fix the test count claim** (`design.md:88`): "50+ existing tests" → "19 tests in `tests/transcript.test.ts` (plus 15 in `foreground-stream.test.ts`)." Doesn't change the decision but cleans up the rationale paragraph.
3. **Slice 2 types.ts hedge.** Plan should explicitly verify `AgentMessage.is_error` is already exposed by `@earendil-works/pi-agent-core` before opening the slice; if so, drop `src/types.ts` from the file list.

### Load-bearing slice ordering (do not reorder)

- **0 first.** Slices 1–5 import from `src/tool-summary.ts` and `src/status-glyph.ts`; reordering creates a mid-tree merge.
- **6 before 7.** `transcript-classify.ts` must exist before `transcript-style.ts` consumes it.
- **7 before 8/9/10/11.** All overlay slices depend on the styling layer being live (or they hard-code colors and need re-touching).
- **12 last.** Backlog-close cites SHAs.

### Flexible ordering

- **Slices 1, 2, 3, 4, 5** can land in any order or in parallel. Each has independent test files.
- **Slices 8, 9, 10, 11** can land in any order after 7 (they touch `focused-stream-overlay.ts` but in different methods/regions). If two run in parallel, expect one trivial merge in the imports block.

### Slices recommended for further split

- **Slice 5 (live activity).** If the design's `idle ≥5s` predicate needs `Run.lastEventAt`, split into:
  - **5a.** Add `Run.lastEventAt` written by `event-handler.ts` on every `messages.push`. (S, no UX change.)
  - **5b.** `renderHeader` consumes 5a + `run.messages.at(-1)` to derive activity. (S.)
  Planner should make this split contingent on whether the field already exists.
- All other slices stay as-is.

### Things planner should specifically watch for

- Slice 9 acceptance must pin Esc → `onClose` survives footer-binding refactor.
- Slice 11 acceptance must pin "re-opening overlay does not stack registry listeners" (factory's documented invariant).
- Slice 3 must specify whether foreground stream renders the thinking-summary line unconditionally or honors a flag (recommend unconditionally — there's no model to consult).
- Slice 7 should include a 5-min spike confirming `theme.fg` is stub-compatible for component-layer tests.
