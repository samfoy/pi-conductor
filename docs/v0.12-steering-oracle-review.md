# v0.12 — Steering — Oracle goal-alignment review

Gate review of the four locked decisions (Q1/Q2/Q6/Q9) against the PRD,
the inspector map, and the existing v0.5 / v0.9.x / v0.10 contracts.
This is a goal-alignment check, not a design.

---

## 1. Verdict

**APPROVE WITH FIXES.**

The locks are consistent with PRD principles and don't violate any
decision-log entry. Two items must be settled before the designer
locks scope, and one should be tightened during design:

1. **Cascade-shape asymmetry vs `kill_on_stall`.** Lock 1 admits a
   persona-frontmatter layer for `steerable`; `kill_on_stall`
   explicitly *deferred* that layer (`PRD.md:517` — "persona-frontmatter
   layer deferred"). Designer must either (a) ship `steerable` with the
   same deferred shape (cascade: per-call > project > user > default;
   no frontmatter layer in v0.12), or (b) explicitly upgrade both
   cascades to include frontmatter in v0.12 with a one-line PRD entry.
   Mixed shapes will accrete drift in `effectiveConfig` (`watchdog.ts:240`).

2. **Lock 4 phrasing is too broad.** "Bump `lastEventAt` on every
   inbound RPC line" includes `extension_ui_request` lines, which are
   *blocking on the host*, not progress signals. Bumping on those
   masks a genuine class of stall (sub-agent waiting on an unanswered
   UI request). Narrow to: `response` ack lines and `message_end` /
   `tool_result_end` (already bumped at `event-handler.ts:65/96`).
   Treat `extension_ui_request` as "host owes a reply" and surface it
   separately.

3. **Crash-recovery semantics for live RPC sub-agents.** The
   v0.9.x reconcile (`PRD.md:521`) uses `kill(pid, 0)` for liveness;
   RPC sub-agents stay alive *after* their turn, so a conductor crash
   leaves an orphaned-but-alive RPC subprocess whose stdin pipe the
   reborn conductor can't recover. Inspector map §6 risk 4 names the
   forced-terminate tangle but not this case. Designer must declare:
   on readopt, do we (a) terminate orphaned RPC runs (consistent with
   "lost the wire, lose the run"), or (b) keep the record but mark
   un-steerable? This affects the D8 two-gate active-run safety
   contract (`PRD.md:606`).

## 2. Alignment with PRD principles

- **Q5-deferred / "no speculative knobs without field data"
  (`PRD.md:616`).** Q5 covered the `watchdog_silence` per-spawn
  opt-out — a tuning knob. Lock 1 (`steerable: true`) is the
  *feature gate* itself, not a tuning knob; without it there is no
  v0.12. Default-OFF mirrors `kill_on_stall` v0.10 Q1
  (`PRD.md:612`) — same shape, same risk posture, same "no field
  data justifies flipping default" reasoning. **Compatible.**

- **`kill_on_stall` cascade (`PRD.md:517`, `src/watchdog.ts:240`).**
  Adjacent and overlapping in surface. The two cascades should *share
  layers* (resolution helper) but the *flag values* are independent
  — a sub-agent can be steerable without being kill-on-stall. See §1
  fix #1 for the shape decision. Not a conflict, an asymmetry.

- **Q3 reversal — non-disruptive default (`PRD.md:614`).** Lock 2
  defaults `streaming_behavior: "auto"` to `follow_up` for running
  runs. `follow_up` queues at the next turn boundary; `steer`
  interrupts mid-turn. Defaulting to the non-interrupting verb is
  *exactly* the same posture as the Q3 reversal — soft advisory
  ships `triggerTurn: false` precisely to avoid yanking the
  conductor mid-edit. **Strongly coherent.** Document the parallel
  in the design rationale.

- **v0.9.x post-startup reconcile (D1–D5).** D5's principle was
  "reuse `killed` RunStatus with `errorMessage` discriminator — no
  new RunStatus, avoid consumer-audit churn" (`PRD.md:525`). RPC-mode
  runs produce the same terminal statuses (`running` → `completed` /
  `failed` / `killed` / `timeout`). **No new RunStatus needed.**
  Caveat: see §1 fix #3 — the *transition rules* into those statuses
  change for RPC because the subprocess outlives the turn. The
  classifier (`classifyRecord` per `PRD.md:522`) shouldn't need new
  branches if we land on policy (a).

## 3. AGENTS.md / TDD compliance plan (sketch)

**Unit-level (must pin each lock):**
- Lock 1 cascade resolution: per-call > persona > built-in default
  `false`. Mirror `tests/watchdog-config.test.ts` shape if it exists,
  else mirror `tests/personas.test.ts:` audit-style invariants.
- Lock 2 decision matrix: 2×4 grid of `{running, terminal} ×
  {auto, steer, follow_up, resume}`, including invalid pairs (e.g.
  `steer` on a terminal run → reject with named error).
- Lock 4: `lastEventAt` bump on `response` lines specifically;
  *no* bump on `extension_ui_request` (per fix #2). Pinned by a
  unit test feeding fake JSONL lines into the line handler and
  asserting `Run.lastEventAt` deltas.
- Existing `tests/ensemble-send.test.ts:160` ("rejects a running
  sub-agent") and `:179` ("rejects a paused sub-agent") will
  *change behavior* — the running case becomes accept-and-route;
  the paused case is still a designer call (inspector §5 q7).
  Don't delete these tests, mutate them.

**Integration-level:**
- A live `pi --mode rpc` smoke test gated on
  `CONDUCTOR_LIVE_TESTS=1` is **strongly recommended.** The dist
  surface (`rpc-types.d.ts`, `rpc-mode.js`) is unversioned vs
  pi-conductor; the only way to catch a drift in `RpcCommand` /
  `RpcResponse` shapes is a real round-trip. One `inspector`
  spawn + one `steer` + one `agent_end` is enough. Pin by event-
  type checklist, not by exact bytes.
- Watchdog × steer interaction: integration test under
  `CONDUCTOR_LIVE_TESTS=1` that issues a steer and asserts
  `lastEventAt` bumps on the `response` line within ~1 RPC RTT,
  preventing a soft stall from latching.

## 4. Drift detector

- **Open question #6 (pause/resume; `PRD.md:545`).** Resolved in v0.5
  via SIGSTOP/SIGCONT. Inspector §5 q7 raises the steer-while-paused
  interaction; the four locks are silent on it. **Not a lock
  conflict** — designer must pick: forbid steer while paused (clean,
  consistent with "kill = gone" mental model in Lock 3), buffer in
  conductor state, or trust the kernel pipe. Recommend forbid-while-
  paused for v0.12 — keeps the state space small.

- **v0.5 `ensemble_send` schema rough edges.** The v0.5 send tool
  already accepts `kill_on_stall` and `stall_threshold_seconds` as
  per-call overrides (`tools.ts:481–495`, per inspector §4). Adding
  `streaming_behavior` is a *parallel* override slot, not a
  collision. **No rough edges inherited.** The schema does *not*
  carry per-call `steerable` (Lock 1 is a spawn-time decision; you
  can't make a non-steerable run steerable mid-flight). Confirm
  this is the designer's read.

- **v0.10+ planned: sub-agent → conductor mid-run messaging
  (`PRD.md:536`).** This is the reverse direction. Inspector
  positions v0.12 as conductor → sub-agent. Is this the right
  ordering? **Yes — HOLD.** Conductor → sub-agent is implementable
  with the existing pi-coding-agent RPC surface (no upstream
  changes). Sub-agent → conductor needs an intercom-style
  affordance pi doesn't expose today. Land the cheap half first;
  closed-loop becomes a v2 milestone.

- **D8 two-gate active-run safety (`PRD.md:606`).** Reclaim-eligibility
  requires both disk-terminal AND in-memory-terminal. RPC mode
  doesn't add new RunStatus, so D8 still holds — but see §1 fix #3
  for the orphan-readopt edge case where pid is *alive* but
  unreachable.

## 5. Risk-mitigation gates (inspector map §6)

| # | Risk | Locks address? | Pre-design or in-design? |
|---|---|---|---|
| 1 | Stdin race / partial-event tearing | No | **Pre-design.** Single-writer queue must be spec'd before designer locks scope. Lock 2's overload doesn't help if writes interleave. |
| 2 | `extension_ui_request` deadlock | No | **Pre-design.** Locks 1–4 don't pick a policy. Default-cancel-with-`{cancelled:true}` vs proxy-to-user is a UX-shaping decision; designer can't draft the tool surface without it. |
| 3 | Watchdog confounding | **Yes (Lock 4).** | In-design refinement: narrow per fix #2. |
| 4 | Forced-terminate path tangles | **Yes (Lock 3).** | In-design. SIGTERM-only keeps the existing `applyCloseHandlerTerminal` race-fix and W7 idempotency intact (`runs.ts:1296`/`:1386`/`:1454`). Designer adds nothing here. |
| 5 | Session-file divergence | Implicit | In-design. Locks don't expose `switch_session`/`fork`/`clone`/`set_session_name` — designer should make this *explicit* in the v0.12 scope cut. |

**Pre-design blockers: risks 1 and 2.** Designer should not start
without a stance on those two. They shape the wire format and the
tool surface respectively.

## 6. Open questions for the designer

(Three additions — inspector map's §5 q1–q10 are already in scope.)

1. **RPC schema versioning.** `[pi-dist] modes/rpc/rpc-types.d.ts`
   is unversioned vs pi-conductor's pinned pi-coding-agent dep.
   Pi 0.x bumps may rename commands or add required fields with no
   compile-time signal. Vendor a copy of the relevant subset and
   assert at runtime via a shape check on first `get_state`? Or
   trust pi-coding-agent semver and ship a `/conductor doctor`
   probe? The cost of "vendor + assert" is real (drift between
   vendored copy and live pi) but the cost of silent breakage is
   higher in autonomous chains.

2. **Telemetry / observability in `record.json`.** Should the
   designer surface `streaming_behavior` chosen for each
   `ensemble_send` call, plus the `response.success` ack, into
   `record.json` (or a sibling `rpc-events.jsonl`)? Useful for
   the same field-data argument that drove `kill_on_stall`'s
   default-OFF — we want to know how often `auto` resolves to
   `follow_up` vs `resume` vs (future) `steer` before flipping
   any defaults.

3. **Migration / readopt of in-flight runs at extension load.**
   The post-startup reconcile (`PRD.md:521`) currently maps a
   live pid → readopt branch. RPC-mode adds a new failure mode:
   pid is alive, but the conductor doesn't own the stdin pipe
   anymore (the previous parent process closed it). Should
   reconcile (a) terminate any `running` RPC record on
   conductor cold-start (consistent with print-mode's "we never
   recover an in-flight stream"), or (b) carve a "readopt-
   readonly" state that lets you watch but not steer? Option (a)
   is consistent with v0.9.x D1–D5 (no new RunStatus); option
   (b) is more useful but adds a substate inside `running`.

## Confidence

**High.** The locks are well-anchored in the v0.10 / v0.9.x /
Q3-reversal precedent. The two pre-design blockers (stdin writer
queue, `extension_ui_request` policy) are concrete and small;
neither requires re-litigating a lock. The cascade-asymmetry fix
is editorial. Designer can start as soon as fixes #1–#3 in §1 are
declared.
