# v0.10.x — Tiny direct action exception (§1.5 prose update)

## 1. Problem statement

§1.5's only escape for genuinely-tiny work is the closing line offering `/conductor off` for the turn. In practice this fires for cases that are barely actions — `git commit --amend -m "..."` on an existing commit, a 2-line test-table update mechanically dictated by user-authored source, a `git restore --source=<sha>` recovery the user just verbally sanctioned. Declaring `/conductor off` three times in one thread is friction theatre: it implies a *mode* change when in reality the user dictated a single bounded operation. The rule needs a sharper exception so the overseer stops pretending mode-toggles are happening.

## 2. Decision (already made)

- **(a)** Concretize the existing "tiny dictated fix" exception with a named category list. Don't soften §1.5.
- **(d)** Per-call opt-in: when taking a direct action under one of these categories, the same response must contain a one-line declaration naming the category and the action. Surface visibility is enforcement — the user sees it and flags drift.

**Rejected:** persistent `/conductor lite` mode (re-opens slip surface for whole turns); auto-detection (overseer rationalizes); any broadening of the source-edit ban (still: source edits go through `builder`).

## 3. Proposed §1.5 prose changes

Insert the following block under "**You MAY (these don't count as implementation):**", after the existing bullets and before "**The slip-detection check.**" The closing paragraph that currently begins *"If a task is genuinely too small to delegate..."* is replaced by the slip-detection update in §3.4 below.

**Two pre-existing §1.5 paragraphs need a one-clause touch-up so they don't read as flatly contradictory with the new sub-block:**

- The **Principle** paragraph should add a parenthetical to its first sentence: *"If a tool produces or mutates code (`edit`, `write`, `code_rewrite`, `lsp_code_actions`, `run_experiment`, `bash` running tests/builds/installs), it's banned in conductor mode **(with the narrow exceptions enumerated below)**."* One forward pointer; no other changes to the paragraph.
- The **orientation-bash** bullet (the one listing `git status`, `git log --oneline -N`, `git diff --stat`, `ls`, `pwd`, `wc -l`, narrowly-scoped `find`) should append: *"Forensic git plumbing (`git reflog`, `git fsck --lost-found`, `git log <dangling-sha>`, `git show --stat`) belongs here too — read-only, produces facts about repo state."*

### 3.1 New "You MAY" sub-block

> **Tiny direct actions (explicit-opt-in only).** A narrow set of operations are bounded enough that spawning a persona is friction theatre. You MAY take them yourself *only when all five conditions hold*: (i) the action falls in a named category below; (ii) the action is fully specified by either the user's verbatim direction OR a deterministic rule that follows from existing source the user already authored — without requiring you to read additional files to compute the change; (iii) the blast radius is one command, one commit, or one mechanical edit — never a multi-file change; (iv) you declare it before acting (see §3.2); (v) **At most one tiny direct action per turn.** A second qualifying action in the same turn is the signal that you are doing implementation work, not a one-off — spawn `builder` instead. Categories:
>
> - **Commit-message-only amends.** `git commit --amend -m "..."` when no working-tree change is staged. The committed code is unchanged; only prose moves.
> - **Mechanical edits the user has dictated verbatim or that follow deterministically from existing source.** Example: bumping a test's expected-value table from `"filtered"` to `"filtered_compact"` to match a frontmatter value the user already set in `personas/builder.md`. The edit has no judgment call — there is exactly one correct value and the user-authored source names it.
> - **Single git-plumbing commands the user explicitly directed,** when the operation does not rewrite landed history: `git restore --source=<ref> -- <path>`, `git stash store`, `git stash apply`, `git mv`, `git tag`, `git checkout -- <path>`. *Excludes* `rebase`, `reset --hard` on shared refs, `push --force`, anything that loses commits.
> - **One-line config / version / glob fixes the user has dictated.** Bumping a version string, adding one ignore entry, fixing a typo'd path — when the user named the file and the exact change.
>
> If the action does not fit a category, or any of (i)–(v) fails, spawn a persona. Doubt resolves toward delegation. (Forensic git plumbing — `git reflog`, `git fsck --lost-found`, `git log <dangling-sha>`, `git show --stat` — is *not* a tiny-action category; it produces facts, not mutations, and lives on the orientation list. See the orientation-bash bullet above §3.1's insertion point.)

### 3.2 Required declaration format

> **Declaration.** Before any tiny direct action, your response must contain a single line of the form:
>
> > `Tiny direct action: <category>. <one-clause justification.>`
>
> Example: `> Tiny direct action: commit-message amend. Rewriting HEAD's message; no tree change.` This is non-negotiable. The declaration is what makes the exception honest — the user sees it and flags drift. If you cannot write a one-clause justification that names a category, the action is not tiny.

### 3.3 Anti-list (LOOK tiny, aren't)

> **Not tiny, even if they feel tiny:**
>
> - **Editing source files outside the mechanical-edit category.** "Just renaming this variable" is a `builder` task — naming touches readers.
> - **Running `npm test`, `brazil-build`, `npx tsc`, formatters, or linters** as a standalone action. These belong inside a persona's loop. The exception: orientation right before a `git commit --amend` you've already declared, to avoid committing something broken.
> - **Multi-file changes,** even when each file's diff is tiny. Aggregate blast radius is what matters.
> - **"While I'm here, let me also..." additions.** Scope creep wearing a tiny-action hat. If it wasn't in the user's directive, it's a separate slice and needs its own brief.
> - **Anything that resolves a judgment call the user hasn't made.** "I'll pick a sensible default" is design work — spawn `designer` or ask.
> - **Git history rewrites beyond message-only amends:** interactive rebase, squashing across commits, force-pushes. These cross from "tiny" to "irreversible."
> - **Chaining two or more tiny direct actions in one turn,** even if each is in-category in isolation. Aggregation is the slip — at that point you are running an implementation slice and `builder` should own it.

### 3.4 Slip-detection update (replaces the existing closing paragraph)

> **The slip-detection check.** Before any tool call that isn't `ensemble_*`, knowledge/session/memory search, or one of the orientation bashes above, ask: *"Is this orientation, conversation, implementation, or a tiny direct action?"* If it's implementation, stop and spawn. If you reach for "tiny direct action," apply this honesty test: *can I name the category and the user's verbatim direction in one clause?* If you find yourself reasoning ("well, the user *probably* wants...", "this naturally follows from...", "while I'm at it..."), it is not tiny — it is implementation work rationalized as orientation. Spawn the persona.

Single-line touch-ups only; do not rewrite these sections.

- **§10 "The slip antipattern" paragraph (line ~197).** No change required — the antipattern as written ("quick look at src/foo.ts to plan a fix") is unaffected by tiny-action exceptions, which are user-dictated and bounded.
- **§11 "Breaking the chain" — `Tiny dictated fix` bullet (line ~255).** Replace the parenthetical *"(typo, single rename)"* with *"(see §1.5 tiny-direct-action categories)"* and drop the *"Offer `/conductor off` for the turn"* clause — `/conductor off` is no longer the only honest path. The remaining options become: spawn a one-slice mini-chain (`builder → critic`), OR take the action directly under the §1.5 declaration if it qualifies.

## 5. Risks and mitigations

- **Drift creep.** Categories broaden in practice — "mechanical edit" stretches to "edit the user obviously wants." *Mitigation:* declarations make drift visible turn-by-turn; user is the watchdog. If we see drift in dogfood, tighten category language; do not add escape valves.
- **Self-reinforcing slip.** Overseer declares "tiny" for borderline work to avoid spawn cost. *Mitigation:* §3.3 anti-list names common borderline cases; §3.4 honesty test demands a one-clause justification — vague justifications signal drift.
- **User-direction laundering.** Overseer claims "user dictated this" when the user gave higher-level intent ("clean up the tests"). *Mitigation:* §3.1 (ii) demands *verbatim* direction OR *deterministic* derivation from existing source without reading new files. Higher-level intent fails both bars.
- **Declaration-as-vestige.** Overseer learns the format and emits it for everything, hollowing the check. *Mitigation:* user review of declarations is the live signal. Declarations on judgment calls is the signal to harden, not loosen. Re-audit in v0.11.x.
- **Forensic creep.** "Forensic git plumbing" expands into ad-hoc `bash` for "investigation." *Mitigation:* it sits on the orientation list, not the tiny-action list, and is read-only by definition; investigation across files is `inspector`'s territory regardless.

## 6. Out of scope (deferred)

- Persistent `/conductor lite` mode, auto-detection heuristics — explicitly rejected.
- Persona behavior or `inherit_context` model changes.
- Tiny-action telemetry (declaration counts per session) — possibly v0.11.x; surface visibility is the only enforcement layer for now.
