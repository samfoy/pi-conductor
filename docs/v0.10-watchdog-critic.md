## 1. Verdict

**APPROVE WITH AMENDMENTS** — design is internally coherent, decisions
are concrete (not survey-only), and §2 picks the right primitive on the
right evidence. Four amendments needed before slice 1 opens; none are
structural. The `TerminationReason` type gap (A1) is the only one with
non-trivial blast radius.

Amendments:

- **A1.** Extend `TerminationReason` to include `"stalled"` (or document
  why `"killed"` is reused). Currently `runs.ts:564` allows only
  `"killed" | "timeout"`; the design's `forceTerminate(run, reason)` call
  on hard-stall transitions implicitly assumes a third reason for the
  completion-envelope glyph + history annotation. Slice 2 cannot ship
  cleanly without picking one.
- **A2.** Pin a re-check immediately before `forceTerminate` fires from
  the enforcer (`run.lastEventAt` could have advanced between the
  detector tick that produced the transition and the enforcer's await).
  Without this, a `kill_on_stall: true` spawn risks losing a run that
  just recovered. Slice 2 acceptance criterion.
- **A3.** Add a slice-1 case for the **streaming-thinking spurious-soft
  scenario** (designer/planner thinking >120s with `message_update`s
  arriving but no `message_end`). Either (a) detector also accepts
  `message_update` as a recency signal via a new `lastStreamUpdateAt`
  field, or (b) document that the watchdog *will* warn on long thinking
  streams and rely on the recovery-advisory to cancel. Recommend (b)
  with the open question kept; (a) is bigger.
- **A4.** Split slice 4 (UX) into 4a (widget + transcript-style) and 4b
  (slash + doctor). Five files in one S-sized slice is too much; this
  touches the v0.8.3 styling layer + a new slash command + a new
  history annotation in one diff.

## 3. Threshold sanity (120s soft / 600s hard / 30s grace)

### Will 120s catch the witnessed hangs?

**Yes by a wide margin.**

- `critic-yjsn`: 1327 transcript lines spanning timestamps
  1779227685784 → 1779228079907 = 394s of activity, then silence until
  manual kill at ~3236s. Soft would have fired at +120s of silence
  (~514s into the run). Hard at +600s (~994s). User would have had a
  warning ≥45 min before they noticed manually.
- `builder-z98m`: timestamps span 1779294809171 → 1779295234729 = 425s
  of activity then silence until manual kill at ~810s. Same shape.
- The orchestrator's 60s `npm test` hang in this session was
  bash-blocked (closing-quote issue). Soft would have fired at 120s.

### Will 120s false-positive on legitimate work?

**Risk is real for the workloads the user already runs in this repo.**
Concrete cases stress-tested:

| Workload | Output cadence | Likely result |
|---|---|---|
| `npm test` in this repo (~5–7s wall, dotted reporter) | Continuous | Won't stall |
| `brazil-build` (per AGENTS.md, "build/compile after making changes") | Long phases between log lines; AGENTS.md notes "takes minutes for some packages" | **Likely false-positive** mid-phase |
| `npm install` cold (no node_modules) | Silent for 60–300s during dependency tree build | **Likely false-positive** |
| `ada credentials update --conduit-…` (per AGENTS.md credential workflow) | Silent network call; 30–180s on slow MPA | **Borderline** |
| `mwinit -o` | Silent until the auth flow completes | **Borderline** |
| Skill that does `sleep 90` or polls a long-running process | Silent by design | **Will stall once** |
| `cdk synth` (used in autoresearch flows per memory) | Multi-second silent phases | Probably won't stall but tight |

The design's mitigation (R1) is correct in spirit — `kill_on_stall: false`
default makes these advisories not catastrophic — but the cry-wolf risk
is severely under-stated. Spawning a builder for any Brazil work in
this org will routinely tickle 120s silence. After the third spurious
advisory, the conductor learns to ignore them, and the watchdog's
signal-to-noise collapses.

### Mitigations the planner should pin

- **Slice 3 must ship a defaults-doc** with concrete per-workload
  override recipes. Specifically: `stall_threshold_seconds: 600` for
  any spawn whose persona's expected workload includes `brazil-build`,
  `npm install`, AWS auth flows, or polling skills.
- **Consider raising soft default to 180s.** All three witnessed cases
  were >5 min; 180s still catches them ≥3 min before completion-envelope
  cost amortizes a kill. Costs ≤30s of detection lag.
- **Consider raising graceSeconds to 60s.** A fresh spawn with
  `inherit_skills: true` (just landed in `c19950a`) loads the user's
  skill catalog at boot; on this user's machine that catalog is non-trivial
  (40+ skills per AGENTS.md). 30s grace is tight if pi parses skill
  files synchronously at startup.

### Per-spawn override exists

✅ Confirmed at design.md:117 + slice 3 at design.md:298. The override
plumbs through `ensemble_spawn` args analogous to `timeout_minutes`.

## 5. Architecture concerns

### `lastEventAt` is set in the right places

✅ Verified: `event-handler.ts:64` (message_end) and `:97`
(tool_result_end) update it. Critically, `message_update` (the
streaming-deltas event) does **NOT** update it — design.md:30 names
this as a feature. Confirmed by reading the event-handler: there's no
`message_update` branch that touches `lastEventAt`.

### Streaming-thinking gap (the case the design dismisses)

A sub-agent in a long thinking stream (e.g. designer-w2a5 in the v0.9
chain produced 500+ MB transcripts; planner-pnq7 similar) emits
hundreds of `message_update` events but no `message_end` until
streaming completes. If a thinking phase lasts >120s with no
intermediate tool calls, the detector will see stale `lastEventAt`
and fire a spurious soft transition.

The design names this in non-goals (design.md:34) but does NOT pin a
test or mitigation. **Amendment A3** above. Either accept the
spurious-soft advisories during long thinking (and document) or extend
detector to consume `message_update` ticks.

### Race between detector tick and event-handler

Detector tick at T=120.0s; at T=120.4s a `tool_result_end` arrives;
event-handler sets `lastEventAt=120.4s`; tick callback (still
running) calls `enforcer.forceTerminate(run)` based on the stale
snapshot it captured at T=120.0s.

**Real risk under `kill_on_stall: true`.** The design does not specify
an enforcer-side recheck. **Amendment A2** — slice 2 must re-read
`run.lastEventAt` in the enforcer immediately before invoking
`forceTerminate`, and abort the kill if it advanced past the threshold.

### Memory/perf

`Map<runId, WatchdogState>` per parent. Even at 100+ historical runs,
GC's terminal-state cleanup means the live working set is bounded by
`maxConcurrent` (default 4). Tick cost is O(active) arithmetic.
Negligible. ✅

### `TerminationReason` type gap (load-bearing)

`runs.ts:564` exports `type TerminationReason = "killed" | "timeout"`.
Design's `forceTerminate(run, reason)` for hard-stall implicitly wants
a third reason `"stalled"` so the completion envelope renders the
right glyph and `/conductor history` annotates the kill. Currently the
design would have to reuse `"killed"`, indistinguishable from
user-initiated `ensemble_kill`. **Amendment A1** — slice 1 (types) or
slice 2 (enforcer integration) extends this union.

## 8. Recommendation to planner

### Critic gates

Per design §8: gates on slices 1 and 2 are **correct** — both are
load-bearing logic with mutation-witness targets pre-declared. **Add a
critic gate on slice 4a** (widget glyph rendering) — it touches the
v0.8.3 component-layer styling pattern that has its own snapshot-test
discipline; one round of critic catches drift cheaply.

Slices 3, 4b, 5 stay verifier-covered.

### Chain shape

Linear, as design proposes:

```
designer (done) → critic (this doc) → planner →
  builder ⇄ critic (slice 1) →
  builder ⇄ critic (slice 2) →
  builder (slice 3) →
  builder ⇄ critic (slice 4a) →
  builder (slice 4b) →
  builder (slice 5) →
  verifier → finalizer
```

Estimated total: $30–40 (above design's $25–35 because of A1+A2+A4
amendment overhead). Still well under the v0.9 GC chain's ~$140.

### Mutation witnesses to add to slice 1's pre-declared list

- Drop the `lastStreamUpdateAt` recency check (if A3 chooses option (a))
  → killing test: `evaluateRun: long-thinking run with recent message_update returns null`.
- Drop the recheck guard in slice 2 (if A2 lands as a separate gate) →
  killing test: `Watchdog.tickOnce: stall-then-recovery between tick and kill aborts the kill`.

### Acceptance criteria for slice 1 the planner should pin

- `evaluateRun` has zero imports from `node:fs`, `node:fs/promises`,
  `child_process`, or `runs.ts` (mirrors v0.9 gc-policy purity gate).
- Mutation witnesses (the 5 in design §8 + the 1–2 above) all red on
  mutation, green post-restore.
- `Run.stalledSince` field added to types but unused in this slice
  (consumer is slice 4a; field allocation now to avoid double-dipping
  the schema).
