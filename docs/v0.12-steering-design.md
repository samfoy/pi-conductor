## 1. Goals & non-goals

### Goals

- **Close the witnessed `validateSendable` rejection.** Today
  `src/runs.ts:1161` rejects `running` runs out of hand
  (*"sub-agent ${run.id} is currently running; wait for it to
  finish before sending"*). The conductor (and user, via the
  overlay's `s` keybinding) cannot interject — even when the
  sub-agent is going off-track on a 30-minute task and a
  one-line correction would save it.
- **Keep print-mode the default.** Sub-agents not marked
  `steerable: true` continue to spawn under
  `pi --mode json -p` exactly as today. Zero regression for the
  ~15 personas that don't need steering (`oracle`, `redteam`,
  `inspector`, `verifier`, `scribe`, etc. — already
  `inherit_context: none` per v0.8.1 audit).
- **Reuse the v0.10 cascade pattern for opt-in.** `kill_on_stall`
  cascades per-call > project config > user config > built-in
  default (`PRD.md:516`–`:517`); persona-frontmatter layer
  deferred. `steerable: true` mirrors that exactly (oracle fix #1
  keeps the two cascades isomorphic so a future PRD entry can
  upgrade them together).
- **Clean default for the new send-mode arg.** `streaming_behavior:
  "auto"` resolves to the safe choice: `follow_up` (queue) for
  running runs, `resume` for terminal runs. Explicit `"steer"`
  stays available for users / conductors that genuinely want to
  interrupt the current turn.
- **Single-writer stdin invariant.** Per-run JSON-line writer queue
  with framing identical to pi's `serializeJsonLine`
  (`[pi-dist] modes/rpc/jsonl.d.ts`). No raw `proc.stdin.write`
  from anywhere else.
- **Watchdog correctness preserved.** Bump `Run.lastEventAt` on
  every inbound RPC stdout line — including `response`,
  `extension_ui_request`, and any new RPC-mode-only line — not
  just `message_end` / `tool_result_end`. Otherwise the
  conductor's *own* steer can trip a soft-stall (Risk #3 below;
  PRD `:614` reversal shows the team's low tolerance for
  spurious stall events).

### Non-goals

- **Replacing print-mode universally.** v0.12 keeps both modes
  alive. No persona is forced to RPC.
- **`ensemble_abort` (RPC graceful turn-cancel).** Out of scope.
  Listed in §9 as a v2 item under PRD §"v0.10+ planned".
  `ensemble_kill` stays SIGTERM ("kill = gone"). Adding a
  middle verb confuses the LLM's mental model. Build on demand.
- **Auto-routing of `extension_ui_request` to the conductor's
  TUI.** Out of scope. v0.12 forbids steerable personas from
  using `ctx.ui.*` extensions. Soft-fail with a startup
  diagnostic. (See §4.2.) A future v0.13+ may proxy these
  through the focused-stream overlay.
- **Sub-agent → conductor mid-run messaging.** Listed in PRD
  §"v0.10+ planned" as a distinct line item ("intercom-shaped").
  v0.12 is strictly conductor → sub-agent. Different shape,
  different surface (LLM tool emitted *from inside* the
  sub-agent), different design doc.
- **Exposing `switch_session` / `fork` / `clone` /
  `set_session_name` / mid-process `compact`.** RPC supports
  them; we do not. v0.12 = one subprocess = one session.
  Defer.
- **Replacing or weakening the v0.9 GC contract.** Steerable
  runs are still subject to cold-archive + delete on the same
  TTL ladder. RPC subprocess survives across turns but does NOT
  survive across `session_start` boundaries — a re-`ensemble_send`
  on a terminal RPC run goes through the resume-via-`pi --session`
  path exactly like a print-mode run, not via the original
  long-lived subprocess (which is gone). Pinned in §4.4.
- **Phase-aware default for `steerable`.** No persona ships
  `steerable: true` by default. Decision rests with the
  spawner. Same posture as `kill_on_stall` (PRD `:611` Q1
  default OFF).

## 2. Problem statement

### The witnessed gap

`src/runs.ts:1161` (`validateSendable`):

```
if (run.status === "running") {
  return {
    ok: false,
    reason: `sub-agent ${run.id} is currently running; wait for it to finish before sending.`,
  };
}
```

Pinned by `tests/ensemble-send.test.ts:160` (*"ensemble_send
rejects a running sub-agent (busy)"*) and the LLM-tool guidance
block at `src/tools.ts:451` (*"running, paused, and queued
sub-agents are rejected"*). All three of:

- `ensemble_send` (LLM tool, `src/tools.ts:520`)
- The TUI overlay's `s` keybinding (`src/focused-stream-overlay.ts:228`–`:246` → `promptAndSendToRun` → `executePromptAndSend` at `src/prompt-and-send.ts:82`)
- The `/conductor send` slash command (does NOT exist today; `src/commands.ts` has no `send`/`steer` subcommand — see §4.8)

go through `validateSendable` and bail on running runs. So a
running sub-agent is genuinely uninterruptible. The user can
watch it (overlay), pause it (SIGSTOP), or kill it (SIGTERM) —
but cannot say "stop, you're going off track, do this instead."

### Why conductor → sub-agent is the right *next* step

PRD §"v0.10+ planned" lists:

> - Sub-agent → conductor mid-run messaging (analogous to pi-intercom).

That is the **opposite** direction (sub-agent emits a message to
the conductor mid-run). It needs a tool surface *inside* the
sub-agent, a routing layer, and conductor-side handling. It's
its own design doc.

The conductor → sub-agent direction is far cheaper:

1. Pi already exposes a stable RPC protocol on stdin. `steer` /
   `follow_up` / `prompt(streamingBehavior)` / `abort` are
   first-class commands documented at
   `[pi-dist] modes/rpc/rpc-types.d.ts:13`–`:113`.
2. The conductor already owns each sub-agent's `proc` handle
   (`src/runs.ts:949`).
3. The conductor's event seam (`event-handler.ts:36`) already
   parses JSON lines from the child's stdout — adding two new
   line types (`response`, `extension_ui_request`) is additive.
4. Print-mode keeps working unchanged for the steerable=false
   majority. Existing tests (1178 green, 1184 total) keep
   passing without modification.

### Why this is the right milestone now

- All three witnessed builder cascades (`builder-z98m`, the
  recent `builder-p66e` refusal cascade addressed by v0.11
  `filtered_compact`, and the W7 `ensemble-send.test.ts:160`
  pin) reduce in part to "the parent could not interject mid-run
  to redirect a misbehaving sub-agent." The v0.11 hook gate
  catches *post-completion* (a critic flunks the build), but if
  the builder is *currently* running a 20-minute task on a wrong
  branch the hook fires far too late.
- The user has explicitly listed steering as the next milestone.
- Pi-coding-agent's RPC protocol is already there. We're not
  asking pi to ship anything new; we're using existing surface.

### Q1 — Opt-in granularity (revised — oracle fix #1)

**Per-spawn `steerable: true` LLM arg only. Cascade: per-call
`ensemble_spawn` arg > project config > user config > built-in
default `false`. Persona-frontmatter layer DEFERRED.** Mirrors
v0.10 `kill_on_stall`'s deferred shape exactly
(`PRD.md:517` — *"persona-frontmatter layer deferred"*).
Sub-agents NOT marked steerable continue to use print-mode
`pi --mode json -p` exactly as today (zero regression).

Why the frontmatter layer goes back on the shelf: oracle review
flagged the asymmetry — v0.10 `kill_on_stall` and v0.12
`steerable` are adjacent and overlapping in surface, and the
two cascades should stay isomorphic so a future PRD entry can
upgrade them together. Adding the persona layer here
unilaterally would force divergent maintenance and constitute a
speculative knob (`PRD.md:616` Q5-deferred pattern). Persona
frontmatter `steerable: true` parsing lands later, gated on
field-data justification.

Citations: `kill_on_stall` cascade at `PRD.md:516`–`:517`
(per-spawn args on `ensemble_spawn` and `ensemble_send`, cascade
per-call > project config > user config > built-in default;
persona-frontmatter layer deferred). Resolver shape model:
`src/watchdog.ts: resolveKillOnStall` pure function pinned by W1
mutation witness (`PRD.md:618`); v0.12's `resolveSteerable` MUST
match its signature so the two cascades share resolver shape.
Default OFF justification: PRD `:611` Q1 — *"No autonomous-chain
dogfood data yet; advisory-only by default avoids surprising
kills on legitimately slow workloads."* Same posture for
`steerable`.

### Q2 — Tool surface

**Overload `ensemble_send` with optional `streaming_behavior:
"auto" | "steer" | "follow_up" | "resume"`. Default `"auto"`:**
- run is `running` (RPC mode) → `follow_up` (queue, safe)
- run is terminal (any mode) → `resume` (existing behaviour, spawn fresh `pi --session`)
- run is `running` (print mode) → reject with helpful message ("this run is not steerable; mark steerable: true at spawn")

**Explicit `"steer"` available for interrupt; explicit
`"follow_up"` for queue; explicit `"resume"` for terminal-runs
(no auto-detect needed).**

Citations: `ensemble_send` rejects all running runs today at
`src/runs.ts:1161`, `src/tools.ts:451`, and
`tests/ensemble-send.test.ts:160`. Pi RPC `prompt`/`steer`/`follow_up`
semantics: `[pi-dist] modes/rpc/rpc-types.d.ts:14`–`:34`;
`streamingBehavior` field on `prompt`: `:18`. "auto" defaults to
`follow_up` (not `steer`) — same risk-aversion posture as PRD
`:614` Q3 reversal where soft advisories ship `triggerTurn:
false`. Queueing is the safe default; interrupting is the
explicit verb.

### Q6 — Kill verbs

**`ensemble_kill` stays SIGTERM ("kill = gone"). `ensemble_abort`
(RPC abort, cancel turn, keep subprocess) is a v2 item — call
out in §"Out of scope" and link to PRD §"v0.10+ planned" for
tracking.**

Citations: today's `forceTerminate` SIGTERM at `src/runs.ts:1438`.
RPC `abort` at `[pi-dist] modes/rpc/rpc-types.d.ts:34`,
`[pi-dist] modes/rpc/rpc-mode.js:319`–`:322`: cancels the current
LLM turn but the subprocess survives — semantically different
from `ensemble_kill`'s "gone" contract. Build-on-demand precedent:
PRD `:616` Q5 deferred `watchdog_silence` (per-spawn opt-out) on
the same logic ("no field data justifying it").

### Q9 — Watchdog policy (revised — oracle fix #2)

**Stay strict (no special steer-window threshold extension).
Engineering fix: bump `Run.lastEventAt` on `response` ack lines
(NEW) plus today's `message_end` (`event-handler.ts:65`) and
`tool_result_end` (`event-handler.ts:96`). Do NOT bump on
`extension_ui_request` or any other non-progress RPC line type.**

Why the asymmetry. `response` lines are sub-agent acknowledgements
of conductor commands (`steer`, `follow_up`, `prompt`) — concrete
evidence the subprocess parsed our line and is acting on it.
Bumping on those is correct: a steer the conductor itself just
sent must not contribute to its own soft-stall.
`extension_ui_request` lines are the opposite shape — the
sub-agent has *blocked* waiting on the host to reply. Bumping
on those would mask a genuine class of stall (sub-agent waiting
on an unanswered UI request). With v0.12's auto-cancel policy
(§4.2) the reply happens in the same tick, so in practice the
subsequent `response` line bumps `lastEventAt` anyway — but the
bump policy MUST be tied to the progress signal, not the
blocking one.

Citations: current bump sites at `src/event-handler.ts:64` and
`:97` — only `message_end` and `tool_result_end` bump today.
v0.10 reversal of analogous knob: `PRD.md:614` Q3 — the team
has low tolerance for spurious stall events. Why bump on
`response`: pi answers a `steer` command with
`{type:"response", command:"steer", success:true}`
(`[pi-dist] modes/rpc/rpc-types.d.ts:155`–`:160`); that line is
observable evidence the subprocess is alive. PRD `:618` recheck
pattern (slice 2, `b7a36a9`) handles transient false positives
but not continuous ones.

## 4. Design

### 4.1 Spawn-time mode selection

**Pure cascade resolver.** New file
`src/steerable.ts` (placement mirrors `src/watchdog.ts:240` —
`effectiveConfig` and `resolveKillOnStall` live in the same
module as the watchdog because they're tightly coupled to its
detector; here we keep the resolver in its own module because
it's the only consumer outside `runs.ts`/`tools.ts`).

**Signature isomorphic to `resolveKillOnStall`.** Oracle fix #1
requires the two cascades stay shape-equivalent so a future PRD
entry can upgrade them together. Today's actual signature at
`src/watchdog.ts:287` is a one-liner with two inputs:

```ts
export function resolveKillOnStall(
  run: Run,
  defaultKillOnStall: boolean,
): boolean {
  return run.killOnStall ?? defaultKillOnStall;
}
```

The 4-layer cascade (per-call > project > user > built-in
default) is collapsed UPSTREAM at spawn time — the per-call /
project / user values are merged onto `run.killOnStall`
(`boolean | undefined`) before the resolver ever sees them.
The resolver itself just falls back to the default when the
run's value is undefined. `resolveSteerable` matches this shape
verbatim (oracle gate 2 ADJUST: do NOT refactor
`resolveKillOnStall` upstream to add 4-layer plumbing — that
would be adjacent code churn in a feature slice; match what
exists):

```ts
// src/steerable.ts

import type { Run } from "./types.ts";

/**
 * Pure cascade resolver. Mirrors `resolveKillOnStall` shape exactly
 * (`src/watchdog.ts:287`). Layer collapse happens upstream:
 *   - per-call from ensemble_spawn LLM tool arg
 *   - project config (.pi/conductor.json)
 *   - user config (~/.pi/agent/extensions/conductor/config.json)
 * are merged onto `run.steerable` (boolean | undefined) before
 * spawn time. The resolver itself just falls back to the default
 * when run.steerable is undefined.
 *
 * NO persona-frontmatter layer (PRD.md:517 — "persona-frontmatter
 * layer deferred"). The two cascades MUST stay shape-isomorphic;
 * a future PRD entry that upgrades one upgrades both.
 *
 * Pure: no I/O, deterministic on (run, defaultSteerable). Exposed
 * for tests so the W2 mutation witness can pin the formula directly
 * (mirrors `resolveKillOnStall`'s W1 pin per `docs/wdd.md`).
 */
export function resolveSteerable(
  run: Run,
  defaultSteerable: boolean,
): boolean {
  return run.steerable ?? defaultSteerable;
}
```

**Cascade behaviour at spawn time.** The 4-layer collapse onto
`run.steerable` is the spawn pipeline's job, not the resolver's.
Higher layers shadow lower layers: per-call wins over project
wins over user wins over the built-in default. Truthy or
falsy explicit values short-circuit the cascade; only
`undefined` falls through. Slice 1 wires this in the spawn
pipeline (`src/runs.ts: spawnRun` reads the cascade inputs and
stamps `run.steerable` before invoking the resolver).

**Decision matrix (pinned for slice 1 unit tests — resolver
level only, post-collapse).** The resolver sees one input
(`run.steerable`) plus the default; it returns a boolean. The
W2 mutation witness pins the formula `run.steerable ??
defaultSteerable` per `docs/wdd.md` parallel-formula rule:

| `run.steerable` | `defaultSteerable` | Result |
|---|---|---|
| `true`      | (any)   | `true`  |
| `false`     | (any)   | `false` |
| `undefined` | `false` | `false` |
| `undefined` | `true`  | `true`  |

Unit tests for the upstream cascade-collapse (per-call > project
> user > default merging onto `run.steerable`) live in the
spawn-pipeline test file (`tests/steerable-spawn-cascade.test.ts`),
not the resolver test file. Provenance (`source: "per-call" |
"project" | "user" | "default"`) can be reconstructed from
spawn-time diagnostics if the doctor surface ever needs it; we
do NOT bake provenance into the resolver — the v0.10 cascade
doesn't, so v0.12 doesn't either.

**No `Persona.steerable` field in v0.12.** Frontmatter parsing
is NOT extended. The `Persona` interface in `src/types.ts`
stays unchanged for steering. (Adding a parsed-but-unused stub
"for future use" is exactly the speculative-knob pattern PRD
`:616` Q5 deferred — don't.)

**`ConductorConfig.defaultSteerable` field.** Per the cascade
shape, the project/user config layers need a place to land:

```ts
// src/types.ts: ConductorConfig (additive)

/**
 * v0.12: project/user-config default for ensemble_spawn's
 * steerable arg. Cascade per-call > project > user > built-in
 * default false. Mirrors WatchdogConfigDefaults.defaultKillOnStall.
 */
defaultSteerable?: boolean;
```

Project config (`.pi/conductor.json`) overrides user config
(`~/.pi/agent/extensions/conductor/config.json`); both are
optional. Built-in default is `false`.

### 4.2 RPC subprocess plumbing

**Argv composition.** `buildPiArgs` (`src/runs.ts:333`) gains a
`steerable: boolean` parameter that flips the `--mode` flag.
All other flags (`--session-dir` / `--session`, `--model`,
`--thinking`, `--append-system-prompt`, `--skill`) are
mode-agnostic per inspector map §2 ("All four flags we use
today are mode-agnostic — consumed before mode dispatch").

```ts
// src/runs.ts — new signature shape, additive
export type PiArgsOptions =
  | {
      kind: "fresh";
      sessionDir: string;
      systemPrompt: string;
      prompt: string;
      model?: string;
      thinking?: ThinkingLevel;
      skillPaths?: string[];
      steerable: boolean;  // NEW: drives --mode
    }
  | { /* "resume" variant gets the same field */ };
```

`buildPiArgs` body change:
- `steerable === false` (today's path): `["--mode", "json", "-p", ...]` (unchanged from `runs.ts:334`).
- `steerable === true`: `["--mode", "rpc", ...]`. Drop `-p`. The trailing positional prompt is **NOT auto-prompted in RPC mode** (inspector map §2 last bullet). We pass the prompt as the first explicit RPC `prompt` command on stdin once the subprocess starts (see "Initial prompt injection" below).

**Subprocess plumbing changes — `runPiSubprocess`
(`src/runs.ts:925`).** The current `spawn()` call uses
`stdio: ["ignore", "pipe", "pipe"]` (`runs.ts:935`). Steerable
runs flip stdin to `"pipe"`:

```ts
const stdin = invocation.steerable ? "pipe" : "ignore";
proc = spawn(invocation.command, invocation.args, {
  stdio: [stdin, "pipe", "pipe"],
  shell: false,
  env: buildSubagentEnv(),
});
```

**Single-writer stdin queue.** New
`src/rpc-stdin.ts`:

```ts
// src/rpc-stdin.ts

import type { Writable } from "node:stream";

/**
 * Single-writer JSON-line queue for one sub-agent's stdin. All
 * RPC commands flow through this. Absolutely no other code may
 * call `proc.stdin.write(...)` directly. Sequencing guarantee:
 * commands are framed and delivered in submit order; back-pressure
 * is awaited; partial writes are joined; framing is LF-only
 * (matches pi's serializeJsonLine — see [pi-dist]
 * modes/rpc/jsonl.d.ts: "Framing is LF-only").
 *
 * Returned promise resolves when the line is fully written to the
 * pipe (kernel acknowledges the write). It does NOT resolve when
 * the pi server has acted on the command — that's the response
 * line on stdout (see RpcResponseRouter, §4.4).
 */
export class RpcStdinQueue {
  constructor(stdin: Writable);
  /** Enqueue a command for sequenced JSON-line write. */
  enqueue(cmd: object): Promise<void>;
  /** Reject all pending writes; called from forceTerminate / close. */
  destroy(reason: string): void;
}
```

**Implementation outline (signature pinned for slice 2).** One
in-flight write at a time; new writes await the previous write's
drain. Serialise with `JSON.stringify(cmd) + "\n"`. Reject the
enqueue promise if `cmd` contains an embedded newline in any
string field after stringification (defensive — pi already
handles U+2028/U+2029 transparently per inspector map §2 framing
note, but we don't generate them). On `stdin` `error` (EPIPE,
ECANCELED) or close, drain outstanding promises with a rejection
containing the cause. This is how Q3 (steer-into-finishing race)
surfaces — see §5.

**Stdout line reader extension.** `processLine`
(`src/runs.ts:1068`) currently parses every line as an
`AgentEvent` and hands it to `applyEvent`. RPC mode emits two
new line shapes that are NOT `AgentEvent`s:

- `RpcResponse` lines: `{type: "response", command, success, ...}`
- `RpcExtensionUIRequest` lines: `{type: "extension_ui_request", id, method, ...}`

`applyEvent` must learn to dispatch by `event.type`:

```ts
// src/event-handler.ts (extended)

export function applyEvent(run: Run, event: unknown): EventEffect {
  const evt = event as { type?: string };
  if (typeof evt?.type !== "string") return { kind: "none" };

  // NEW: RPC-only line types route to the RPC routers.
  if (evt.type === "response") {
    routeRpcResponse(run, evt as RpcResponse);
    run.lastEventAt = Date.now();   // §4.7 — bumps on `response`
    return { kind: "updated" };
  }
  if (evt.type === "extension_ui_request") {
    handleExtensionUiRequest(run, evt as RpcExtensionUIRequest);
    // §4.7 — does NOT bump lastEventAt. Sub-agent is BLOCKED
    // waiting on the host's reply; bumping would mask the stall
    // class "sub-agent waiting on unanswered UI request."
    return { kind: "updated" };
  }

  // EXISTING: AgentEvent dispatch (message_update, message_end,
  // tool_result_end, turn_end, agent_end, ...).
  // ...
}
```

`routeRpcResponse` resolves correlation promises (§4.4 ack
envelope). `handleExtensionUiRequest` answers per the policy
in the next paragraph.

**`extension_ui_request` policy (v0.12) — always-cancel-and-warn.**
User-locked per oracle Risk 2 review. Every inbound
`extension_ui_request` line, regardless of `method` or
`source`, is immediately answered with the canonical
cancellation envelope from
`[pi-dist] modes/rpc/rpc-types.d.ts:376`–`:383`:
`{type: "extension_ui_response", id, cancelled: true}`. The
response is enqueued via `RpcStdinQueue.enqueue` synchronously
inside `handleExtensionUiRequest` — no diagnostic loop, no
dispatch to the conductor's UI, no per-method branching beyond
what pi's response shape requires. A one-line warning is logged
(*"sub-agent ${run.id} emitted ${method} request under
steerable=true; auto-cancelled"*); the run continues.

The sub-agent's UI extension code receives the cancellation
(`cancelled: true`) and the persona must handle it as "user
cancelled" — same shape as Esc on the parent. This is a
constraint documented in persona-author guidance: personas that
depend on `ctx.ui.confirm` / `ctx.ui.select` / `ctx.ui.input` /
`ctx.ui.editor` will see those calls return cancellation when
spawned with `steerable: true` and must fail gracefully. We do
not statically detect `ctx.ui` use. v0.13+ may proxy to the
focused-stream overlay (kept open in §9 out-of-scope).

Note on watchdog interaction: per §4.7 (oracle fix #2), the
`extension_ui_request` line itself does NOT bump `lastEventAt`
— sub-agent is blocked. The conductor's auto-cancel response
triggers the sub-agent's resumption, which produces a normal
`message_end` / `tool_result_end` shortly after, bumping
`lastEventAt` then.

**Initial prompt injection.** In RPC mode the trailing
positional argument is ignored (inspector map §2). Once
`runPiSubprocess` confirms the subprocess is up (first line
read on stdout, OR a synchronous "session loaded" RPC event —
TBD in slice 2 prototyping), the conductor sends:

```jsonl
{"id":"init-${run.id}","type":"prompt","message":"<task body from buildSubAgentPrompt>"}
```

via `RpcStdinQueue.enqueue`. Subsequent `ensemble_send` calls
issue further `{type: "prompt"|"steer"|"follow_up"}` lines.

### 4.3 `streaming_behavior: "auto"` resolver

**Pure decision matrix.** New helper in `src/runs.ts`:

```ts
// src/runs.ts (additive)

export type StreamingBehavior = "auto" | "steer" | "follow_up" | "resume";

export interface ResolvedSendStrategy {
  /** What the conductor will actually do. */
  strategy:
    | { kind: "rpc-steer" }       // RPC steer command on live subprocess
    | { kind: "rpc-follow-up" }   // RPC follow_up command on live subprocess
    | { kind: "spawn-resume" }    // pi --session <path> fresh subprocess
    | { kind: "rejected"; reason: string };
}

/**
 * Pure resolver. Pinned for slice 3 unit tests.
 *
 * Invariants:
 *   - run.status === "running"   AND run.streamingMode === "rpc":
 *       "auto"      → rpc-follow-up   (safe queue)
 *       "steer"     → rpc-steer
 *       "follow_up" → rpc-follow-up
 *       "resume"    → rejected: "...is currently running; resume is for terminal runs..."
 *
 *   - run.status === "running"   AND run.streamingMode === "print":
 *       (any)       → rejected: "...is not steerable; mark steerable: true at spawn..."
 *
 *   - run.status === "paused"    AND run.streamingMode === "rpc":
 *       (any)       → rejected: "...is paused; resume it first via /conductor resume <id>..."
 *       (locked: §4.5; mirrors today's validateSendable behaviour)
 *
 *   - run.status === "queued":
 *       (any)       → rejected: "...is queued and has not started yet..."
 *
 *   - isTerminal(run.status):
 *       "auto"      → spawn-resume
 *       "resume"    → spawn-resume
 *       "steer"     → rejected: "...has already finished; cannot steer a terminal run..."
 *       "follow_up" → rejected: same shape
 *
 *   - sessionPath missing on terminal runs:
 *       (any)       → rejected: "...has no resumable session on disk..."
 *       (mirrors today's validateSendable §1166).
 */
export function resolveSendStrategy(
  run: Run,
  behavior: StreamingBehavior = "auto",
): ResolvedSendStrategy;
```

**`validateSendable` becomes a thin shim.** It was always a
"can-I-send?" pre-check. With strategy resolution it becomes:

```ts
export function validateSendable(run: Run): { ok: true } | { ok: false; reason: string } {
  const r = resolveSendStrategy(run, "auto");
  return r.strategy.kind === "rejected"
    ? { ok: false, reason: r.strategy.reason }
    : { ok: true };
}
```

…which preserves all 6 callers (LLM tool, prompt-and-send pre-check, slash command, overlay, etc.) without changing their shape. Tests at `tests/ensemble-send.test.ts:160`/`:179` (rejection cases) need updates: the "rejects a running sub-agent (busy)" case now requires steerable=false to still reject; an additive case "accepts a running steerable sub-agent under streaming_behavior=follow_up" lands in the same file.

### 4.4 Run state-machine impact

**Two new optional fields on `Run` (`src/types.ts:191`).**

```ts
/**
 * v0.12: per-spawn steerable flag, resolved at spawn time from
 * the 4-layer cascade (per-call > project > user > built-in
 * default) collapsed onto this field, then read by
 * `resolveSteerable(run, defaultSteerable)` in src/steerable.ts
 * (one-line `run.steerable ?? defaultSteerable`, mirrors
 * `src/watchdog.ts:287` resolveKillOnStall). Drives:
 *   - --mode flag in buildPiArgs (rpc vs json -p)
 *   - stdio: ["pipe"|"ignore", "pipe", "pipe"] in spawn opts
 *   - resolveSendStrategy's reject-on-running-print-mode branch
 * Persisted on RunRecord so post-startup reconcile (v0.9.x)
 * knows which mode the orphan was launched in.
 */
steerable?: boolean;

/**
 * v0.12: derived from `steerable`. Either "print" (today's
 * default) or "rpc". Stored separately from `steerable` because
 * it captures *what mode the subprocess is actually running in*
 * — useful for diagnostics, doctor surface, and the tiny chance
 * that future cascade layers invalidate `steerable` post-spawn.
 */
streamingMode?: "print" | "rpc";
```

**No new `RunStatus` enum value.** Per PRD `:606` D1 (v0.9.x:
*"Reuse `killed` status with errorMessage prefix instead of
adding a new RunStatus."*). Steerable runs use the same
`running` status as print-mode runs. The substate
"`isStreaming` (mid-turn) vs idle" is observable from
`Run.lastEventAt` + outstanding-toolCall test exactly the same
way the watchdog already infers it (`docs/v0.10-watchdog-design.md`
§"Detection primitives"). v0.12 does NOT add a new "isStreaming"
field. If §5 Q4 (ack envelope) shows we genuinely need to
gate-keep on isStreaming we revisit.

**`RunRecord` schema bump.** Add `steerable?: boolean` and
`streamingMode?: "print" | "rpc"` to the on-disk `record.json`
shape. Optional → backward compatible with pre-v0.12 records.
`toRunRecord` (search for it in `src/runs.ts`) gains two
property copies.

**Crash recovery for orphaned RPC runs (oracle fix #3).** RPC
subprocesses outlive their turn. Conductor crashes →
orphaned-but-alive RPC subprocess. v0.9.x reconcile's
`kill(pid, 0)` says "alive" but the stdin pipe is gone (the
previous parent process closed it on exit), so the subprocess
is stranded — no conductor can ever steer it again, and pi may
or may not realise its host vanished.

**Locked policy:** terminate orphaned RPC runs on readopt.
`classifyRecord` in `src/reconcile-startup.ts` extends its
existing `readopt` branch with a `streamingMode === "rpc"`
guard. When it fires, the run is reclassified to `killed` with
`errorMessage` prefix `"orphaned: rpc-stream-detached"` and
the subprocess is SIGTERMed (the pid is still ours by
ownership; the child is the only entity holding the now-orphan
stdin/stdout). Print-mode runs continue using the existing
readopt branch unchanged — they never had a steerable channel
to detach from.

**No new `RunStatus`.** Reuses the v0.9.x D1 pattern
(`PRD.md:524` — *"reuse `killed` status with errorMessage
prefix instead of adding a new RunStatus"*). The
`"orphaned: rpc-stream-detached"` discriminator is symmetric
with the existing `"orphaned: ..."` family from v0.9.x. History
rendering's `⎘ orphaned:` glyph (PRD `:528`) renders the new
message shape without further changes.

**Archived-run compat (Q10 lock).** A v0.9-cold-archived run has
`run.proc === undefined`, `run.pid === undefined`, and (after
archive) `transcript.jsonl` deleted. Sending to such a run
takes the `spawn-resume` strategy unconditionally, regardless
of the original `streamingMode`. The fresh subprocess can
choose its own mode based on the per-call `steerable` arg and
the project/user config defaults. No "RPC mode is sticky
across archive boundaries" promise. Pinned in §4.4.

### 4.5 Pause/resume × steer queue

**Locked: forbid steers while paused. Reject with reason.**

`resolveSendStrategy` returns
`{kind: "rejected", reason: "...is paused; resume it first via /conductor resume <id>..."}`
for any non-`auto` `streaming_behavior` (and for `auto`,
which falls through to `follow_up` and is then rejected as
"can't follow_up a paused run"). Today's `validateSendable`
already does this for the resume-via-`pi --session` path; we
preserve the contract.

**Why not buffer in conductor-side state.** Two reasons:

1. **Liveness invariant.** A paused sub-agent is (per PRD
   `:298`) *"alive but not consuming tokens. Pause/resume is
   reflected in the ensemble panel as `⏸ paused`."* The user's
   model is that no input is being delivered while paused. A
   buffered steer that flushes on resume would violate that
   without a separate UX surface to inspect/cancel pending
   steers.
2. **No demand signal.** Build on demand. If field data shows
   users genuinely want a buffered queue, add it later with a
   dedicated `Run.pendingSteers: string[]` field and a
   `/conductor cancel-steer <id>` command. Not now.

**SIGSTOP × kernel pipe buffer.** Inspector map §3 raised the
concern that a steer written while SIGSTOP'd sits in the kernel
pipe buffer (default 64KB Linux). With v0.12's "forbid steers
while paused" lock this can't happen — `resolveSendStrategy`
short-circuits before reaching `RpcStdinQueue.enqueue`.

### 4.6 `forceTerminate` path

**Unchanged for v0.12.** SIGTERM straight on `run.proc`
(`src/runs.ts:1438`). For RPC subprocesses we do NOT first
issue an RPC `abort` and wait for graceful close. Reasons:

1. **Q6 lock.** `ensemble_kill` = "gone". Adding a graceful
   step changes the LLM's mental model and risks introducing a
   "graceful close requested but not yet observed" state that
   tangles with the W7 idempotency guard
   (`runs.ts:1386`/`:1454`) and the `applyCloseHandlerTerminal`
   race-fix (`:1296`). Inspector map Risk #4 — preserved.
2. **Out-of-scope.** Graceful-abort verb is `ensemble_abort`
   (§9 v2 list).

**`RpcStdinQueue.destroy` is still called** before SIGTERM, to
reject any pending stdin writes. This is housekeeping, not a
graceful-close protocol. The kernel will reap the pipe on
process exit anyway; we surface the rejection to anything
awaiting a confirmation (e.g. an `ensemble_send` tool-call
promise — §4.4 ack envelope below).

### 4.7 Watchdog `lastEventAt` bump policy

**Bump on `response` ack lines (NEW), plus today's
`message_end` and `tool_result_end`. Do NOT bump on
`extension_ui_request` or any other non-progress RPC line type.**

Today's bump sites: `src/event-handler.ts:64` (in `message_end`
handling) and `:97` (in `tool_result_end` handling). Q9 lock
extends, with the asymmetry oracle fix #2 enforced:

```ts
// src/event-handler.ts — extended dispatch (sketch from §4.2)

export function applyEvent(run: Run, event: unknown): EventEffect {
  const evt = event as { type?: string };
  if (typeof evt?.type !== "string") return { kind: "none" };

  // RPC progress signals bump lastEventAt. response = sub-agent
  // acked our command. message_end / tool_result_end already do
  // (see existing switch below). extension_ui_request does NOT
  // bump — sub-agent is BLOCKED waiting on the host's reply.
  if (run.streamingMode === "rpc" && evt.type === "response") {
    run.lastEventAt = Date.now();
  }

  // Dispatch to the existing AgentEvent handlers (message_end /
  // tool_result_end / agent_end / turn_end). They bump
  // lastEventAt on their existing branches.

  // ...existing switch...
}
```

**Asymmetry pinned (oracle fix #2).** `response` lines bump
`lastEventAt`; `extension_ui_request` lines do NOT. Rationale:
`response` is a sub-agent acknowledgement of a conductor
command (`steer`/`follow_up`/`prompt`), i.e. concrete progress
evidence. `extension_ui_request` is the *opposite* — the
sub-agent has blocked waiting on the host. Bumping on it would
mask the stall class "sub-agent waiting on unanswered UI
request" (in v0.12 the conductor auto-cancels in the same tick
so this rarely matters; in v0.13+ if `extension_ui_request` is
proxied to the user the no-bump policy becomes load-bearing).

Print-mode runs (`streamingMode !== "rpc"`) keep the old bump
sites only — preserves the v0.10 behaviour and witnesses
exactly. Mutation witness from PRD `:618` (W1 imports
`resolveKillOnStall` from `src/watchdog.ts` directly per
`docs/wdd.md` parallel-formula rule) extends: a new W2 witness
imports `bumpOnRpcLine(run, now, evt.type)` and verifies that
(a) flipping the `response` branch to no-op fails a test, AND
(b) flipping the `extension_ui_request` no-op branch to *do*
bump fails a different test (the asymmetry must be detectable
in both directions).

### 4.8 Slash command surface

**Add `/conductor send <agent_id> <message>` (Q8 lock).**

Mirrors the LLM tool's `streaming_behavior: "auto"` default —
no flag, the command picks the right strategy via
`resolveSendStrategy`. Power users who want explicit control
can pass `--steer` or `--follow-up` flags:

- `/conductor send <id> <msg>` — auto (running steerable →
  follow_up; terminal → resume).
- `/conductor send <id> --steer <msg>` — interrupt now.
- `/conductor send <id> --follow-up <msg>` — queue.
- `/conductor send <id> --resume <msg>` — explicit resume
  (errors if run is running).

`src/commands.ts` adds a `case "send":` branch alongside the
existing `pause`/`resume`/`stop`/`kill`/`focus`/`history`/`gc`/
`reconcile`/`watchdog`/`pin`/`unpin` cases (`src/commands.ts:833`–
`:870` — listed by inspector map §4). Same shape as the v0.10
slice 4 `/conductor watchdog status` dispatcher.

The `s` keybinding in `src/focused-stream-overlay.ts:228` does
not need to change — it already routes through `promptAndSendToRun`
which calls `resolveSendStrategy` via `validateSendable`. The
overlay just stops getting "is currently running" rejections
on RPC runs.

## 5. Designer's calls on remaining inspector questions

Six questions in inspector map §5 weren't pre-locked. Each gets
a default with one paragraph of justification.

### Q3 (steer-into-finishing race)

**EPIPE → mark steer dropped, surface to caller as
`{kind: "rejected", reason: "sub-agent finished before steer
was delivered"}`.**

Mechanics. `RpcStdinQueue.enqueue` returns a promise. When the
subprocess exits between line submit and kernel write, Node
emits `EPIPE` on `stdin.write`. The queue's drain handler
rejects the in-flight promise with cause-tagged error. The
LLM-tool / overlay caller catches and renders the rejection
verbatim. The conductor's `<sub-agent-completed>` notification
will follow shortly; no special envelope needed. We do *not*
re-route the dropped message anywhere (e.g. into a "next
spawn" buffer) — it's content the LLM/user produced and they
can re-send if they still care.

### Q4 (ack envelope)

**Surface RPC `response` lines as a new internal event the
conductor's tool-call promise can await before resolving.**

The LLM calling `ensemble_send` with `streaming_behavior:
"steer"` reasonably wants to know "did the steer reach the
agent?" Pi's `{type: "response", command: "steer",
success: true}` line (`[pi-dist]
modes/rpc/rpc-types.d.ts:155`) is the affirmative ack. We
correlate by the `id` field we attach when enqueuing
(`{"id":"steer-${nanoid()}","type":"steer","message":...}`).

`RpcResponseRouter` (a Map<id, {resolve, reject}> kept on `Run`
or in a registry-adjacent module) holds pending acks.
`routeRpcResponse` (§4.2) looks up the id, resolves the
promise, deletes the entry. The LLM tool's response payload
includes:

```jsonc
{
  "kind": "started",
  "agentId": "builder-x12y",
  "strategy": "rpc-steer",
  "delivered": true,        // pi acknowledged the steer
  "deliveredAt": 1716845210
}
```

For `follow_up`: same shape. For `resume`: no ack (resume goes
through the existing fresh-subprocess path). Pinned in slice 3
unit tests.

**Timeout for unanswered ack.** 30 seconds. If pi doesn't
respond inside the window we surface
`{kind: "rejected", reason: "ack timeout"}`. The actual steer
*may* have been delivered (pi might just be slow to write the
response line); the LLM should re-check the run's state via
`ensemble_status` rather than re-firing.

### Q5 (session-file binding)

**v0.12 = one subprocess = one session. Do NOT expose
`switch_session` / `fork` / `clone`. Defer.**

Reasons:

1. Latent risk. Inspector map Risk #5: `findSessionFile` /
   `discoverSessionPathIfMissing` (`runs.ts:129`/`:170`) were
   never tested against post-`switch_session` /
   post-`fork` session-file mutations. Any of those verbs
   could leave `run.sessionPath` stale.
2. No use case. The persona model is one-task-per-spawn. A
   sub-agent that wants a sub-sub-agent should call
   `ensemble_spawn` itself (recursion is supported by the
   shared registry).
3. `fork` is interesting for "what-if" runs but we have
   `ensemble_spawn` with explicit context for that.

If a future v0.13+ wants any of these verbs, gate them as
opt-in fields on `Run` (e.g. `Run.allowSessionMutation:
boolean`) and re-test the discovery functions exhaustively.

### Q7 — covered by §4.5

### Q8 (slash command)

**Add `/conductor send <agent_id> <message>` mirroring the LLM
tool surface.**

Pinned in §4.8 above.

### Q10 (archived-run compat)

**Unchanged — terminal/archived runs continue
resume-via-`pi --session`. Pinned in §4.4.**

## 6. Risk mitigations

For each of the inspector-map risks (§6 in the inspector doc),
a concrete mitigation pinned for the planner to slot into a
slice.

### Risk 1 — Stdin race / partial-event tearing

**Single-writer `RpcStdinQueue` (§4.2).** All `proc.stdin.write`
calls go through `enqueue`. Tested at unit level with a fake
`Writable` simulating back-pressure (returns `false` from
`write`, emits `drain`) and a multi-thread race (concurrent
enqueues land in submit order). Pinned in slice 2.

### Risk 2 — `extension_ui_request` blocking

**Always-cancel-and-warn (§4.2; oracle Risk 2 lock).** Every
inbound `extension_ui_request` is auto-answered with
`{type: "extension_ui_response", id, cancelled: true}` enqueued
synchronously via `RpcStdinQueue.enqueue`. No diagnostic loop,
no UI dispatch, no per-method branching. Persona-author
guidance: personas with `ctx.ui.*` extensions must handle
cancellation gracefully when spawned `steerable: true`. v0.13+
may proxy (kept in §9 out-of-scope). Tested by feeding a mock
`extension_ui_request` line through `applyEvent` and asserting
(a) the queue receives the cancellation envelope within the
same tick, AND (b) `lastEventAt` is NOT bumped (§4.7
asymmetry).

### Risk 3 — Watchdog confounding

**Bump `lastEventAt` on `response` only, not on
`extension_ui_request` (§4.7; oracle fix #2).** Asymmetry
verified by the W2 witness test that pins both directions of
the bump policy.

### Risk 4 — Forced-terminate path tangles

**Unchanged from today's path (§4.6).** `forceTerminate` keeps
SIGTERMing on `run.proc`. `RpcStdinQueue.destroy` rejects pending
writes but does not block termination. Tested by an integration
case (gated on `CONDUCTOR_LIVE_TESTS=1`) that spawns a steerable
subprocess, fires `forceTerminate` while a steer is in flight,
and asserts subprocess is dead, stdin promise rejects with cause
`"force-terminate"`, `Run.status` flips to `killed` exactly once
(W7 idempotency).

**Risk 4-adjacent: orphaned RPC runs on parent-pi crash
(oracle fix #3).** Pinned in §4.4 "Crash recovery for orphaned
RPC runs": reconcile-startup detects `streamingMode === "rpc"`
on readopt, SIGTERMs the orphaned subprocess, and reclassifies
to `killed` with `errorMessage: "orphaned:
rpc-stream-detached"`. Reuses v0.9.x D1 pattern (PRD `:524`) —
NO new RunStatus. Tested at unit level by extending
`tests/reconcile-startup-classifier.test.ts` with a steerable
orphan fixture.

### Risk 5 — Session-file divergence between modes

**Out-of-scope-elimination (§"Out of scope").** v0.12 doesn't
expose `switch_session` / `fork` / `clone`. Both modes write the
same on-disk schema; `findSessionFile` /
`discoverSessionPathIfMissing` keep working unchanged. Re-test
if any of those verbs land in v0.13+.

### Integration (gated on `CONDUCTOR_LIVE_TESTS=1`)

- **`tests/integration-rpc-spawn.test.ts`** — spawn a steerable
  sub-agent against a real `pi --mode rpc`. Cases (each ~5s,

### Mutation witnesses (`docs/wdd.md`)

## 10. Open questions for review

≤5 items. Things the designer genuinely couldn't decide and
wants oracle/user to gate before slicing.

1. **Initial-prompt-injection trigger.** §4.2 sketches sending
   the first `prompt` command "once the subprocess confirms
   it's up." Is the right trigger (a) the first stdout line
   (any line; the simplest indicator the subprocess survived
   exec); or (b) a synchronous wait on `get_state` succeeding;
   or (c) just write the line immediately after `spawn`
   (relying on the kernel pipe buffer). Inspector map didn't
   pin this. Slice 2 prototyping should answer; the design
   defaults to (a) but will revisit.

2. **`get_state` use during `auto` decision.** §4.3's resolver
   uses `Run.streamingMode` for the running × auto branch
   (always `rpc-follow-up`). It does NOT poll
   `get_state.isStreaming`. The cost: a steerable sub-agent
   that's between turns (idle) gets `follow_up`-queued instead
   of starting a fresh prompt directly. Acceptable tradeoff
   (queue and immediate-prompt-into-idle behave identically
   from pi's side per `[pi-dist] core/agent-session.js:888`).
   Confirm with oracle: this skips a `get_state` round-trip
   per send and is correct?

3. **Conductor-prompt §10 / §11 update.** The §10 delegation
   playbook (`PRD.md:259`-block; `src/conductor-prompt.ts:103`
   per inspector §4) advertises the persona roster + tools.
   `ensemble_send` gains the new `streaming_behavior` arg;
   §10 must mention the auto/steer/follow_up/resume distinction
   and the queue-don't-interrupt default. The conductor system
   prompt is the LLM's only doc — v0.12 closure should include
   a critic-gated §10 patch. Is that a v0.12 slice or a
   follow-up?

4. **Reconcile-orphan SIGTERM ordering.** §4.4's crash-recovery
   policy SIGTERMs the orphaned RPC subprocess and reclassifies
   to `killed`. The current v0.9.x reconcile classifier is pure
   (returns a decision; doesn't perform I/O); the SIGTERM is
   the executor's job. Should slice 4 introduce a new executor
   verb (`reclassify-killed-and-terminate`) or reuse the
   existing `reclassify-killed` branch and have the executor
   conditionally signal based on the errorMessage prefix? My
   instinct is the latter (less surface area), but it couples
   the executor to the errorMessage shape. Confirm with oracle.
