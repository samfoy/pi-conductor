# Design: (B-4) `toolUse.name` sanitizer via `pi.on("context")`

> Designer output for the v0.8.2 wedge-class fix. Scope: defense-in-depth in
> pi-conductor against malformed `toolCall.name` shipping to a provider that
> server-validates the field. Single slice. Pure-function core. Hook is
> `pi.on("context", …)` (not the brief's "transformContext" — that is the
> `pi-agent-core` AgentLoopConfig field; the public extension event for the
> same callback is named `"context"`. See evidence in §1.4 below.)

## Goal

Prevent the wedge class where a model hallucinates a `toolCall.name` that
violates the provider's tool-name charset (Bedrock's
`[a-zA-Z0-9_-]+`), which causes every subsequent LLM turn to fail server-side
validation and permanently soft-bricks the session. Close the wedge in
pi-conductor today, independent of upstream fixes.

## Approach

Register a single `pi.on("context", …)` handler in `src/index.ts` that runs
just before each API call. The handler walks the in-memory `AgentMessage[]`,
finds any assistant `toolCall` content block whose `name` violates the
strictest acceptable charset, mutates it to a regex-safe placeholder, and
mirrors the same rewrite onto the matching `toolResult` message(s) by
`toolCallId`. The on-disk JSONL is **not** touched. A loud one-time
`console.warn` (per `toolCallId`) and a `widget.notify(...)` surface inform
the user that sanitization fired.

The hook is the right boundary because it is the last user-visible rewrite
point in the AgentMessage pipeline before serialization to provider format
(`convertToLlm` runs after `transformContext` per pi-agent-core's
`AgentLoopConfig`; `pi.on("context")` is wired into `transformContext` —
verified in `node_modules/@earendil-works/pi-coding-agent/dist/core/extensions/runner.js:639–668`).
This means: any corruption present in the JSONL gets sanitized on the next
turn after session resume; sanitization runs every turn, so we don't need a
separate session-load hook.

The fix is wholly inside pi-conductor's existing footprint and reuses the
same pattern used for filtered-context inheritance (a pure function tested
in isolation, called from a thin extension-event hook). It does **not**
share state or interleave with `filterParentContext` — different surfaces,
different invariants (see §5 *Interaction with v0.8.1 Item 1*).

### 1.1 Scope of the wire-side problem

Bedrock's validation error path is:

```
messages.<N>.member.content.<M>.member.toolUse.name
```

i.e. the validated field is the **toolCall's** `name`, NOT the toolResult's
`toolName`. Empirical evidence from the live wedged session
(`~/.pi/agent/sessions/--workplace-samfp-Rosie--/2026-05-15T22-07-30-698Z_…jsonl`,
lines 106–107) confirms this:

- L106 `message.content[1].type === "toolCall"`, `name === 'ensemble_kill" >\n</invoke>'`
  — this is the field Bedrock rejects.
- L107 `message.role === "toolResult"`, `toolName === 'ensemble_kill" >\n</invoke>'`
  — Bedrock's serializer (`amazon-bedrock.js:560–595`) does NOT send
  `toolName` on the wire; only `toolUseId`, `content`, and `status`.
- L107 `content[0].text` contains the bad sequence verbatim
  (`"Tool ensemble_kill\" >\n</invoke> not found"`) — this DOES go on the
  wire but is unvalidated free-form text.

For Bedrock alone, sanitizing only the `toolCall.name` would be sufficient
to unwedge. **Other providers differ**:

- Anthropic (`anthropic.js:860,870`): only `tool_use_id` is on the wire for
  toolResults. Like Bedrock — toolName not on wire.
- OpenAI-completions (`openai-completions.js:737–739`): when
  `compat.requiresToolResultName` is true (some OpenAI-compatible providers),
  `toolMsg.toolName` is sent on the wire as the toolResult's `name` field.

So **for cross-provider correctness, we must sanitize both fields.** The
extra work is trivial (one paired write).

### 1.2 What we sanitize

Three locations per corrupted `toolCallId` (matched on the toolCall's `id`):

1. **`assistantMsg.content[i].name`** where `content[i].type === "toolCall"`
   — the field that triggers the validation error. **Required.**
2. **`toolResultMsg.toolName`** for every `toolResult` whose
   `toolCallId === toolCall.id` — required for OpenAI-compat providers, free
   for Bedrock/Anthropic. **Required for cross-provider safety.**
3. **`toolResultMsg.content[*].text`** when the text contains the bad
   substring (e.g. `"Tool <bad-name> not found"`) — purely cosmetic, goes
   on the wire as free text. **Optional but cheap; do it.**

The toolCall's `id` itself is a UUID-ish opaque token that is NOT validated
against any charset. It is the load-bearing pairing key. We never mutate it.

### 1.3 Slugification rule

Given an original bad name string `s`, produce the placeholder via:

```
slug = s.match(/[a-zA-Z0-9_-]+/g)?.join("_") ?? ""
slug = slug.replace(/^[_-]+|[_-]+$/g, "")    // trim leading/trailing _ -
slug = slug.length > 64 ? slug.slice(0, 64) : slug
return slug.length > 0 ? `${slug}_INVALID` : "INVALID_TOOL_NAME"
```

For the live wedge (`'ensemble_kill" >\n</invoke>'`) this yields
`"ensemble_kill_invoke_INVALID"` — the model on the next turn sees a
"tool not found" toolResult naming a tool clearly recognizable as a
sanitized derivation of what it tried, and recovers gracefully.

## Public interface

Two new exports in `src/sanitizer.ts`:

```ts
export const TOOL_NAME_REGEX = /^[a-zA-Z0-9_-]+$/;

/**
 * Sanitize an AgentMessage[] in-place-safe (returns a new array; does not
 * mutate input). Replaces toolCall.name violations against TOOL_NAME_REGEX
 * with a slugified placeholder, mirrors the rewrite onto matching
 * toolResult.toolName fields, and rewrites embedded mentions in
 * toolResult content text. Reports each rewrite via the supplied callback
 * once per toolCallId. Pure-function core; no I/O, no global state.
 */
export interface SanitizeReport {
  /** The toolCall id whose name was sanitized. */
  toolCallId: string;
  /** The original (unsafe) name, byte-for-byte. */
  originalName: string;
  /** The placeholder we replaced it with. */
  sanitizedName: string;
}

export interface SanitizeOptions {
  /** Charset regex tested via .test(name); default TOOL_NAME_REGEX. */
  isValid?: (name: string) => boolean;
  /** Build the placeholder; default = slugifyForBedrock. */
  buildPlaceholder?: (originalName: string) => string;
  /** Called once per sanitized toolCallId. Default no-op. */
  onSanitize?: (report: SanitizeReport) => void;
}

export function sanitizeToolNames(
  messages: AgentMessage[],
  opts?: SanitizeOptions,
): AgentMessage[];
```

The hook wiring (single new block in `src/index.ts`, no new file in the
extension surface):

```ts
pi.on("context", async (event) => {
  const reports: SanitizeReport[] = [];
  const messages = sanitizeToolNames(event.messages, {
    onSanitize: (r) => reports.push(r),
  });
  for (const r of reports) {
    console.warn(
      `[pi-conductor] sanitized malformed toolUse.name ${JSON.stringify(r.originalName)} → ${r.sanitizedName} (id=${r.toolCallId})`,
    );
    try {
      ctxRef?.ui.notify(
        `pi-conductor: sanitized malformed tool name → ${r.sanitizedName}`,
        "warning",
      );
    } catch {}
  }
  return { messages };
});
```

The handler does NOT short-circuit when `reports.length === 0`; it always
returns `{ messages }` (the array is the function's deep copy regardless).
This keeps the hook idempotent and avoids a "fast-path returns the input
array reference, slow-path returns a new array" inconsistency that has
historically tripped up tests in `filterParentContext`.

The sanitizer is a pure function — no `ctxRef`, no `pi`, no `console`
inside `sanitizeToolNames`. Logging is the caller's responsibility (the
hook). This makes the sanitizer testable in `tests/sanitizer.test.ts`
without mocking the extension API.

## Data flow

```
Parent LLM turn begins
  ↓
pi-coding-agent calls Agent.streamLLM(messages)
  ↓
Agent.config.transformContext(messages)         ← brief calls this “transformContext”
  ↓
ExtensionRunner.emitContext(messages)
  ↓
for each ext handler: pi.on("context", h)       ← we hook here
  ↓
sanitizeToolNames(messages, { onSanitize })
  ↓                          (pure; returns NEW array)
  ↓ logs each rewrite once
  ↓
return { messages: sanitized }
  ↓
agent.config.convertToLlm(sanitized)            ← provider-specific shape
  ↓
provider.streamSimple(...)                      ← wire request
  ↓
Bedrock validates messages.<N>.content.<M>.toolUse.name
  ↓
✓ all names match [a-zA-Z0-9_-]+ → request accepted
```

The on-disk JSONL is unchanged. Original corruption preserved as evidence.

For sub-agent flow (orthogonal):

```
Parent invokes ensemble_spawn
  ↓
runs.ts.seedSubAgentSession() builds the seed
  ↓
filterParentContext(parentMessages)             ← DROPS ensemble_* turns; not modified
  ↓
seed.jsonl written for the sub-agent
  ↓
sub-agent pi process loads its session
  ↓
sub-agent's own pi.on("context") fires on its first LLM call
  ↓
sanitizeToolNames runs again                    ← belt-and-suspenders; same code
```

No double-write of any field; the sub-agent's sanitizer would only fire if
the seed itself contained a corrupted name (rare; would require parent's
sanitizer to have somehow missed it). Idempotent: a second pass over a
sanitized array is a no-op (the placeholder matches `[a-zA-Z0-9_-]+`).

## Decision log

### Q1 — Sanitization shape: **(a) replace name with slugified placeholder**

Counterarguments rejected:

- **(b) drop the toolCall block, keep the message.** Rejected: leaves the
  parent assistant turn with no toolCall, but the next-turn matched
  toolResult would still arrive (or fail to arrive). Either way the
  sequence is incoherent — at best the model gets confused, at worst the
  provider rejects an unmatched toolResult. Either-way disposal of the
  toolResult creates an asymmetric history.
- **(c) drop the entire message.** Rejected: same orphan problem, larger
  surface (also drops the legitimate `thinking` block on the same turn,
  which in the live wedge had load-bearing reasoning the model would
  benefit from re-reading post-recovery — `"I'm going to remove the
  analyst from consideration since…"`). Information loss is a worse trade
  than a recognizable placeholder.
- **(d) refuse to send.** Rejected: this *is* the wedge UX. We're trying
  to escape it, not give it a different name. (d) is acceptable only if
  paired with a JSONL-rewrite recovery action, and even then the user has
  to do something — the whole point of (a) is automatic, transparent
  recovery.

(a) wins because: (i) preserves message coherence by mirroring the rewrite
to the toolResult, (ii) gives the model a recognizable hint via the
slugified original (`ensemble_kill_invoke_INVALID` reads as "the thing
that didn't exist"), (iii) deterministic and reversible — the JSONL still
has the original byte sequence for audit, only in-memory representation
mutates, (iv) idempotent over multiple passes (the placeholder is
self-stable).

### Q2 — Orphan toolResult handling: **paired transaction; rewrite both sides**

Invariant pinned: post-sanitization, for every `toolCallId T` that was
sanitized:

- The unique `toolCall` block with `id === T` has `name === placeholder(T)`.
- Every `toolResult` message with `toolCallId === T` has
  `toolName === placeholder(T)` AND any literal substring of the original
  bad name in its `content[*].text` is replaced with the placeholder.
- All other `toolName === toolCall.name` remains untouched.
- `toolUseId` / `toolCallId` / `id` opaque tokens are NEVER modified.

Edge case: multiple toolResults share a single `toolCallId`. Pi's schema
allows this in principle; the existing Bedrock serializer
(`amazon-bedrock.js:574–588`) loops over consecutive toolResults. The
sanitizer iterates all matched toolResults, not just the first.

Edge case: orphan toolResult whose toolCall is missing entirely (history
truncation, custom seeding). The sanitizer cannot rewrite a missing
toolCall, so it sanitizes the toolResult's `toolName` independently
against the same regex — never leaves a bad name in any position.

Edge case: orphan toolCall whose toolResult is missing entirely. Same —
sanitize the toolCall.name in isolation. The provider will see a normal
"tool was called, no result" partial-turn shape and can resume.

### Q3 — Lifecycle: **(i) `pi.on("context")` only**

Counterarguments rejected:

- **(ii) appendMessage hook.** Rejected: pi-coding-agent does not expose a
  pre-write hook on the JSONL append path (`session-manager.ts`). Adding
  one would be an upstream change. Out of scope for v0.8.2.
- **(iii) session-load sanitization.** Rejected: redundant. The first API
  call after session resume invokes `pi.on("context")`; the sanitizer fires
  on the loaded messages just like any other turn. There is no API call
  between session-load and the first context hook, so the JSONL has no
  opportunity to flow to the wire un-sanitized.

(i) is sufficient and minimal. It defends every API call uniformly,
including the first call after a wedged-session resume.

### Q4 — Live vs persisted: **do NOT write back to JSONL**

Counterarguments rejected:

- **Write-back option.** Pros: future sessions don't repay sanitization
  cost; on-disk JSONL is recoverable without manual `jq`. Cons: silently
  mutates user data; harder to audit; introduces a new write path that
  could itself fail and corrupt the JSONL further; surprising behavior
  ("why did pi rewrite my session?"). Cost of repaying sanitization is
  negligible (a single regex test per toolCall per turn — sub-microsecond).
- **Conditional write-back gated by user prompt.** Pros: explicit user
  consent. Cons: this is a recovery UX surface, not a sanitizer surface.
  The recovery snippet from `investigator-bdgz` already exists for users
  who want disk-side cleanup. Don't bloat the sanitizer with UI.

Decision: in-memory only. Document the rationale in code (`docs/sanitizer.md`
inline as a JSDoc on `sanitizeToolNames`). Recovery script remains a
separately-shipped utility (out of scope here).

### Charset choice — **always enforce `[a-zA-Z0-9_-]+`**

Bedrock's regex is the strictest among major providers we ship. Anthropic
and OpenAI-completions are asserted on the basis of provider-doc reading;
no in-tree code witness was found (their pi-ai serializers pass `c.name`
to the wire verbatim without client-side regex validation, so we can't
cite a charset-test line — server-side acceptance is a doc claim). Pluggable
via `SanitizeOptions.isValid` if a future provider needs different rules.
Provider-aware regex would be a v0.8.3+
enhancement (different regex per `model.provider`), explicitly out of
scope here. Enforcing the strictest regex always means: pi-conductor never
ships a name to any provider that any major provider would reject. Cost is
zero — no false positives on legitimate tool names (every shipped pi tool
name and every documented `ensemble_*` name matches `[a-zA-Z0-9_-]+`).

### Logging — **`console.warn` + `widget.notify`, both, once per toolCallId**

Silence on auto-mutation is a debugging nightmare. Log each sanitization
once per `toolCallId` (not per turn — the same sanitized name will pass
through the hook on every subsequent turn until the toolCall ages out of
context). The hook tracks a `Set<string>` of `toolCallId`s already warned
about across turns (session-scoped state in the extension entry point) so
the same rewrite doesn't spam logs every turn. The `warnedToolCallIds`
set is cleared in `pi.on('session_shutdown', …)` alongside the existing
widget/listener teardown, mirroring the pattern in `src/index.ts:266`.

Surface:

- `console.warn(...)` with byte-precise representation of the original
  (`JSON.stringify(originalName)` — preserves `\n`, `"`, control chars).
  Goes to stderr; visible to the user when they `tail -f` pi's log or
  when they're running pi attached to a terminal.
- `ctx.ui.notify(...)` with a short summary. Surfaces in the TUI as a
  warning banner.

The notify text omits the original byte sequence (could include unprintable
characters that mangle the TUI). Console log carries the full evidence.

## Failure modes

- **Sanitizer throws** during message walk (e.g. unexpected message shape).
  The `pi.on("context")` runner catches throws via its `try/catch` block
  (`extensions/runner.js:654–663`) and logs to the extension error stream;
  the original `messages` flow through unchanged. Net effect: a bug in the
  sanitizer cannot wedge a session worse than no sanitizer at all. The
  emit error is surfaced via `ctx.uiError`/extension error events, so
  bugs are observable. **No try/catch needed inside the handler** —
  `runner.emitContext` wraps it.
- **Slugification produces empty string.** Fallback to `"INVALID_TOOL_NAME"`.
  Bedrock will return "tool not found" on next turn (because no tool with
  that name is registered); model recovers.
- **Sanitization runs every turn over the same corrupted history.** The
  rewrite is idempotent — second pass sees `name === placeholder` (already
  matches the regex), no-op, no log. Cost is one regex test per toolCall
  per turn.
- **Multiple corrupted toolCalls in one turn.** Each is sanitized
  independently, each emits one report. No interaction.
- **Corrupted name happens to be a registered ensemble tool name plus
  noise** (e.g. `"ensemble_spawn\""`). Slug = `ensemble_spawn`, placeholder
  = `ensemble_spawn_INVALID`. The model sees a tool-not-found result, NOT a
  successful spawn. Important: we do NOT slug-then-match-existing-tool-then-reuse
  — that would let a corruption succeed as a different tool than the model
  intended. Always append `_INVALID` so the placeholder name is never a
  real tool.
- **Provider added that uses a different valid charset and rejects
  `_INVALID` suffix.** Not currently a known provider. If introduced, the
  charset choice can be relaxed via `SanitizeOptions.isValid` in the hook
  wiring without touching the sanitizer.
- **Race with v0.8.1's `filterParentContext`.** None — the filter runs at
  sub-agent-seed time (in `runs.ts` worker code path), the sanitizer runs
  at parent API-call time (in `pi.on("context")` hook). Different
  call-stacks, different inputs, no shared mutable state. See §5 below.
- **Streaming partial state.** The wedge is on a *fully-formed* assistant
  message that has already been persisted. The sanitizer never sees a
  partial message — `pi.on("context")` fires on the assembled history just
  before serialization. Streaming-time contamination would be a different
  bug class (and is outside this slice).

## Risks and edge cases

### Load-bearing assumptions

1. **`pi.on("context")` runs on every API call before serialization.**
   Verified in `extensions/runner.js:639–668` and `core/sdk.js`'s
   `transformContext` wiring. Risk: an upstream rename or removal would
   silently disable the sanitizer. Mitigation: add a startup self-test in
   the hook handler that verifies the handler is registered (low value,
   high noise — defer unless we actually see breakage).
2. **`structuredClone` of messages happens upstream
   (`runner.emitContext` line 641).** Means we get a deep copy and can
   mutate freely. We still return a new array for explicitness — that's
   what the docs say to do.
3. **Bedrock's regex is `[a-zA-Z0-9_-]+`.** Confirmed by the live error
   message verbatim. Risk: Bedrock tightens the regex (e.g. requires
   length ≥ 2). Mitigation: `SanitizeOptions.isValid` is pluggable; the
   hook wiring can be updated without touching the sanitizer. Length
   constraint already handled (slug max 64; placeholder always ≥ 16 chars
   from `_INVALID` suffix).
4. **No provider exists for which `_INVALID`-suffixed slugified names
   would themselves fail validation.** Verified for Bedrock, Anthropic,
   OpenAI. New provider added later might require relaxation; see Q4
   counterargument.

### Things we deliberately don't handle

- **Same-turn streaming corruption.** If a stream parser interpolates
  XML mid-stream (different bug class), the malformed name would arrive
  as part of a completed message and would still be sanitized at the next
  context hook — but we don't intervene at the streaming layer. Out of
  scope.
- **Corrupted `toolCall.id`.** The id is opaque (UUID-ish from the
  provider). Theoretical malformed ids would be uncatchable on the wire
  (no charset rule is enforced) and unrelated to the witnessed wedge. Out
  of scope.
- **Corrupted `arguments`.** A toolCall with valid name but
  malformed/unparseable arguments would not trip Bedrock's name regex —
  it's a different validation path (typically per-tool schema validation
  on the input). Out of scope.
- **Corrupted text content blocks.** Free text is not regex-validated
  on the wire. We rewrite text content only when it embeds the bad name
  substring (cosmetic).

### Things we monitor for

- A backlog item to capture sanitizer firing frequency (count via the
  `onSanitize` callback on the `widget` model). High frequency = upstream
  Bedrock instability and worth filing externally. Already logs to
  `console.warn`; aggregating later is trivial.
- Append a v0.8.3+ followup to `docs/v0.8.2-backlog.md`: provider-aware
  charset (when more than one charset matters in practice). Currently a
  single regex is sufficient.

### Interaction with v0.8.1 Item 1's `(a')` filter

Different surfaces. `filterParentContext` operates on
`getParentMessages()` → seed slice for sub-agents. `sanitizeToolNames`
operates on the parent's own outgoing context immediately before the
provider call. Order of operations:

| Caller                     | Function                  | Input                  | Output            |
|--                          |--                         |--                      |--                 |
| Parent's API turn          | `sanitizeToolNames`       | parent history         | parent history (sanitized) |
| `ensemble_spawn`           | `filterParentContext`     | parent history         | filtered seed slice |
| Sub-agent's first API turn | `sanitizeToolNames`       | sub-agent history      | sub-agent history (sanitized) |

`filterParentContext` does NOT call `sanitizeToolNames`. The seeded slice
flows through the sub-agent's own context hook on its first turn, so
sanitization happens at the API boundary either way. We do not collapse
the two functions into one — orthogonal concerns, orthogonal tests.

## Open questions for the design's own oracle gate

1. **Should the sanitizer also run inside `filterParentContext`?**
   Current decision: no — orthogonal. But there's a subtle case: the
   parent's seeding code calls `filterParentContext` on the parent
   history; the seed is written to the sub-agent's session JSONL; the
   sub-agent's first API call then triggers ITS sanitizer. If the parent
   has a corrupted toolCall in its history that did NOT trip the parent's
   own sanitizer (e.g. because the parent has been wedged across a
   session-restart and the user spawned a sub-agent before sending any
   parent turn), the seed could carry corruption forward. The sub-agent
   catches it on its first turn — but that's an extra-context-hop where
   we could short-circuit by sanitizing at seed time. **Cost/benefit**:
   cost = ~5 LoC in `runs.ts` (call sanitizer after filter); benefit =
   defense-in-depth, no realistic scenario where it changes outcomes.
   Lean: leave out for now; capture as a v0.8.3 backlog if anyone hits it.

2. **Slug truncation length (64) — too short / too long?** No data. 64 is
   chosen because Bedrock tool names are typically ≤30 chars and 64 is a
   round power-of-2 approximation. If the original is dramatically long,
   the model might benefit from seeing the whole thing slugified for
   recovery context. Lean: 64 is fine; revisit if a real corruption
   exceeds it.

3. **`onSanitize` callback signature.** Should we also include a snapshot
   of the message index, so the caller can correlate to the JSONL? Lean:
   no — the `toolCallId` is the canonical correlation key, present in
   the JSONL. Adding index would invite "but which index?" confusion
   (pre/post-filter, pre/post-thinking-drop, etc.).

4. **Should the placeholder be configurable per-extension-config?** E.g.
   a user might want `__INVALID_NAME` instead of `_INVALID` suffix. Lean:
   not in v0.8.2 — `SanitizeOptions.buildPlaceholder` is exposed for
   advanced cases and tests; the default is shipped opinionated.

5. **Cross-tool sanitization (skill-installed tools beyond pi's
   builtins).** Does the sanitizer fire on names like `andes_query` or
   `taskei_create`? Yes — uniformly. They all match the safe regex
   already. Confirmed no false positives from a quick grep against
   `node_modules/@earendil-works/pi-coding-agent/dist/core/tools/`.
